## 1. การวิเคราะห์ช่องว่างเนื้อหา (Gap Analysis)

จากการตรวจสอบเนื้อหาเดิมใน `chapters/ch06-wireless/README.md` พบข้อจำกัดและประเด็นที่จำเป็นต้องขยายเพิ่มเติมดังนี้:

1.  **ด้านพื้นฐานการสื่อสารไร้สาย (Section 6.1):**
    *   *เดิม:* มีการอธิบายย่าน ISM และระยะทางแบบคร่าว ๆ ขาดสูตรคำนวณการลดทอนเชิงคณิตศาสตร์
    *   *การปรับปรุง:* เพิ่มการวิเคราะห์เปรียบเทียบเชิงวิทยุระหว่างย่าน **2.4 GHz vs Sub-GHz** (การทะลวง/ดูดซับ/การกระจายตัว), เพิ่มสูตร **Inverse Square Law** ($P_r \propto 1/d^2$) และโมเดล **Free Space Path Loss (FSPL)** ในหน่วยลอการิทึม (dB) พร้อมตัวแปรความเร็วแสงและ gain ของสายอากาศ, อธิบายแนวคิด **RSSI (dBm)** และ **SNR (dB)** เชิงลึกและการประยุกต์ใช้ในงบประมาณการเชื่อมโยงสัญญาณ (Link Budget)
2.  **ด้าน Wi-Fi (Section 6.2):**
    *   *เดิม:* มีตารางเปรียบเทียบมาตรฐานแบบคร่าวๆ ขาดมาตรฐาน IoT สำคัญอย่าง 802.11ah (Wi-Fi HaLow) และไม่ได้ลงลึกกลไกการต่อเน็ตเวิร์ก
    *   *การปรับปรุง:* ขยายรายละเอียดตารางมาตรฐาน 802.11 b/g/n/ac/ax/ah, เพิ่มคำอธิบายการทำงานของโหมด **AP+STA (Dual Mode)**, เพิ่มรายละเอียดขั้นตอนการเชื่อมต่อวิทยุ (**Scanning -> Authentication -> Association -> 4-way Handshake**), เปรียบเทียบ **DHCP vs Static IP** และอธิบายการเชื่อมโยงบริการด้วยพอร์ต **TCP/IP** (Port 80, 443, 1883, 502)
3.  **ด้าน Bluetooth และ BLE (Section 6.3):**
    *   *เดิม:* อธิบายสถาปัตยกรรม GATT สั้นมาก ขาดรายละเอียดชนิดของ Attribute และขั้นตอนการทำงานวิทยุ
    *   *การปรับปรุง:* เปรียบเทียบบลูทูธคลาสสิกกับ BLE เชิงพลังงานและสถาปัตยกรรม, อธิบายรายละเอียดโครงสร้าง **GATT** (Profile, Service, Characteristic, Descriptor, 16-bit vs 128-bit UUID), และอธิบายการทำงานของสถานะ **Advertising vs Connection** (Broadcaster/Observer vs Central/Peripheral)
4.  **ด้าน Zigbee (Section 6.4):**
    *   *เดิม:* มีรายละเอียดพื้นฐาน แต่ขาดทฤษฎีการจัดส่งข้อมูลระดับ Mesh และบทบาทหน้าที่เชิงลึก
    *   *การปรับปรุง:* อธิบายโครงสร้างระดับล่างตามมาตรฐาน **IEEE 802.15.4** ย่านความถี่และแบนด์วิดท์, อธิบายกระบวนการส่งต่อแบบ **Multi-hop Routing** และระบบการหาเส้นทางสำรองอัตโนมัติ (**Self-Healing**), และหน้าที่ของอุปกรณ์ทั้งสามตัว (**Coordinator, Router, End Device**)
5.  **ด้าน LoRa / LoRaWAN (Section 6.5):**
    *   *เดิม:* ขาดการอธิบายเชิงคณิตศาสตร์และพารามิเตอร์การตั้งค่าวิทยุ
    *   *การปรับปรุง:* อธิบายกลไก **CSS (Chirp Spread Spectrum) Modulation** รวมถึงพารามิเตอร์สำคัญ: Spreading Factor (SF), Bandwidth (BW) และ Coding Rate (CR) ที่มีผลต่อระยะทางและเวลาเดินทางในอากาศ (Time-on-Air), อธิบายสถาปัตยกรรม LoRaWAN (Device -> Gateway -> Network Server -> Application Server), และพฤติกรรมอุปกรณ์คลาส **Class A, B, C** รวมถึงข้อจำกัดทางกฎหมายด้าน **Duty Cycle (1%)** และข้อตกลง **Fair Access Policy**
6.  **ด้าน Cellular IoT & 5G (Section 6.6):**
    *   *เดิม:* มีภาพรวมสั้นๆ ขาดทฤษฎีการจัดการพลังงานของระบบเซลลูลาร์
    *   *การปรับปรุง:* อธิบายฟังก์ชัน **PSM (Power Saving Mode)** และ **eDRX** ที่ใช้ใน NB-IoT, การรองรับ **Handover** ของ LTE-M, และทิศทางการเติบโตของเทคโนโลยี 5G ในด้าน **mMTC** (การต่อเชื่อมหนาแน่นสูง) และ **URLLC** (ความหน่วงต่ำพิเศษเพื่อการควบคุมความเร็วสูง)
7.  **กรณีศึกษาเชิงวิศวกรรมเครื่องกลและอุตสาหกรรม (Section 6.8):**
    *   *เดิม:* มีตัวอย่างสั้นมากในเชิงชีวิตประจำวัน
    *   *การปรับปรุง:* สร้างกรณีศึกษาทางวิศวกรรมที่มีความสมบูรณ์และชัดเจน 3 รูปแบบ:
        *   *กรณีศึกษาที่ 1:* ระบบตรวจวัดการสั่นสะเทือนมอเตอร์ไฟฟ้า (Predictive Maintenance for ID Fan) วิเคราะห์ความเร็วส่งข้อมูลระดับสูงเพื่อทำ FFT โดยเลือกใช้ **Wi-Fi**
        *   *กรณีศึกษาที่ 2:* ระบบสายพานลำเลียงคลังสินค้าอัตโนมัติ (Conveyor Automation) เน้นความเสถียรและการครอบคลุมพื้นที่โดยปราศจากการกวนจากเหล็กโครงสร้าง เลือกใช้ **Zigbee Mesh**
        *   *กรณีศึกษาที่ 3:* ระบบควบคุมฟาร์มปาล์มขนาด 200 เอเคอร์ (Smart Irrigation) วิเคราะห์การทะลุผ่านของคลื่นวิทยุย่านความชื้นสูงและไม่มีสายส่งไฟ เลือกใช้ **LoRaWAN**
8.  **โค้ดตัวอย่าง Arduino C++ สำหรับ ESP32 (Section 6.9):**
    *   *เดิม:* เป็นโค้ดสั้น ๆ มีโอกาสเกิดปัญหาค้างเมื่อสัญญาณเน็ตหลุด และไม่ได้แสดงโค้ด BLE
    *   *การปรับปรุง:* จัดทำโค้ดตัวอย่างที่สมบูรณ์และคอมไพล์ผ่านจริง 2 ตัวอย่าง:
        *   *โค้ด Wi-Fi:* รองรับทั้ง Static IP / DHCP การสลับวิทยุกลับขึ้นมาเชื่อมใหม่แบบไม่บล็อกการทำงานหลัก (Non-blocking Connection via `millis()`) และมีระบบป้องกันบอร์ดค้างจาก Client ที่ไม่ยอมส่งข้อมูล (Client Timeout limit)
        *   *โค้ด BLE:* จำลองระบบวัดเซ็นเซอร์อุณหภูมิของโรงงานตามมาตรฐาน SIG (16-bit UUID Environmental Sensing Service) และช่องรับข้อมูลควบคุมมอเตอร์แบบเขียนกลับ (128-bit Custom UUID Write callback) พร้อมฟังก์ชันสั่งรันกระจายโฆษณาต่อเนื่องทันทีเมื่อมือถือตัดการเชื่อมต่อ (Auto-Restart Advertising)

### 2.1 รายละเอียดการแก้ไขที่แนะนำในแต่ละส่วน

#### ส่วนที่ 6.1: พื้นฐานการสื่อสารไร้สาย (Wireless Communication Fundamentals)
*   **แนวทางการเขียน:** เพิ่มสมการคณิตศาสตร์และอธิบายหน่วยวัดความสูงต่ำของคลื่นวิทยุให้เห็นภาพชัดเจนตามทฤษฎี
*   **สมการหลักที่เพิ่ม:**
    *   *Inverse Square Law:* $P_r \propto \frac{P_t}{d^2}$
    *   *Free Space Path Loss (FSPL):*
        $$\text{FSPL (dB)} = 20\log_{10}(d_{\text{km}}) + 20\log_{10}(f_{\text{MHz}}) + 32.44$$
    *   *Signal-to-Noise Ratio (SNR):*
        $$\text{SNR (dB)} = P_{\text{signal (dBm)}} - P_{\text{noise (dBm)}}$$

#### ส่วนที่ 6.2: Wi-Fi และโค้ดตัวอย่างการต่อเครือข่ายแบบ Non-blocking
*   **การปรับปรุงตารางมาตรฐาน 802.11:** เพิ่มแถวสำหรับ **802.11ah (Wi-Fi HaLow)** เพื่อแสดงแนวทางเทคโนโลยี Wi-Fi ย่าน Sub-GHz ที่ประหยัดไฟและส่งระยะไกล 1 กิโลเมตร
*   **กระบวนการเชื่อมต่อ (Connection Flow):** บรรยายการแลกเปลี่ยนเฟรมวิทยุ (Probe Request -> Probe Response -> Authentication -> Association) และกระบวนการเข้ารหัสความปลอดภัยด้วย **4-Way Handshake** (WPA2/WPA3 PSK -> PTK)
*   **โค้ดตัวอย่าง ESP32 Wi-Fi:** เขียนโค้ดที่รองรับการเปลี่ยนโหมด Static IP และ DHCP โดยใช้โครงร่างแบบ Non-blocking
    ```cpp
    // โครงสร้างการเชื่อมต่อใหม่วิทยุแบบไม่บล็อกลูปการทำงานหลัก
    if (WiFi.status() != WL_CONNECTED) {
      isCurrentlyConnected = false;
      if (currentMillis - lastAttemptTime >= reconnectInterval) {
        WiFi.disconnect();
        WiFi.begin(ssid, password);
        lastAttemptTime = currentMillis;
      }
    }
    ```

#### ส่วนที่ 6.3: Bluetooth/BLE และโค้ดตัวอย่าง GATT Server
*   **ทฤษฎี GATT (Generic Attribute Profile):** ขยายการแบ่งโครงสร้างข้อมูลจากระดับ Profile -> Service -> Characteristic -> Descriptor พร้อมอธิบายแนวคิด 16-bit UUID และ 128-bit UUID
*   **กระบวนการสื่อสาร (Advertising vs Connection):** อธิบายจังหวะการทำงานโฆษณาในช่องวิทยุพิเศษ (Channel 37, 38, 39) เพื่อลดการรบกวนสัญญาณ Wi-Fi
*   **โค้ดตัวอย่าง ESP32 BLE GATT Server:** การจัดเก็บข้อมูลตามข้อกำหนด SIG ตัวเลขทศนิยมให้คูณทวีคูณก่อนบันทึก เช่น อุณหภูมิ $28.50\ ^\circ\text{C}$ จัดเก็บเป็น $2850$ (ขนาด `int16_t` หรือ 2 ไบต์) และมี Descriptor `BLE2902` สำหรับเปิดให้ Client สั่ง Notify ได้
    ```cpp
    // การเตรียมข้อมูลก่อนส่ง Notify
    int16_t tempPayload = (int16_t)(simulatedTemp * 100);
    pTempChar->setValue((uint8_t*)&tempPayload, 2);
    pTempChar->notify();
    ```

#### ส่วนที่ 6.4: Zigbee (IEEE 802.15.4)
*   **สถาปัตยกรรม Mesh:** อธิบายการทำงานแบบ Multi-hop Routing และการหาเส้นทางสำรองอัตโนมัติ (**Self-Healing**) เมื่อมีเราเตอร์จุดใดจุดหนึ่งในโรงงานเสียหายผ่านอัลกอริทึม AODV
*   **หน้าที่อุปกรณ์:** Coordinator (ผู้สร้างและดูแลกุญแจเข้ารหัส), Router (ผู้ส่งต่อข้อมูลและอนุญาตให้อุปกรณ์ปลายทางเชื่อมต่อ), End Device (เซ็นเซอร์ปลายทางที่ตื่นนอนตามรอบเวลา ปิดระบบวิทยุช่วยประหยัดไฟ)

#### ส่วนที่ 6.5: LoRa / LoRaWAN
*   **กลไก CSS Modulation:** อธิบายกระบวนการเปลี่ยนระดับความถี่ (Chirp) และผลกระทบของการกำหนดค่าพารามิเตอร์ของระบบวิทยุ:
    *   *Spreading Factor (SF):* SF7 (ใกล้ เร็ว) ถึง SF12 (ไกล ช้า กินไฟสูง)
    *   *Coding Rate (CR):* อัตราส่วนของการเข้ารหัสแก้ปัญหาความผิดพลาด (เช่น 4/5, 4/8)
*   **LoRaWAN Classes:** รายละเอียด Class A (ตื่นเฉพาะตอนส่ง Uplink เพื่อรับ Downlink ป้องกันการใช้พลังงาน), Class B (ตื่นตามจังหวะรับสัญญาณ Beacon), และ Class C (เปิดเครื่องรับสัญญาณตลอดเวลา)
*   **ข้อจำกัดการใช้งาน:** ข้อกำหนดกำลังส่งและช่องสัญญาณตามกฎหมาย รวมถึงกฎ **Duty Cycle (1%)** และนโยบายความเท่าเทียมในเครือข่าย TTN (**Fair Access Policy**) ที่จำกัดการส่งไม่เกิน 30 วินาทีต่อวันต่อโหนด

#### ส่วนที่ 6.6: Cellular IoT (NB-IoT, LTE-M, 5G)
*   **ประเด็นการประหยัดพลังงาน:** การปรับจูนระดับลึกด้วยฟังก์ชัน **PSM (Power Saving Mode)** และ **eDRX** เพื่อลดพลังงานของชิปวิทยุระบบมือถือ
*   **ประเภท 5G สำหรับอุปกรณ์อุตสาหกรรม:** การจำแนกประเภทบริการ ได้แก่ **eMBB** (เน้นความเร็วสูงสุด), **mMTC** (เน้นการเชื่อมต่อหนาแน่น), และ **URLLC** (เน้นความเสถียรและความหน่วงระดับมิลลิวินาทีสำหรับการควบคุมความเร็วสูง)

#### ส่วนที่ 6.8: กรณีศึกษาทางวิศวกรรม (Engineering Case Studies)
*   **มอเตอร์ไฟฟ้า ID Fan (Predictive Maintenance):** วิเคราะห์สัญญาณความเร็วสูง 5 kHz การทำ FFT ต้องการขนาดทราฟฟิกข้อมูล 8 KB ทุกๆ 10 วินาที จึงจำใจต้องระบุใช้ **Wi-Fi** ซึ่งมีกำลังไฟพอนอกอาคารและท่อสัญญาณเพียงพอ
*   **สายพานลำเลียงคลังสินค้าอัตโนมัติ (Conveyor Belt):** วิเคราะห์ความต้องการส่งข้อมูลขนาดเล็กแต่เป็นจำนวนมาก (~120 โหนด) กระจายตามโรงงานที่มีผนังเหล็กบดบังคลื่นวิทยุได้ง่าย เลือกใช้ **Zigbee Mesh** เพื่อส่งผ่านข้อมูลแบบ Multi-hop และมีระบบ Self-Healing ซ่อมแซมระบบได้ทันที
*   **แปลงปาล์มน้ำมัน 200 เอเคอร์ (Smart Irrigation):** วิเคราะห์ขอบเขตพื้นที่ป่าปาล์มที่มีพฤติกรรมดูดซับคลื่นวิทยุความถี่สูงได้ดี จึงคัดเลือกคลื่นวิทยุย่าน Sub-GHz อย่าง **LoRaWAN** ที่มอดดูเลตแบบ CSS เพื่อลดการลดทอน และใช้อุปกรณ์ Class A และ B ร่วมกับการใช้แบตเตอรี่และโซลาร์เซลล์ขนาดเล็ก

## 3. ขั้นตอนการตรวจสอบความถูกต้อง (Verification Plan)

เพื่อพิสูจน์ทราบว่าแผนและโค้ดตัวอย่างที่เสนอแนะมีความถูกต้อง ปราศจากข้อผิดพลาดในการแปลและเขียนโปรแกรม:

1.  **การคอมไพล์โค้ดในเชิงฟิสิกส์ (ESP32 C++ Code Compilation):**
    *   *วิธีการ:* นำโค้ดในหัวข้อที่ 6.2.6 (Wi-Fi Web Server) และ 6.3.4 (BLE GATT Server) ไปทำการคอมไพล์ผ่านซอฟต์แวร์ Arduino IDE (หรือ VS Code ร่วมกับ PlatformIO) โดยกำหนดเป้าหมายอุปกรณ์บอร์ดเป็น `ESP32 Dev Module`
    *   *เกณฑ์การยืนยัน:* คอมไพล์ผ่านสมบูรณ์โดยไม่แสดงผลลัพธ์ข้อผิดพลาดเรื่อง Syntax หรือขาดแคลน Libraries ปลายทาง (ในโค้ดใช้เฉพาะ Libraries มาตรฐานในตัว ESP32 Core: `WiFi.h`, `BLEDevice.h`, `BLEUtils.h`, `BLEServer.h`, `BLE2902.h`)
2.  **การทดสอบเสมือนจริง (Simulator / Wokwi Verification):**
    *   *วิธีการ:* นำโค้ด Wi-Fi Setup ในส่วน 6.2.6 ไปทดสอบทำงานบนแพลตฟอร์ม Wokwi simulator
    *   *เกณฑ์การยืนยัน:* ESP32 เสมือนจำลองสามารถเชื่อมต่อ Wi-Fi เครือข่ายของ Wokwi (`Wokwi-GUEST`) และสามารถตอบรับคำสั่ง GET Request จากเบราว์เซอร์จำลองได้เป็นผลสำเร็จ
3.  **การตรวจสอบความสอดคล้องตาม PROJECT.md:**
    *   *เกณฑ์การยืนยัน:* เนื้อหาภาษาไทยมีความเป็นวิชาการและเป็นทางการ เหมาะสำหรับนิสิตคณะวิศวกรรมศาสตร์ ชั้นปีที่ 1 และโครงสร้างหัวข้อ Markdown ตรงกับรูปแบบระบบแฟ้มในโครงงาน
