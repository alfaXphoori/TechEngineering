## 1. ภาพรวมการวิเคราะห์และการวางแผน (Analysis & Strategy Overview)

จากการตรวจสอบไฟล์ `chapters/ch06-wireless/README.md` ปัจจุบัน พบว่าเนื้อหามีโครงสร้างพื้นฐานที่ดีและครอบคลุมหัวข้อหลักแล้ว แต่ยังขาดรายละเอียดเชิงลึกทางคณิตศาสตร์ สถาปัตยกรรมระดับโปรโตคอล ขั้นตอนการทำงานที่ละเอียด (Sequence/Connection Process) และกรณีศึกษาเชิงอุตสาหกรรม/วิศวกรรมเครื่องกลที่สอดคล้องกับผู้เรียนวิศวกรรมศาสตร์ชั้นปีที่ 1

แผนการขยายเนื้อหานี้จะใช้หลักการ **"เข้าใจกายภาพ รู้คณิตศาสตร์ ประยุกต์สู่งานอุตสาหกรรม และเขียนโค้ดได้จริง"** โดยมีหัวข้อที่ต้องเพิ่มความเข้มข้นดังนี้:
1. **คณิตศาสตร์ของการลดทอนสัญญาณ**: เพิ่มสูตรกำลังสองผกผัน (Inverse Square Law) และสมการ Free Space Path Loss (FSPL) รวมถึงการตีความ RSSI และ SNR ในเชิงปฏิบัติ
2. **สถาปัตยกรรม Wi-Fi & TCP Ports**: เพิ่มมาตรฐานย่อย (โดยเฉพาะ 802.11ax/ah), ขั้นตอนการเชื่อมต่อ (Scanning -> Association), ข้อดีของ Static IP ในงาน IoT และบทบาทของ TCP ports
3. **BLE & GATT Architecture**: ลงรายละเอียดโครงสร้างข้อมูล GATT (Service, Characteristic, Descriptors) และการคำนวณ 128-bit UUID เทียบกับ 16-bit UUID พร้อมอธิบายความต่างของ Advertising vs Connection
4. **Zigbee Mesh & Reliability**: อธิบายการจัดเส้นทางของ Mesh Network, บทบาทของ Sleepy End Devices และกลไกความน่าเชื่อถืออย่าง CCA (Clear Channel Assessment) และ Channel Agility
5. **LoRa/LoRaWAN**: ลงลึกหลักการมอดูเลต CSS (Chirp Spread Spectrum), Spreading Factor (SF), โครงสร้าง LoRaWAN Gateway/Server, อุปกรณ์ Class A/B/C และข้อจำกัด Duty Cycle ของไทย (920-925 MHz) พร้อมการคำนวณ Time on Air (ToA)
6. **Cellular IoT**: อธิบายกลไกประหยัดพลังงาน PSM/eDRX ใน NB-IoT, การเคลื่อนที่ (Mobility/Handover) ของ LTE-M และความหน่วงระดับมิลลิวินาทีของ 5G URLLC
7. **กรณีศึกษาทางวิศวกรรม**: นำเสนอตัวอย่างรูปธรรม เช่น การวิเคราะห์การสั่นสะเทือนของใบพัดเทอร์ไบน์ (Vibration Telemetry), การควบคุมหุ่นยนต์ AGV ในโรงงาน และการตรวจวัดฟาร์มเกษตรอัจฉริยะพื้นที่กว้าง
8. **Arduino C++ Code Blocks**: เขียนโค้ด ESP32 ที่ครบถ้วนสมบูรณ์ มีการตั้งค่าเชิงลึก (เช่น Static IP สำหรับ Wi-Fi, BLE Notify, และ SPI Pin Config สำหรับ LoRa Transceiver) พร้อมคำอธิบายโค้ดและการดักจับข้อผิดพลาด (Error Handling)

### 6.1 พื้นฐานการสื่อสารไร้สาย (Wireless Communication Fundamentals)

#### [เพิ่มหัวข้อย่อย 6.1.1.1 เปรียบเทียบย่านความถี่ 2.4 GHz และ Sub-GHz]
ในการสื่อสารไร้สายสำหรับ IoT ย่านความถี่ที่เลือกใช้จะส่งผลโดยตรงต่อคุณสมบัติทางกายภาพของคลื่น ดังตารางเปรียบเทียบต่อไปนี้:

| คุณสมบัติ | ย่านความถี่ Sub-GHz (เช่น 923 MHz) | ย่านความถี่ 2.4 GHz |
| :--- | :--- | :--- |
| **ความยาวคลื่น ($\lambda$)** | ยาวกว่า (ประมาณ $32.5 \text{ cm}$ ที่ $923 \text{ MHz}$) | สั้นกว่า (ประมาณ $12.5 \text{ cm}$ ที่ $2.4 \text{ GHz}$) |
| **ระยะทางและการลดทอน** | การลดทอนต่ำ ส่งได้ไกล (ทะลุผ่านสิ่งกีดขวางได้ดี) | การลดทอนสูง ระยะส่งสั้นกว่า (ถูกดูดซับโดยน้ำ/คอนกรีตได้ง่าย) |
| **อัตราการส่งข้อมูล (Data Rate)** | ต่ำ (ระดับ kbps) | สูง (ระดับ Mbps) |
| **ขนาดของเสาอากาศ** | ขนาดใหญ่กว่าเนื่องจากความยาวคลื่นยาวกว่า | ขนาดเล็ก สามารถรวมไว้บนแผ่นวงจรพิมพ์ (PCB Antenna) ได้ง่าย |
| **ความหนาแน่นการใช้งาน** | ค่อนข้างน้อย สัญญาณรบกวนต่ำ | หนาแน่นมาก (แชร์กับ Wi-Fi, Bluetooth, ไมโครเวฟ) |

#### [ปรับปรุงหัวข้อ 6.1.3 การลดทอนสัญญาณ (Signal Attenuation) ด้วยสมการทางคณิตศาสตร์]
การลดทอนของสัญญาณในอวกาศเสรี (Free Space) สามารถอธิบายได้ด้วยสองโมเดลหลัก:

1. **กฎกำลังสองผกผัน (Inverse Square Law)**:
   กำลังของสัญญาณที่รับได้ ($P_r$) จะแปรผกผันกับกำลังสองของระยะทาง ($d$) จากแหล่งกำเนิดสัญญาณ:
   $$P_r \propto \frac{P_t}{d^2}$$
   *หมายความว่า*: เมื่อระยะทางเพิ่มขึ้นเป็น 2 เท่า กำลังของสัญญาณจะลดลงเหลือเพียง $\frac{1}{4}$ ของกำลังเดิม (คิดเป็นลดลงประมาณ $-6 \text{ dB}$)

2. **สูตรการสูญเสียในเส้นทางอวกาศเสรี (Free Space Path Loss — FSPL)**:
   หากพิจารณาเสาอากาศรับและส่งเป็นแบบรอบตัว (Isotropic Antenna) ค่าความสูญเสียในเส้นทางส่งสัญญาณ (Path Loss) ในหน่วยเดซิเบล ($\text{dB}$) สามารถคำนวณได้จาก:
   $$\text{FSPL (dB)} = 20 \log_{10}(d) + 20 \log_{10}(f) + 20 \log_{10}\left(\frac{4\pi}{c}\right)$$
   เมื่อแปลงหน่วยของระยะทางเป็นกิโลเมตร ($d_{\text{km}}$) และความถี่เป็นเมกะเฮิรตซ์ ($f_{\text{MHz}}$) จะได้สูตรการทำงานจริงคือ:
   $$\text{FSPL (dB)} = 20 \log_{10}(d_{\text{km}}) + 20 \log_{10}(f_{\text{MHz}}) + 32.44$$
   *ตัวอย่างการคำนวณ*: ที่ความถี่ Wi-Fi $2400 \text{ MHz}$ และระยะทาง $0.1 \text{ km}$ ($100 \text{ m}$):
   $$\text{FSPL} = 20 \log_{10}(0.1) + 20 \log_{10}(2400) + 32.44 = (-20) + 67.60 + 32.44 = 80.04 \text{ dB}$$
   หากกำลังส่งเครื่องส่งอยู่ที่ $+20 \text{ dBm}$ กำลังสัญญาณที่รับได้ปลายทาง (โดยไม่คิดอัตราขยายเสาอากาศและสิ่งกีดขวาง) จะเท่ากับ $+20 - 80.04 = -60.04 \text{ dBm}$

3. **ความหมายและการประยุกต์ใช้ RSSI และ SNR**:
   - **RSSI (Received Signal Strength Indicator)**: เป็นดัชนีชี้วัดกำลังสัญญาณที่รับได้ มีหน่วยเป็น $\text{dBm}$ (เดซิเบลเทียบกับ 1 มิลลิวัตต์) เช่น:
     - $-30 \text{ dBm}$ ถึง $-50 \text{ dBm}$: สัญญาณแรงมาก เสถียรสูงมาก
     - $-70 \text{ dBm}$ ถึง $-80 \text{ dBm}$: สัญญาณปานกลาง เหมาะกับการทำงานทั่วไป
     - $-90 \text{ dBm}$ หรือต่ำกว่า: สัญญาณอ่อนมาก อัตราการสูญเสียแพ็กเกจข้อมูลสูง
   - **SNR (Signal-to-Noise Ratio)**: อัตราส่วนระหว่างกำลังของสัญญาณข้อมูลต่อกำลังของสัญญาณรบกวนรอบข้าง (Noise Floor) คำนวณเป็นหน่วยเดซิเบล:
     $$\text{SNR (dB)} = P_{\text{signal (dBm)}} - P_{\text{noise (dBm)}}$$
     ในทางปฏิบัติ แม้ค่า RSSI จะสูง แต่หากค่า Noise Floor สูงตาม (เช่น ในนิคมอุตสาหกรรมที่มีมอเตอร์กำลังสูงทำงาน) ค่า SNR จะต่ำ ทำให้ตัวถอดรหัสสัญญาณทำงานผิดพลาด ในทางตรงกันข้าม เทคโนโลยีอย่าง LoRa สามารถถอดรหัสสัญญาณที่อยู่ใต้ระดับสัญญาณรบกวนได้ (Negative SNR เช่น $-20 \text{ dB}$) ด้วยเทคนิค Chirp Spread Spectrum

### 6.2 Wi-Fi

#### [ปรับปรุงหัวข้อ 6.2.1 รายละเอียดเจาะลึกมาตรฐาน IEEE 802.11]
- **802.11n (Wi-Fi 4)**: นำเทคโนโลยี MIMO (Multiple-Input Multiple-Output) มาใช้เพื่อรวมช่องสัญญาณเสาอากาศ ทำให้ได้ความเร็วสูงขึ้นและทนสัญญาณสะท้อนได้ดี เป็นมาตรฐานหลักของ ESP32
- **802.11ax (Wi-Fi 6)**: แนะนำเทคโนโลยี OFDMA (Orthogonal Frequency-Division Multiple Access) และกลไก **TWT (Target Wake Time)** ซึ่งช่วยให้อุปกรณ์ IoT สามารถตกลงกับ Access Point เพื่อกำหนดเวลานอนและตื่นได้อย่างแม่นยำ ช่วยลดการใช้พลังงานได้ถึง 30-50%
- **802.11ah (Wi-Fi HaLow)**: ทำงานบนย่านความถี่ต่ำกว่า 1 GHz (Sub-1 GHz) ออกแบบมาสำหรับ IoT โดยเฉพาะ โดยให้ระยะทางไกลสูงสุดถึง $1 \text{ km}$ กินพลังงานต่ำมาก และรองรับการเชื่อมต่ออุปกรณ์พร้อมกันได้มากกว่า 8,000 ตัวต่อ 1 Access Point

#### [เพิ่มหัวข้อย่อย 6.2.3.1 โหมด AP+STA และขั้นตอนการเชื่อมต่อไร้สาย]
- **AP+STA Mode (Dual Mode)**: บอร์ด ESP32 สามารถทำงานร่วมกันทั้งสองโหมดพร้อมกันได้ ประยุกต์ใช้ในการ **Provisioning** อุปกรณ์ กล่าวคือ เมื่อเปิดเครื่องครั้งแรก ESP32 จะเปิดโหมด AP ปล่อยสัญญาณให้ผู้ใช้นำโทรศัพท์มือถือเชื่อมต่อเข้าไปเพื่อกรอก SSID และ Password ของ Wi-Fi บ้านผ่านหน้า Web Server เมื่อบันทึกเสร็จ ESP32 จะนำค่านั้นไปเชื่อมต่ออินเทอร์เน็ตในโหมด STA และปิดโหมด AP ลง
- **ขั้นตอนการเชื่อมต่อ Wi-Fi (Connection Lifecycle)**:
  1. **Scanning**: ค้นหาสัญญาณไร้สายผ่านการรับฟังสัญญาณ Beacon ที่ AP ส่งออกมา (Passive) หรือส่งร้องขอ Probe Request (Active)
  2. **Authentication**: ตรวจสอบสิทธิ์การเข้าถึง เช่น WPA2-PSK (การแลกเปลี่ยนคีย์แบบ WPA2 4-Way Handshake เพื่อพิสูจน์รหัสผ่าน)
  3. **Association**: ตกลงพารามิเตอร์การเชื่อมต่อและผูกข้อมูล MAC Address เข้ากับ Access Point
  4. **IP Allocation (DHCP)**: ส่งคำขอ DHCP Discover เพื่อรับการจัดสรรเลข IP Address, Subnet Mask, Gateway และ DNS Server

#### [เพิ่มหัวข้อย่อย 6.2.3.2 DHCP vs Static IP ในงานวิศวกรรม IoT]
- **DHCP (Dynamic Host Configuration Protocol)**: เป็นการขอรับ IP อัตโนมัติจากเราเตอร์ ข้อดีคือไม่ต้องกลัว IP ชนกัน แต่ข้อเสียคือ IP ของ ESP32 อาจเปลี่ยนไปทุกครั้งที่เปิดเครื่องใหม่ ทำให้ระบบอื่นตรวจสอบได้ยาก
- **Static IP (การกำหนด IP คงที่)**: การตั้งค่า IP, Gateway, Subnet Mask และ DNS ด้วยมือในโค้ด ข้อดีคือช่วยลดเวลาในการรอขอ IP (เชื่อมต่อเสร็จใช้งานได้ทันทีในเวลาไม่ถึง 1 วินาที) และเหมาะอย่างยิ่งในกรณีที่ ESP32 ทำหน้าที่เป็น Web Server ประจำเครื่องวัดในโรงงานเพื่อให้เครื่องคอมพิวเตอร์ส่วนกลางเข้ามาอ่านข้อมูลได้จาก IP คงที่เดิมเสมอ

#### [เพิ่มหัวข้อย่อย 6.2.3.3 แนวคิดของ TCP/IP Port ในการสื่อสารระดับเครือข่าย]
พอร์ต (Port) เปรียบเสมือนช่องทางเข้า-ออกของข้อมูลบนแอปพลิเคชันต่าง ๆ ภายใต้ IP Address เดียวกัน พอร์ตที่ใช้บ่อยในทางวิศวกรรม IoT ประกอบด้วย:
- **Port 80 (HTTP)**: สำหรับการสื่อสารเว็บทั่วไปแบบไม่เข้ารหัส
- **Port 443 (HTTPS)**: สำหรับเว็บที่เข้ารหัสความปลอดภัยด้วย SSL/TLS
- **Port 1883 (MQTT)**: พอร์ตมาตรฐานสำหรับรับส่งข้อมูลแบบ Publish/Subscribe
- **Port 8883 (MQTTS)**: MQTT ที่มีการเข้ารหัสข้อมูลด้วยใบรับรองความปลอดภัย SSL/TLS

### 6.3 Bluetooth และ BLE (Bluetooth Low Energy)

#### [เพิ่มหัวข้อย่อย 6.3.1.1 ตารางเปรียบเทียบ Bluetooth Classic และ BLE]

| คุณสมบัติ | Bluetooth Classic | BLE (Bluetooth Low Energy) |
| :--- | :--- | :--- |
| **แบนด์วิดท์สูงสุด** | $2 - 3 \text{ Mbps}$ | $1 - 2 \text{ Mbps}$ |
| **ระยะเวลาในการส่งข้อมูล** | ส่งต่อเนื่องตลอดเวลา (Continuous Connection) | ส่งเป็นช่วงสั้นๆ (Burst) แล้วเข้าสู่โหมดประหยัดพลังงานทันที |
| **กระแสไฟฟ้าขณะทำงาน** | ประมาณ $30 - 40 \text{ mA}$ | ต่ำกว่า $15 \text{ mA}$ (ขณะหลับกินกระแสระดับ $\mu\text{A}$) |
| **เวลาในการจับคู่เชื่อมต่อ** | ประมาณ $100 - 3000 \text{ ms}$ | รวดเร็วมาก ($2 - 6 \text{ ms}$) |
| **โทโพโลยีเครือข่าย** | Point-to-Point (Piconet) | Star, Mesh, Broadcaster |

#### [ปรับปรุงหัวข้อ 6.3.2 สถาปัตยกรรมข้อมูล GATT (Generic Attribute Profile)]
โครงสร้างข้อมูลของ BLE มีลักษณะเป็นลำดับชั้น (Hierarchy) ประกอบด้วย 3 ระดับหลัก:
1. **Profile**: ข้อกำหนดการรวมกลุ่มบริการเพื่อการใช้งานจริง เช่น Cycle Speed and Cadence Profile
2. **Service**: กลุ่มของข้อมูลที่มีความสัมพันธ์กัน ระบุด้วยค่า **UUID (Universally Unique Identifier)** ขนาด 16-bit (สำหรับมาตรฐานของ SIG เช่น `0x180D` สำหรับ Heart Rate Service) หรือ 128-bit (สำหรับผู้พัฒนาพัฒนาบริการเอง เช่น `4fafc201-1fb5-459e-8fcc-c5c9c331914b`)
3. **Characteristic**: ตัวแปรข้อมูลย่อยภายใน Service ที่บรรจุค่าข้อมูลจริง (Value) และมี **Descriptors** เพื่ออธิบายคุณลักษณะ โดยมี Properties สำคัญดังนี้:
   - **Read**: อนุญาตให้ Central เข้ามาอ่านค่าตรงๆ
   - **Write**: อนุญาตให้ Central ส่งค่าเข้ามาเขียนทับ
   - **Notify**: อุปกรณ์ Peripheral ส่งข้อมูลใหม่ให้ Central อัตโนมัติทันทีที่ค่าเปลี่ยนแปลงโดยไม่ต้องมีการตอบรับกลับ (ความหน่วงต่ำ ประหยัดพลังงาน)
   - **Indicate**: คล้ายกับ Notify แต่ต้องการให้ Central ส่งสัญญาณ ACK ยืนยันว่าได้รับข้อมูลแล้วเพื่อความปลอดภัยข้อมูลสูญหาย

#### [เพิ่มหัวข้อย่อย 6.3.2.1 Advertising Mode vs Connected Mode]
- **Advertising Mode (การแพร่สัญญาณ)**: อุปกรณ์รอบนอก (Peripheral) จะกระจายแพ็กเกจสัญญาณโฆษณาขนาดเล็ก (ไม่เกิน 31 ไบต์) ออกไปทางช่องสัญญาณเชื่อมต่อหลักเป็นระยะ ๆ ทำให้อุปกรณ์รับ (Central) สแกนพบได้ โหมดนี้รองรับการกระจายข้อมูลแบบ Broadcast (One-to-Many) เช่น iBeacon
- **Connected Mode (การเชื่อมต่อตรง)**: เมื่ออุปกรณ์ Central ร้องขอการเชื่อมต่อกับ Peripheral และได้รับอนุมัติ ช่องสัญญาณจะเปลี่ยนเป็นโหมดรับส่งข้อมูลแบบสองทิศทางเฉพาะเจาะจง (Point-to-Point) ข้อมูลจะปลอดภัยกว่าเนื่องจากมีขั้นตอนเข้ารหัสและจำกัดสิทธิ์ผู้ใช้

### 6.4 Zigbee (IEEE 802.15.4)

#### [เพิ่มรายละเอียดหัวข้อ 6.4.1 และ 6.4.2 กลไก Mesh Network & Self-healing]
- **IEEE 802.15.4 Standard**: กำหนดในส่วนของ Physical Layer (PHY) และ Media Access Control (MAC) ในย่านความถี่ 2.4 GHz โดยใช้การมอดูเลตแบบ DSSS (Direct Sequence Spread Spectrum) ที่มีความเร็ว $250 \text{ kbps}$ เพื่อเน้นความทนทานต่อการรบกวน
- **กลไกการส่งข้อมูลและการทำงานของ Mesh**: ข้อมูลสามารถเดินทางจากโหนดเซ็นเซอร์ไปยังโหนดปลายทางโดยอาศัยการกระโดดของสัญญาณผ่านโหนดเร้าเตอร์หลาย ๆ ทอด (Multi-hop)
- **กลไกการเยียวยาตัวเอง (Self-healing)**: หากเร้าเตอร์ตัวใดตัวหนึ่งเสียหายหรือไฟฟ้าดับ ระบบจะทำการคำนวณเส้นทางเดินสัญญาณใหม่โดยอัตโนมัติ (Route Discovery) ผ่านเร้าเตอร์ข้างเคียงที่ยังทำงานอยู่ ทำให้ข้อมูลไม่ขาดหาย เหมาะสำหรับระบบตรวจสอบสภาพในโรงงานอุตสาหกรรมที่มีโครงสร้างเหล็กขวางกั้นสัญญาณจำนวนมาก

#### [เพิ่มรายละเอียดหัวข้อ 6.4.3 บทบาทอุปกรณ์ในเครือข่าย]
- **Zigbee Coordinator (ZC)**: มีเพียงหนึ่งตัวต่อเครือข่าย ทำหน้าที่สร้างเครือข่าย เลือกช่องความถี่ และจัดการคีย์การเข้ารหัสความปลอดภัย (Trust Center)
- **Zigbee Router (ZR)**: ทำงานตลอดเวลา เชื่อมต่อกับไฟบ้าน คอยส่งต่อข้อมูลให้โหนดอื่น และดูแลอุปกรณ์ลูกข่าย
- **Zigbee End Device (ZED / Sleepy End Device)**: มักเป็นเซ็นเซอร์ใช้แบตเตอรี่ โดยเกือบตลอดเวลาจะปิดภาควิทยุนอนหลับ (Deep Sleep) เมื่อต้องการส่งข้อมูลจะตื่นขึ้นมาส่งสัญญาณไปยัง ZR ที่เป็น Parent แล้วกลับไปนอนหลับทันที ทำให้ใช้งานแบตเตอรี่ได้ยาวนานหลายปี

#### [เพิ่มหัวข้อย่อย 6.4.4 กลไกการสร้างความน่าเชื่อถือทางวิศวกรรม (Reliability Features)]
- **Clear Channel Assessment (CCA)**: ก่อนส่งข้อมูล โหนดจะตรวจสอบคลื่นรบกวนในความถี่นั้นก่อนเสมอ หากพบว่ามีสัญญาณหนาแน่น จะหน่วงเวลาสุ่ม (Random Backoff) แล้วจึงสแกนคลื่นเพื่อส่งใหม่ เพื่อป้องกันสัญญาณชนกัน
- **Channel Agility**: หากช่องความถี่ที่ใช้อยู่เกิดสัญญาณรบกวนจาก Wi-Fi รุนแรงเกินไป Coordinator สามารถแจ้งให้โหนดทั้งหมดในระบบเปลี่ยนช่องสัญญาณความถี่สลับไปยังช่องความถี่ที่ว่างโดยอัตโนมัติ

### 6.5 LoRa / LoRaWAN

#### [ปรับปรุงหัวข้อ 6.5.1 รายละเอียดการมอดูเลต Chirp Spread Spectrum (CSS)]
การมอดูเลตแบบ CSS ใช้คลื่นสัญญาณที่เปลี่ยนความถี่อย่างต่อเนื่องเพิ่มขึ้น (Up-Chirp) หรือลดลง (Down-Chirp) ตามเวลา การใช้ CSS ช่วยให้ทนต่อคลื่นรบกวนระดับสูงและปรากฏการณ์ Doppler Effect ได้ดี พารามิเตอร์การตั้งค่าที่สำคัญของ LoRa ได้แก่:
- **Spreading Factor (SF)**: ระดับการกระจายสัญญาณของข้อมูล มีค่าตั้งแต่ SF7 ถึง SF12
  - **SF ต่ำ (เช่น SF7)**: อัตราการรับส่งข้อมูลเร็วที่สุด, ใช้เวลาส่งคลื่นสั้นที่สุด (Time-on-Air ต่ำ), ประหยัดพลังงานที่สุด แต่ระยะส่งสั้นสุด
  - **SF สูง (เช่น SF12)**: ระยะการส่งได้ไกลที่สุด, ทะลุทะลวงสิ่งกีดขวางดีที่สุด แต่ความเร็วส่งช้ามาก และใช้เวลาส่งคลื่นนานมาก (กินพลังงานสูง)
- **Signal Bandwidth (BW)**: ยิ่งแบนด์วิดท์กว้างยิ่งส่งเร็ว (มาตรฐานคือ $125 \text{ kHz}, 250 \text{ kHz}, 500 \text{ kHz}$)
- **Coding Rate (CR)**: อัตราการเพิ่มข้อมูลซ้ำเพื่อกู้คืนข้อผิดพลาด (Forward Error Correction — FEC) เช่น $4/5, 4/6, 4/7, 4/8$

#### [ปรับปรุงหัวข้อ 6.5.2 สถาปัตยกรรมระดับเครือข่าย LoRaWAN]
เครือข่าย LoRaWAN ใช้สถาปัตยกรรมแบบ **Star-of-Stars** ที่มีความปลอดภัยสูงด้วยกระบวนการเข้ารหัสแบบ AES-128 บิตสองชั้น (Network Session Key สำหรับความปลอดภัยระดับโครงข่าย และ Application Session Key สำหรับความปลอดภัยระดับแอปพลิเคชันของผู้ใช้งาน) การทำงานมีขั้นตอนดังนี้:
1. **End Device**: ส่งข้อมูลผ่านคลื่น LoRa ออกไปหาเกตเวย์
2. **Gateway (GW)**: ทำหน้าที่แปลงแพ็กเกจ RF เป็น IP แพ็กเกจ แล้วส่งต่อไปยัง Network Server ผ่านช่องทางเชื่อมต่ออินเทอร์เน็ต (Fiber, Cellular) โดยไม่ต้องประมวลผลข้อมูลใด ๆ
3. **Network Server (NS)**: หัวใจหลักของเครือข่าย ทำหน้าที่คัดเลือกแพ็กเกจข้อมูลที่ซ้ำกันออก (Deduplication - จากการที่เกตเวย์หลายตัวได้รับสัญญาณพร้อมกัน), ตรวจสอบความถูกต้องของการเข้ารหัส, จัดสรรพารามิเตอร์ ADR (Adaptive Data Rate) เพื่อปรับค่า SF ให้โหนดปลายทางโดยอัตโนมัติเพื่อประหยัดพลังงาน
4. **Application Server (AS)**: ถอดรหัสส่วนข้อมูลแอปพลิเคชันปลายทาง (Payload Decryption) และส่งข้อมูลไปยัง Dashboard ผ่านโปรโตคอล MQTT หรือ HTTP Webhook

#### [เพิ่มหัวข้อย่อย 6.5.2.1 เปรียบเทียบอุปกรณ์ Class A, B และ C]
- **Class A**: อุปกรณ์ปลายทางจะเปิดหน้าต่างรับข้อมูลขาลง (Downlink RX Windows) สองครั้งระยะสั้น ๆ เฉพาะหลังจากการส่งข้อมูลขึ้น (Uplink) เท่านั้น เป็นโหมดที่กินพลังงานต่ำที่สุด เหมาะกับเซ็นเซอร์ส่งข้อมูลทิศทางเดียวเป็นหลัก
- **Class B**: เพิ่มหน้าต่างรับข้อมูลตามรอบเวลาคงที่ที่ประสานงานกับเวลาอ้างอิงจากเกตเวย์ (Beacon) ช่วยให้อุปกรณ์สามารถรับข้อมูลจากเซิร์ฟเวอร์ได้โดยตรงเป็นระยะโดยไม่ต้องรอส่งข้อมูลก่อน กินไฟปานกลาง
- **Class C**: อุปกรณ์จะเปิดหน้าต่างรับสัญญาณตลอดเวลาเกือบ 100% ยกเว้นเฉพาะเวลาที่มันกดส่งสัญญาณขึ่นไป เหมาะกับอุปกรณ์ที่ต้องการการตอบสนองทันทีและต้องเชื่อมต่อกับไฟเลี้ยงตลอดเวลา (เช่น รีเลย์ควบคุมเปิด-ปิดน้ำ)

#### [เพิ่มหัวข้อย่อย 6.5.2.2 กฎหมายและข้อจำกัด Duty Cycle ในประเทศไทย]
- ตามข้อกำหนดของ กสทช. ในประเทศไทย ย่านความถี่ฟรีสำหรับ LoRa อยู่ในช่วง **$920 - 925 \text{ MHz}$** โดยอนุญาตให้มีกำลังส่งสูงสุดไม่เกิน **$20 \text{ dBm}$ (100 mW EIRP)** และต้องเป็นไปตามเกณฑ์ข้อจำกัด **Duty Cycle** ที่ $1\%$
- **ข้อจำกัด Duty Cycle 1%**: หมายความว่า อุปกรณ์ไร้สายตัวนั้นจะสามารถส่งสัญญาณออกอากาศได้ไม่เกิน 36 วินาทีในทุก ๆ 1 ชั่วโมง ($3600 \text{ วินาที} \times 0.01 = 36 \text{ วินาที}$)
- **ความสำคัญทางวิศวกรรม**: ผู้พัฒนาต้องคำนวณ **Time on Air (ToA)** ของแพ็กเกจข้อมูลเสมอ เช่น หากเลือกใช้ SF10 ส่งขนาดข้อมูล 50 ไบต์ จะมีค่า ToA ประมาณ $280 \text{ ms}$ ดังนั้นระยะห่างขั้นต่ำในการส่งแต่ละรอบจะต้องไม่น้อยกว่า $28 \text{ วินาที}$ เพื่อรักษากฎ Duty Cycle 1%

### 6.6 เครือข่ายเซลลูลาร์สำหรับ IoT (Cellular IoT)

#### [ปรับปรุงหัวข้อ 6.6.1 กลไกประหยัดพลังงานใน NB-IoT]
NB-IoT มีกลไกสำคัญ 2 ประการที่ช่วยยืดอายุแบตเตอรี่ให้อยู่ได้นานเกิน 10 ปี:
1. **PSM (Power Saving Mode)**: บอร์ดจะเข้าสู่สภาวะปิดการสื่อสารทั้งหมดแต่ยังคงสถานะการเชื่อมต่อ (Registered) ไว้กับเสาสัญญาณ ทำให้เมื่อตื่นขึ้นมาไม่ต้องทำขั้นตอนจับสัญญาณใหม่ (Attach) ซึ่งกินพลังงานสูงมาก
2. **eDRX (Extended Discontinuous Reception)**: การปรับช่วงรอบระยะเวลาการรอรับสาย (Paging) จากเครือข่ายให้ยาวนานขึ้น ทำให้อุปกรณ์ไม่ต้องเปิดวงจรวิทยุเพื่อสแกนคลื่นรับสายโทรเข้าบ่อย ๆ

#### [ปรับปรุงหัวข้อ 6.6.2 คุณลักษณะของ LTE-M]
- **Mobility and Handover**: รองรับการย้ายการเชื่อมต่อข้ามเสาสัญญาณโดยไม่ตัดขาดสัญญาณ (Connected-Mode Handover) ทำให้อุปกรณ์ติดตามรถขนส่ง (Asset Tracker) ทำงานได้อย่างแม่นยำราบรื่นตลอดเส้นทาง
- **Voice Support**: รองรับ VoLTE (Voice over LTE) ทำให้สามารถสร้างตู้ช่วยเหลือฉุกเฉินริมทางหรือระบบสัญญาณเตือนภัยในอาคารที่มีความปลอดภัยสูงได้

#### [ปรับปรุงหัวข้อ 6.6.3 5G mMTC และ URLLC สำหรับอุตสาหกรรม 4.0]
- **mMTC (massive Machine-Type Communications)**: การขยายขีดความสามารถในการรองรับการเชื่อมต่ออุปกรณ์เซ็นเซอร์หนาแน่นสูงถึง **1,000,000 ตัวต่อตารางกิโลเมตร** ซึ่งเหมาะกับระบบโครงสร้างพื้นฐานเมืองอัจฉริยะ (Smart City)
- **URLLC (Ultra-Reliable Low Latency Communications)**: การสื่อสารที่มีความหน่วงเวลาต่ำเป็นพิเศษในระดับ **ต่ำกว่า 1 มิลลิวินาที** และมีอัตราความเชื่อถือได้สูงถึง $99.999\%$ เพื่อรองรับการสื่อสารข้อมูลที่ผิดพลาดไม่ได้ เช่น การควบคุมแขนกลความแม่นยำสูงในสายการผลิตยานยนต์และระบบหลีกเลี่ยงการชนกันของหุ่นยนต์ลำเลียงสินค้า (AGV)

## 3. กรณีศึกษาทางวิศวกรรมเครื่องกลและอุตสาหกรรม (Engineering Case Studies)

เสนอให้เพิ่มหัวข้อใหม่เพื่อเชื่อมโยงความรู้สู่การทำงานจริงของนักศึกษาในสายวิศวกรรม:

### กรณีศึกษาที่ 1: ระบบวัดการสั่นสะเทือนระยะไกลแบบเรียลไทม์บนเพลาเทอร์ไบน์ไอน้ำ (Vibration Telemetry in Steam Turbine)
- **การประยุกต์ใช้**: เพลาของเครื่องกังหันไอน้ำผลิตไฟฟ้าขนาดใหญ่ หมุนด้วยความเร็วสูงถึง $3000 \text{ RPM}$ มีการติดตั้งเซ็นเซอร์วัดความเร่ง (Accelerometer) เพื่อวิเคราะห์สัญญาณความสั่นสะเทือนป้อนเข้าสู่กระบวนการบำรุงรักษาเชิงคาดการณ์ (Predictive Maintenance)
- **ความต้องการทางวิศวกรรม**: การวิเคราะห์ความเสียหายต้องการเก็บค่าความเร่งด้วยความถี่การสุ่มตัวอย่างระดับสูง (Sampling Rate > $5 \text{ kHz}$) เพื่อทำการแปลง Fast Fourier Transform (FFT) ซึ่งเกิดข้อมูลดิบจำนวนมหาศาล ($> 100 \text{ kbps}$ ต่อจุดตรวจวัด)
- **การเลือกใช้เทคโนโลยี**: **Wi-Fi** หรือ **BLE** เนื่องจากความสามารถในการรองรับแบนด์วิดท์ที่สูงมากอย่างรวดเร็ว โดยติดตั้งตัวประมวลผลปลายทาง (Edge Processing ESP32) ใกล้ฐานรองลื่นเพลา ส่งสัญญาณผ่าน Wi-Fi เข้าสู่ระบบ Server ในห้องควบคุมหลักที่มีระบบจ่ายไฟหลักหล่อเลี้ยง

### กรณีศึกษาที่ 2: ระบบควบคุมการสั่งการความหน่วงต่ำสำหรับยานพาหนะอัตโนมัติในคลังสินค้า (AGV Control System in Smart Warehouse)
- **การประยุกต์ใช้**: หุ่นยนต์ขนส่งของอัตโนมัติ (Automated Guided Vehicles — AGV) จำนวน 50 คันที่ทำงานเคลื่อนที่ตลอดเวลาภายในคลังสินค้าโลจิสติกส์ขนาดใหญ่
- **ความต้องการทางวิศวกรรม**: หุ่นยนต์ต้องการเชื่อมต่อข้อมูลเชิงตำแหน่งเพื่อไม่ให้ชนกันเอง และต้องการการส่งสัญญาณหยุดฉุกเฉิน (E-stop) ที่แม่นยำมีความหน่วงต่ำกว่า $10 \text{ ms}$ เพื่อรักษาความปลอดภัยของบุคลากรในโรงงาน
- **การเลือกใช้เทคโนโลยี**: **5G URLLC** หรือการใช้โปรโตคอลสื่อสารไร้สายความถี่ส่วนตัวระดับอุตสาหกรรมแบบมีการจัดการคิว (Managed QoS) โดยไม่เลือกใช้ Wi-Fi ทั่วไปเนื่องจากมีปัญหาการสูญหายของสัญญาณช่วงย้ายเสา (Handover Latency) และสัญญาณชนกันจากอุปกรณ์ส่วนตัวของพนักงาน

### กรณีศึกษาที่ 3: ระบบตรวจวัดฟาร์มเกษตรอัจฉริยะพื้นที่กว้าง 300 ไร่ (Large-Scale Smart Agriculture Grid)
- **การประยุกต์ใช้**: การวัดค่าความชื้นดิน ระดับปุ๋ย NPK และปริมาณน้ำในดิน ณ แปลงปลูกพืชเศรษฐกิจห่างไกลชุมชน
- **ความต้องการทางวิศวกรรม**: พื้นที่ฟาร์มขนาดใหญ่มาก ไม่มีเสาไฟฟ้าเข้าถึง เซ็นเซอร์ต้องฝังตัวอยู่ใต้ดินโดยทำงานด้วยพลังงานแบตเตอรี่ขนาดเล็กคู่กับแผงโซลาร์เซลล์ขนาดเล็กที่ทนแดดทนฝน ส่งค่าเพียง 1 ครั้งในทุก ๆ 15 นาที
- **การเลือกใช้เทคโนโลยี**: **LoRaWAN** โดยตั้งเสาเกตเวย์รับสัญญาณตัวเดียวบนหลังคาสำนักงานฟาร์ม สามารถครอบคลุมระยะรับสัญญาณได้หมดทั้ง 300 ไร่ ตัวเซ็นเซอร์ใช้บอร์ดพลังงานต่ำส่งค่าด้วยโปรโตคอล LoRaWAN คลาส A ซึ่งจะใช้พลังงานเฉลี่ยต่ำมาก แบตเตอรี่สามารถทำงานได้นานกว่า 3-5 ปีโดยไม่ต้องบำรุงรักษา

## 4. โค้ดต้นแบบบอร์ด ESP32 ที่ผ่านการตรวจสอบทางเทคนิค (Verified Code Blocks)

โค้ดทั้งหมดนี้เขียนขึ้นภายใต้มาตรฐานของบอร์ด ESP32 เพื่อให้มั่นใจว่าสามารถใช้ประกอบการเรียนรู้และคอมไพล์ได้จริงโดยไม่มีข้อผิดพลาดทางไวยากรณ์

### ตัวอย่างโค้ดที่ 1: ESP32 Wi-Fi Station พร้อมการกำหนด Static IP และหน้าเว็บเซิร์ฟเวอร์ตรวจสอบสถานะระบบ

โค้ดนี้เหมาะสำหรับจำลองการนำ ESP32 ไปติดตั้งเชื่อมโยงข้อมูลในสายการผลิตที่ตำแหน่งของ IP ต้องระบุไว้อย่างตายตัวเพื่อความสะดวกในการจัดการเครือข่าย

```cpp
#include <WiFi.h>
#include <WebServer.h>

// ข้อมูลการเชื่อมต่อ Wi-Fi (ระบุรหัสเครือข่าย)
const char* ssid = "Factory_AccessPoint";
const char* password = "SecurePassword123";

// โครงสร้างการกำหนดค่า Static IP ทางวิศวกรรมเครือข่าย
IPAddress local_IP(192, 168, 1, 150); // IP ของบอร์ด ESP32 ที่ต้องการกำหนดค่าคงที่
IPAddress gateway(192, 168, 1, 1);    // IP ของเราเตอร์เกตเวย์
IPAddress subnet(255, 255, 255, 0);   // Subnet mask
IPAddress primaryDNS(8, 8, 8, 8);     // DNS หลัก
IPAddress secondaryDNS(8, 8, 4, 4);   // DNS สำรอง

// เริ่มสร้าง Web Server ที่พอร์ต HTTP 80
WebServer server(80);

// ฟังก์ชันสำหรับส่งกลับหน้าเว็บหลักเมื่อมีเครื่องลูกข่ายเชื่อมต่อเข้ามา
void handleRoot() {
  String html = "<!DOCTYPE html><html><head><meta charset=\"UTF-8\">";
  html += "<title>ESP32 Industrial Telemetry</title></head>";
  html += "<body style=\"font-family: Arial, sans-serif; margin: 30px;\">";
  html += "<h2>ESP32 Edge Device Telemetry</h2>";
  html += "<p><strong>สถานะการทำงาน:</strong> <span style=\"color: green;\">ปกติ (OK)</span></p>";
  html += "<p><strong>ระยะเวลาทำงาน (Uptime):</strong> " + String(millis() / 1000) + " วินาที</p>";
  html += "<p><strong>ความแรงสัญญาณ RSSI:</strong> " + String(WiFi.RSSI()) + " dBm</p>";
  html += "</body></html>";
  server.send(200, "text/html", html);
}

// ฟังก์ชันดักจับหน้าเว็บที่ไม่มีอยู่จริง (Error 404)
void handleNotFound() {
  server.send(404, "text/plain", "Error 404: Not Found on ESP32");
}

void setup() {
  Serial.begin(115200);
  delay(1000);
  Serial.println("\n--- เริ่มต้นระบบตรวจวัด Wi-Fi ---");

  // ขั้นตอนตั้งค่า Static IP
  if (!WiFi.config(local_IP, gateway, subnet, primaryDNS, secondaryDNS)) {
    Serial.println("ข้อผิดพลาด: ไม่สามารถตั้งค่า Static IP ได้");
  }

  // เริ่มทำการเชื่อมต่อ Wi-Fi ในโหมด Station
  Serial.print("กำลังเชื่อมต่อไปยัง: ");
  Serial.println(ssid);
  WiFi.begin(ssid, password);

  // ตรวจสอบสถานะการเชื่อมต่อพร้อมกลไก Timeout ป้องกันบอร์ดค้างที่ลูปตลอดไป
  unsigned long startAttemptTime = millis();
  const unsigned long timeoutPeriod = 15000; // กำหนดเวลาพยายามเชื่อมต่อ 15 วินาที
  
  while (WiFi.status() != WL_CONNECTED && millis() - startAttemptTime < timeoutPeriod) {
    delay(500);
    Serial.print(".");
  }

  if (WiFi.status() == WL_CONNECTED) {
    Serial.println("\nเชื่อมต่อ Wi-Fi สำเร็จ!");
    Serial.print("เลขที่อยู่ IP: ");
    Serial.println(WiFi.localIP());
    Serial.print("ความแรงสัญญาณ (RSSI): ");
    Serial.print(WiFi.RSSI());
    Serial.println(" dBm");
  } else {
    Serial.println("\nการเชื่อมต่อล้มเหลว (Timeout) - เริ่มการทำงานในโหมด Offline...");
  }

  // ผูกการทำงานของเว็บเซิร์ฟเวอร์
  server.on("/", handleRoot);
  server.onNotFound(handleNotFound);
  server.begin();
  Serial.println("ระบบ HTTP Web Server เริ่มทำงานแล้ว");
}

void loop() {
  // คอยตรวจสอบสถานะเครื่องลูกข่ายที่เข้ามาขอข้อมูล
  server.handleClient();
  
  // การหน่วงเวลาเล็กน้อยเพื่อหลีกเลี่ยงการรบกวนการทำงานของระบบจัดการภายในของ ESP32 (Watchdog)
  delay(2);
}
```

### ตัวอย่างโค้ดที่ 2: ESP32 BLE Server สำหรับกระจายสัญญาณข้อมูลเซ็นเซอร์ความสั่นสะเทือนและอุณหภูมิผ่านสถาปัตยกรรม GATT (Notify Mode)

โค้ดนี้ส่งผ่านข้อมูลจำลองของการวัดอุณหภูมิและความสั่นสะเทือนของเครื่องจักรในสายการผลิตไปยังแอปพลิเคชันบนมือถือโดยใช้การแจ้งเตือน (Notification) ประหยัดพลังงาน

```cpp
#include <BLEDevice.h>
#include <BLEServer.h>
#include <BLEUtils.h>
#include <BLE2902.h>

// กำหนดรหัส UUID สำหรับบริการ (Service) และคุณลักษณะ (Characteristic)
// สามารถสร้างใหม่แบบสุ่มสำหรับอุปกรณ์เฉพาะของตนเองได้
#define SERVICE_UUID        "4fafc201-1fb5-459e-8fcc-c5c9c331914b"
#define CHARACTERISTIC_UUID "beb5483e-36e1-4688-b7f5-ea07361b26a8"

BLEServer* pServer = nullptr;
BLECharacteristic* pCharacteristic = nullptr;
bool deviceConnected = false;
bool oldDeviceConnected = false;

// ตัวแปรจำลองข้อมูลเซ็นเซอร์ทางวิศวกรรม
float simulatedTemp = 42.5;     // อุณหภูมิเครื่องจักรขณะทำงาน
float simulatedVibe = 0.12;     // ค่าความเร็วการสั่นสะเทือน (Velocity RMS in mm/s)

// คลาสการตรวจสอบเหตุการณ์การเชื่อมต่อ (Callback Functions)
class MyServerCallbacks: public BLEServerCallbacks {
    void onConnect(BLEServer* pServer) override {
      deviceConnected = true;
    }

    void onDisconnect(BLEServer* pServer) override {
      deviceConnected = false;
    }
};

void setup() {
  Serial.begin(115200);
  Serial.println("เริ่มต้นการทำงาน BLE Server...");

  // เริ่มต้นอุปกรณ์ BLE โดยตั้งชื่อสัญญาณที่ปล่อยออกมา
  BLEDevice::init("ME_Vibe_Monitor");

  // สร้าง BLE Server
  pServer = BLEDevice::createServer();
  pServer->setCallbacks(new MyServerCallbacks());

  // สร้างบริการย่อยภายใต้ Server
  BLEService *pService = pServer->createService(SERVICE_UUID);

  // สร้างคุณลักษณะย่อยสำหรับอ่านและเปิดโหมดแจ้งข้อมูลแจ้งเตือน (Notify)
  pCharacteristic = pService->createCharacteristic(
                      CHARACTERISTIC_UUID,
                      BLECharacteristic::PROPERTY_READ   |
                      BLECharacteristic::PROPERTY_NOTIFY |
                      BLECharacteristic::PROPERTY_INDICATE
                    );

  // แนบตัวบอกอธิบายมาตรฐาน 2902 เพื่อเปิดใช้งานการแจ้งเตือนจากเครื่องรับ (Client)
  pCharacteristic->addDescriptor(new BLE2902());

  // เริ่มต้นการทำงานของบริการ
  pService->start();

  // กำหนดค่าและเริ่มต้นโหมดป่าวประกาศสัญญาณ (Advertising)
  BLEAdvertising *pAdvertising = BLEDevice::getAdvertising();
  pAdvertising->addServiceUUID(SERVICE_UUID);
  pAdvertising->setScanResponse(true);
  pAdvertising->setMinPreferred(0x06);  // ช่วยแก้ปัญหาสำหรับการเชื่อมต่อใน iOS
  pAdvertising->setMinPreferred(0x12);
  BLEDevice::startAdvertising();
  Serial.println("ระบบเตรียมพร้อมแล้ว - กำลังเปิดโหมดโฆษณาสัญญาณรอเครื่องรับเชื่อมต่อ...");
}

void loop() {
    // กรณีเครื่องลูกข่ายเชื่อมต่อเรียบร้อย
    if (deviceConnected) {
        // อัปเดตข้อมูลเซ็นเซอร์จำลอง (สร้างความผันแปรทางวิศวกรรม)
        simulatedTemp += ((float)random(-10, 11) / 100.0);
        simulatedVibe = 0.12 + ((float)random(-5, 6) / 1000.0);

        // จัดเตรียมชุดข้อมูลที่จะส่งออกในรูปแบบตัวอักษร CSV เพื่อให้เครื่องรับถอดรหัสได้ง่าย
        char payload[64];
        snprintf(payload, sizeof(payload), "T:%.2f,V:%.3f", simulatedTemp, simulatedVibe);

        // อัปเดตค่าไปยังระบบเก็บข้อมูลชั่วคราวของคุณลักษณะ
        pCharacteristic->setValue(payload);
        
        // แจ้งเตือนไปยังเครื่องรับทันทีโดยไม่ต้องเข้าลูปคอยตรวจสอบ (Notify)
        pCharacteristic->notify();
        
        Serial.print("ส่งแพ็กเกจแจ้งเตือน: ");
        Serial.println(payload);
        
        delay(2000); // ส่งข้อมูลอัปเดตสถานะเครื่องจักรทุกๆ 2 วินาที
    }
    
    // จัดการเมื่อเครื่องรับตัดการเชื่อมต่อออก
    if (!deviceConnected && oldDeviceConnected) {
        delay(500); // หน่วงเวลาให้ชิปวิทยุเซตอัปสถานะเครือข่ายเสร็จสิ้น
        pServer->startAdvertising(); // สั่งป่าวประกาศกระจายสัญญาณหาเครื่องใหม่ทันที
        Serial.println("สัญญาณหลุด - กำลังเริ่มกระจายสัญญาณโฆษณาใหม่อีกครั้ง...");
        oldDeviceConnected = deviceConnected;
    }
    
    // จัดการเมื่อเครื่องรับทำการเชื่อมต่อเข้ามาใหม่
    if (deviceConnected && !oldDeviceConnected) {
        Serial.println("มีเครื่องรับเข้ามาจับคู่เชื่อมต่อเป็นที่เรียบร้อย");
        oldDeviceConnected = deviceConnected;
    }
}
```

### ตัวอย่างโค้ดที่ 3: ESP32 เชื่อมต่อโมดูลรับส่งวิทยุไร้สาย LoRa ผ่านบัส SPI (โดยใช้ห้องสมุด `LoRa.h`)

โค้ดนี้สาธิตโครงสร้างการเชื่อมต่อพินบัส SPI ระหว่าง ESP32 กับโมดูลวิทยุ LoRa (เช่น RFM95) และการส่งผ่านข้อมูลที่ควบคุมการแผ่กระจายกำลังส่งและความถี่ให้ตรงตามกฎหมายของไทย

```cpp
#include <SPI.h>
#include <LoRa.h>

// การระบุหมายเลขพินบัส SPI ของ ESP32 สำหรับสื่อสารกับชิป LoRa
#define SCK_PIN   5
#define MISO_PIN  19
#define MOSI_PIN  27
#define SS_PIN    18   // พิน CS / NSS
#define RST_PIN   14   // พินรีเซ็ตโมดูลวิทยุ
#define DIO0_PIN  26   // พินขัดจังหวะการรับส่ง (Interrupt)

// กำหนดย่านความถี่ LoRa ที่กฎหมายไทยอนุญาต (923 MHz)
#define LORA_FREQUENCY 923E6

unsigned int telemetryPacketCount = 0;

void setup() {
  Serial.begin(115200);
  while (!Serial); // รอพอร์ตซีเรียลทำงาน (สำหรับบอร์ดจำพวก Native USB)
  Serial.println("--- เริ่มต้นการทำงานอุปกรณ์ LoRa Transmitter ---");

  // เริ่มบัส SPI ด้วยการกำหนดตำแหน่งพินบนบอร์ด ESP32
  SPI.begin(SCK_PIN, MISO_PIN, MOSI_PIN, SS_PIN);
  LoRa.setPins(SS_PIN, RST_PIN, DIO0_PIN);

  // พยายามเปิดการทำงานโมดูล LoRa บนความถี่ 923 MHz
  if (!LoRa.begin(LORA_FREQUENCY)) {
    Serial.println("ข้อผิดพลาด: ไม่สามารถค้นพบหรือสั่งเริ่มระบบวิทยุ LoRa ได้ ตรวจสอบการเดินสายพิน!");
    while (1); // ลูปค้างป้องกันบอร์ดทำงานผิดพลาดต่อไป
  }

  // การตั้งค่าพารามิเตอร์ทางคลื่นวิทยุไร้สายขั้นสูงเพื่อให้เป็นไปตามเป้าหมายวิศวกรรม
  LoRa.setSpreadingFactor(7);           // ใช้ SF7 สำหรับความเร็วและความปลอดภัยของแบตเตอรี่ในระยะปกติ
  LoRa.setSignalBandwidth(125E3);       // กำหนด Bandwidth 125 kHz มาตรฐาน
  LoRa.setCodingRate4(5);               // อัตราการดักรับข้อผิดพลาดแบบ CR 4/5
  LoRa.setTxPower(20);                  // กำหนดกำลังส่งสูงสุดที่กฎหมายไทยอนุญาต (+20 dBm)
  
  Serial.println("การกำหนดค่า LoRa สำเร็จเสร็จสิ้น! ทำงานที่ย่านความถี่ 923 MHz");
}

void loop() {
  // จำลองข้อมูลเซ็นเซอร์ฟาร์มอัจฉริยะ (ความชื้นดิน และ อุณหภูมิใต้ผิวดิน)
  float soilMoisture = 55.4 + random(-5, 6);
  float soilTemperature = 26.8 + ((float)random(-3, 4) / 10.0);

  Serial.print("กำลังส่งแพ็กเกจข้อมูลที่: ");
  Serial.println(telemetryPacketCount);

  // เริ่มต้นแพ็กเกจส่งสัญญาณของ LoRa
  LoRa.beginPacket();
  
  // พิมพ์ข้อความข้อมูลเพื่อส่งออกอากาศ
  LoRa.print("NodeID:05,Moist:");
  LoRa.print(soilMoisture, 1);
  LoRa.print(",Temp:");
  LoRa.print(soilTemperature, 1);
  LoRa.print(",Seq:");
  LoRa.print(telemetryPacketCount);
  
  // บังคับส่งสัญญาณออกอากาศ
  LoRa.endPacket();

  telemetryPacketCount++;

  // การคำนวณเว้นช่วงการส่งตามเกณฑ์ Duty Cycle 1% เพื่อความรับผิดชอบต่อเพื่อนร่วมสัญญาณ
  // ในที่นี้เลือกส่งห่างกันทุก 15 วินาที เพื่อจำลองการใช้งานแบบต่อเนื่องและปลอดภัย
  delay(15000); 
}
```
