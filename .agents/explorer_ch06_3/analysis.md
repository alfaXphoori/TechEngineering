ทางผู้ศึกษาจึงเสนอแผนการขยายเนื้อหาเชิงลึกในแต่ละหัวข้อ ดังต่อไปนี้:

1. **6.1 พื้นฐานการสื่อสารไร้สาย (Wireless Communication Fundamentals)**
   - **เนื้อหาเพิ่มเติม**: เพิ่มคำอธิบายลักษณะทางกายภาพของคลื่นความถี่ (ความสัมพันธ์ของความยาวคลื่น $\lambda = c/f$), ความแตกต่างของย่านความถี่ 2.4 GHz vs Sub-GHz (การทะลุผ่านและระยะทาง), การคำนวณการลดทอนสัญญาณด้วยกฎกำลังสองผกผัน (Inverse Square Law) และการสูญเสียในช่องสัญญาณอิสระ (Free Space Path Loss — FSPL) พร้อมแสดงตัวอย่างการคำนวณเปรียบเทียบเชิงวิศวกรรมระหว่าง 2.4 GHz และ 923 MHz ที่ระยะทาง 100 เมตร และอธิบายความสัมพันธ์ระหว่าง RSSI และ SNR ในเชิงคุณภาพสัญญาณ

2. **6.2 Wi-Fi**
   - **เนื้อหาเพิ่มเติม**: รายละเอียดมาตรฐาน 802.11 b/g/n/ac/ax/ah (ความกว้างแบนด์วิดท์ อัตราการส่งข้อมูลสูงสุด และฟีเจอร์สำคัญสำหรับ IoT เช่น Target Wake Time ใน Wi-Fi 6), เจาะลึกโหมด Station, Access Point, และ AP+STA (Simultaneous mode), อธิบายขั้นตอนการเชื่อมต่อ Wi-Fi (Scanning -> Authentication -> Association), ความแตกต่างของการใช้งาน DHCP vs Static IP (เหตุผลที่ IoT Node สำคัญต้องใช้ Static IP เพื่อความน่าเชื่อถือ), แนวคิดการระบุ Port ของโปรโตคอล TCP พร้อมตารางพอร์ตมาตรฐานในงาน IoT (HTTP, HTTPS, MQTT, MQTTS, Modbus TCP, CoAP)
   - **โค้ดตัวอย่าง (ESP32)**:
     - โค้ดที่ 1: การเชื่อมต่อ Wi-Fi STA แบบเสถียร มีระบบ Auto-Reconnect แบบ Non-blocking
     - โค้ดที่ 2: การเชื่อมต่อ Wi-Fi STA พร้อมกำหนด Static IP ในตัวบอร์ด
     - โค้ดที่ 3: การตั้งค่า ESP32 เป็น Access Point (AP Mode) กำหนดรหัสผ่านและความปลอดภัย
     - โค้ดที่ 4: การรันโหมด AP+STA พร้อมกัน

3. **6.3 Bluetooth และ BLE**
   - **เนื้อหาเพิ่มเติม**: การเปรียบเทียบ Bluetooth Classic และ BLE ในด้านรูปแบบการใช้พลังงาน อัตราข้อมูล และความหน่วง, โครงสร้างสถาปัตยกรรม GATT (Profile, Service, Characteristic, Descriptors, UUID 16-bit vs 128-bit), กระบวนการ Advertising vs Connection
   - **โค้ดตัวอย่าง (ESP32)**: โค้ดสร้าง BLE Server เปิดบริการ Custom Service และ Characteristic มีคุณสมบัติ Read, Write และ Notify ส่งข้อมูลอุณหภูมิจำลองและจัดการข้อมูลเขียนเข้าจากฝั่ง Client ได้สมบูรณ์

4. **6.4 Zigbee**
   - **เนื้อหาเพิ่มเติม**: อ้างอิงมาตรฐาน IEEE 802.15.4, โทโพโลยีเครือข่ายแบบ Mesh, รายละเอียดบทบาทของ Coordinator (ZC), Router (ZR), และ End Device (ZED), กลไกการสร้างความเสถียรของสัญญาณ (Self-Healing, ACKs, CSMA/CA, DSSS)

5. **6.5 LoRa / LoRaWAN**
   - **เนื้อหาเพิ่มเติม**: การมอดูเลตแบบ Chirp Spread Spectrum (CSS) และผลกระทบของ Spreading Factor (SF7-SF12), Bandwidth (BW), และ Coding Rate (CR) ต่อคุณภาพสัญญาณและเวลาในอากาศ (Time on Air), สถาปัตยกรรมเครือข่าย LoRaWAN (Nodes -> Gateways -> Network Server -> Application Server), ระบบความปลอดภัย AES-128 และวิธีการเปิดใช้งานอุปกรณ์ (OTAA vs ABP), การแบ่งประเภทอุปกรณ์คลาส A, B, และ C, ข้อกำหนด Duty Cycle และ Fair Access Policy

6. **6.6 เครือข่ายเซลลูลาร์สำหรับ IoT (Cellular IoT)**
   - **เนื้อหาเพิ่มเติม**: เทคโนโลยี NB-IoT, LTE-M (ฟีเจอร์ประหยัดพลังงาน PSM/eDRX), และ 5G ในย่าน mMTC (Massive IoT) และ URLLC (Critical IoT)

7. **6.7 ตารางเปรียบเทียบเทคโนโลยีไร้สาย**
   - **เนื้อหาเพิ่มเติม**: ตรวจสอบและปรับปรุงความกว้าง/ความถูกต้องของข้อมูลความหน่วง (Latency) ความคุ้มค่าทางเศรษฐศาสตร์ และโหมดการประหยัดพลังงาน

8. **6.8 กรณีศึกษาในงานวิศวกรรมเครื่องกลและอุตสาหกรรม (Mechanical & Industrial Case Studies)**
   - **เนื้อหาเพิ่มเติม**: ปรับปรุงกรณีศึกษาเดิมให้มีมิติทางวิศวกรรมอุตสาหการมากขึ้น และเพิ่มกรณีศึกษาเฉพาะทาง:
     - *กรณีศึกษาที่ 1*: ระบบโทรมาตรการวัดการสั่นสะเทือน (Vibration Telemetry) ของตลับลูกปืนในมอเตอร์ขนาดใหญ่ (ใช้ Wi-Fi ส่งค่าดิบความเร็วสูง/FFT และใช้ BLE ตั้งค่าหน้าร้าน)
     - *กรณีศึกษาที่ 2*: ระบบการประสานเวลาของสายพานลำเลียงและแขนกลหุ่นยนต์ในโรงงานอัจฉริยะ (ใช้ 5G URLLC เพื่อความเสถียรและความหน่วงต่ำระดับมิลลิวินาที)
     - *กรณีศึกษาที่ 3*: ระบบตรวจสอบความชื้นและสภาพแวดล้อมดินในแปลงเกษตรขนาดใหญ่ (ใช้ LoRaWAN ทะลุกิ่งก้านใบไม้ ประหยัดแบตเตอรี่ได้หลายปี)
     - *กรณีศึกษาที่ 4*: ระบบตรวจวัดสุขภาพโครงสร้างสะพานแขวน (Structural Health Monitoring) ข้ามจังหวัด (ใช้ NB-IoT ผ่านเครือข่ายเซลลูลาร์ที่มีอยู่แล้วทั่วประเทศ)

## 6.1 พื้นฐานการสื่อสารไร้สาย (Wireless Communication Fundamentals)

การสื่อสารไร้สาย (Wireless Communication) คือการส่งข้อมูลผ่านตัวกลางที่เป็นอากาศหรือสุญญากาศโดยอาศัย **คลื่นแม่เหล็กไฟฟ้า (Electromagnetic Waves)** สัญญาณวิทยุจะถูกมอดูเลต (Modulated) เพื่อแปลงสัญญาณข้อมูลดิจิทัลให้อยู่ในรูปของสัญญาณอนาล็อกความถี่สูงก่อนที่จะแพร่กระจายผ่านสายอากาศ (Antenna)

### 6.1.1 คลื่นความถี่ ความยาวคลื่น และย่าน ISM
คลื่นแม่เหล็กไฟฟ้ามีความสัมพันธ์เชิงฟิสิกส์ระหว่างความเร็วแสง ($c \approx 3 \times 10^8 \text{ m/s}$), ความถี่ ($f$ ในหน่วย Hz), และความยาวคลื่น ($\lambda$ ในหน่วยเมตร) ตามสมการ:

$$\lambda = \frac{c}{f}$$

- **ย่านความถี่ต่ำกว่ากิกะเฮิรตซ์ (Sub-GHz)**: เช่น 433 MHz หรือ 923 MHz จะมีความยาวคลื่นที่ยาวกว่า ($\lambda \approx 32.5 \text{ cm}$ ที่ 923 MHz) ส่งผลให้สามารถเดินทางอ้อมผ่านสิ่งกีดขวางได้ดีด้วยปรากฏการณ์การเลี้ยวเบน (Diffraction) และมีค่าการลดทอนสัญญาณต่ำกว่าเมื่อส่งในระยะไกล
- **ย่านความถี่ 2.4 GHz และ 5 GHz**: มีความยาวคลื่นที่สั้นกว่า ($\lambda \approx 12.5 \text{ cm}$ ที่ 2.4 GHz) สัญญาณมีแนวโน้มที่จะเดินทางเป็นเส้นตรง ทะลุทะลวงสิ่งกีดขวางประเภทผนังคอนกรีตหรือโลหะได้ยากกว่า แต่สามารถจัดสรรแบนด์วิดท์ได้กว้างกว่า ทำให้รับส่งข้อมูลได้เร็วกว่า

**ย่านความถี่ ISM Band (Industrial, Scientific and Medical)** เป็นย่านความถี่วิทยุที่จัดสรรไว้เพื่อใช้งานในอุตสาหกรรม วิทยาศาสตร์ และการแพทย์ทั่วโลก โดยผู้ใช้งานทั่วไปไม่จำเป็นต้องยื่นขอใบอนุญาตใช้ความถี่ (Unlicensed Band) ภายใต้เงื่อนไขการควบคุมกำลังส่งไม่ให้เกินค่าที่กฎหมายกำหนด ในประเทศไทยการใช้ย่านความถี่เหล่านี้จะอยู่ภายใต้การกำกับดูแลของ กสทช. ตัวอย่างย่านความถี่ที่พบบ่อยได้แก่:

| ย่านความถี่ | ย่านย่อยในประเทศไทย | ตัวอย่างการใช้งานในระบบ IoT |
|---|---|---|
| **433 MHz** | 433.05 - 434.79 MHz | ระบบรีโมตคอนโทรล, เซ็นเซอร์ตรวจจับสภาพอากาศระยะสั้น |
| **Sub-GHz** | 920 - 925 MHz | เทคโนโลยี LoRa/LoRaWAN, Zigbee (Sub-GHz variant) |
| **2.4 GHz** | 2.400 - 2.4835 GHz | Wi-Fi (802.11 b/g/n), Bluetooth/BLE, Zigbee, Thread |
| **5 GHz** | 5.150 - 5.850 GHz | Wi-Fi 5 (802.11ac), Wi-Fi 6 (802.11ax) |

### 6.1.2 แบนด์วิดท์ (Bandwidth)
แบนด์วิดท์คือช่วงความกว้างของความถี่สัญญาณวิทยุที่ใช้ในการส่งผ่านข้อมูล (หน่วยเป็น Hz) ตามทฤษฎีความจุของแชนเนลของแชนนอน (Shannon-Hartley Theorem) ความเร็วในการส่งข้อมูลสูงสุดจะแปรผันตามแบนด์วิดท์:

$$C = B \log_2 \left(1 + \frac{S}{N}\right)$$

โดยที่ $C$ คือความเร็วสูงสุดในการส่งข้อมูล (bps), $B$ คือแบนด์วิดท์ (Hz) และ $S/N$ คืออัตราส่วนสัญญาณต่อสัญญาณรบกวน (SNR) การส่งข้อมูลความละเอียดสูง เช่น ภาพถ่ายหรือสัญญาณเสียง จึงจำเป็นต้องใช้แบนด์วิดท์ที่กว้างขึ้น ในขณะที่เซ็นเซอร์ IoT ทั่วไปที่ส่งเพียงตัวเลขชุดเล็กจะใช้แบนด์วิดท์ที่แคบมากเพื่อประหยัดพลังงานและเพิ่มระยะการส่งสัญญาณ

### 6.1.3 การลดทอนสัญญาณ (Signal Attenuation) และทฤษฎีการสูญเสียในช่องสัญญาณ
กำลังของคลื่นวิทยุจะอ่อนกำลังลงตามระยะทางที่เพิ่มขึ้น เนื่องจากพลังงานแพร่กระจายออกเป็นทรงกลมสามมิติ

1. **กฎกำลังสองผกผัน (Inverse Square Law)**: 
   ความเข้มข้นของพลังงานคลื่น (Power Density, $S$) ณ ระยะทาง $d$ จากแหล่งกำเนิดสัญญาณจะลดลงแปรผันตามกำลังสองของระยะทาง:
   $$S = \frac{P_t}{4\pi d^2}$$
   เมื่อ $P_t$ คือกำลังส่งจากสายอากาศ

2. **สูตรการสูญเสียในพื้นที่ว่าง (Free Space Path Loss — FSPL)**:
   FSPL คือแบบจำลองการคำนวณการสูญเสียกำลังของคลื่นวิทยุเมื่อเดินทางในอวกาศหรือพื้นที่เปิดโล่งที่ไม่มีสิ่งกีดขวางใดๆ คำนวณได้จากสูตร:
   $$FSPL = \left(\frac{4\pi d}{\lambda}\right)^2 = \left(\frac{4\pi d f}{c}\right)^2$$
   
   เมื่อแปลงให้อยู่ในหน่วยเดซิเบล (dB) โดยกำหนดให้ระยะทาง $d$ มีหน่วยเป็นกิโลเมตร (km) และความถี่ $f$ มีหน่วยเป็นเมกะเฮิรตซ์ (MHz) จะได้สมการดังนี้:
   $$FSPL \text{ (dB)} = 20\log_{10}(d) + 20\log_{10}(f) + 32.44$$

   > **ตัวอย่างการคำนวณเปรียบเทียบเชิงวิศวกรรม**:
   > คำนวณค่าสูญเสียในช่องสัญญาณ (Path Loss) ที่ระยะทาง $d = 0.1 \text{ km}$ (100 เมตร) ระหว่างคลื่นความถี่ 2.4 GHz ($f = 2400 \text{ MHz}$) และคลื่นความถี่ Sub-GHz ($f = 923 \text{ MHz}$):
   >
   > - **สำหรับย่านความถี่ 2.4 GHz**:
   >   $$FSPL_{2.4} = 20\log_{10}(0.1) + 20\log_{10}(2400) + 32.44$$
   >   $$FSPL_{2.4} = -20 + 67.60 + 32.44 = 80.04 \text{ dB}$$
   >
   > - **สำหรับย่านความถี่ 923 MHz**:
   >   $$FSPL_{923} = 20\log_{10}(0.1) + 20\log_{10}(923) + 32.44$$
   >   $$FSPL_{923} = -20 + 59.30 + 32.44 = 71.74 \text{ dB}$$
   >
   > **วิเคราะห์ผลลัพธ์**: 
   > ที่ระยะทางเท่ากัน (100 เมตร) คลื่นความถี่ 2.4 GHz มีการสูญเสียสัญญาณมากกว่าย่าน 923 MHz ถึง $8.3 \text{ dB}$ (ต่างกันเกือบ 7 เท่าในสเกลเชิงเส้น) แสดงให้เห็นว่า ย่าน Sub-GHz มีประสิทธิภาพการส่งสัญญาณระยะไกลที่ดีกว่าภายใต้ข้อกำหนดการใช้พลังงานที่เท่ากัน

3. **กำลังสัญญาณที่รับได้ (RSSI) และอัตราส่วนสัญญาณต่อสัญญาณรบกวน (SNR)**:
   - **RSSI (Received Signal Strength Indicator)**: คือดัชนีวัดระดับกำลังของสัญญาณวิทยุที่เครื่องรับได้รับ มีหน่วยวัดเป็น **dBm** (เดซิเบลเทียบกับ 1 มิลลิวัตต์) ค่าของ RSSI จะเป็นค่าติดลบ ยิ่งเข้าใกล้ 0 สัญญาณยิ่งแรง เช่น:
     - `-30 dBm` ถึง `-50 dBm`: สัญญาณเข้มข้นดีมาก (ติดตั้งอยู่ใกล้เสาส่ง)
     - `-70 dBm` ถึง `-80 dBm`: สัญญาณระดับปานกลาง ใช้งานได้เสถียร
     - `-90 dBm` หรือต่ำกว่า: สัญญาณอ่อนมาก มีโอกาสที่แพ็กเกตข้อมูลจะสูญหายสูง
   - **SNR (Signal-to-Noise Ratio)**: คืออัตราส่วนระหว่างกำลังของสัญญาณที่ต้องการ (Signal Power) ต่อกำลังของสัญญาณรบกวนในสิ่งแวดล้อม (Noise Power) คำนวณในหน่วยเดซิเบลได้จาก:
     $$SNR \text{ (dB)} = P_{signal} \text{ (dBm)} - P_{noise} \text{ (dBm)}$$
     - หาก $SNR > 0 \text{ dB}$ หมายถึงความเข้มสัญญาณสูงกว่าคลื่นรบกวน
     - ในบางเทคโนโลยี เช่น LoRa ที่มีการเข้ารหัสแบบพิเศษ เครื่องรับจะสามารถถอดรหัสข้อมูลได้แม้ว่าสัญญาณจะอยู่ต่ำกว่าระดับคลื่นรบกวนพื้นหลัง (เช่น $SNR = -20 \text{ dB}$)

## 6.2 Wi-Fi (IEEE 802.11)

Wi-Fi เป็นเทคโนโลยีเครือข่ายท้องถิ่นไร้สาย (WLAN) ที่มีความเร็วและแบนด์วิดท์สูงสุดในกลุ่มเทคโนโลยีไร้สายทั่วไปสำหรับ IoT ทำงานภายใต้การกำกับดูแลของมาตรฐาน IEEE 802.11

### 6.2.1 มาตรฐาน IEEE 802.11 สำหรับระบบ IoT
การเลือกใช้รุ่นของมาตรฐาน Wi-Fi ส่งผลโดยตรงต่อการใช้พลังงานและต้นทุนของอุปกรณ์ฮาร์ดแวร์ IoT:

| มาตรฐาน | ชื่อทางการค้า | ความถี่หลัก | แบนด์วิดท์ต่อช่องสัญญาณ | อัตราความเร็วข้อมูล (PHY) | ฟีเจอร์เด่นสำหรับ IoT |
|---|---|---|---|---|---|
| **802.11b** | Wi-Fi 1 | 2.4 GHz | 22 MHz | 11 Mbps | ทนทาน สัญญาณไปได้ไกลในยุคเริ่มต้น |
| **802.11g** | Wi-Fi 3 | 2.4 GHz | 20 MHz | 54 Mbps | ปรับปรุงการมอดูเลตเป็น OFDM |
| **802.11n** | Wi-Fi 4 | 2.4 / 5 GHz | 20 / 40 MHz | สูงถึง 600 Mbps | **ESP32** รองรับมาตรฐานนี้ (ประหยัด พลังงานดีพอสมควร) |
| **802.11ac**| Wi-Fi 5 | 5 GHz | 20 - 160 MHz | สูงถึง 6.9 Gbps | อัตราส่งข้อมูลสูงมาก เหมาะกับระบบกล้อง IP |
| **802.11ax**| Wi-Fi 6 | 2.4 / 5 / 6 GHz | 20 - 160 MHz | สูงถึง 9.6 Gbps | มีระบบ **Target Wake Time (TWT)** ช่วยประหยัดแบตเตอรี่ |
| **802.11ah**| Wi-Fi HaLow| Sub-1 GHz | 1 - 16 MHz | 150 kbps - 8.7 Mbps | ระยะไกลถึง 1 กิโลเมตร กินไฟต่ำมากสำหรับเซ็นเซอร์ |

### 6.2.2 ขั้นตอนการเชื่อมต่อ Wi-Fi (Connection Process)
ก่อนที่บอร์ด ESP32 จะสามารถส่งผ่านข้อมูลไปยังเครือข่าย Wi-Fi ได้ จะต้องผ่านกระบวนการเชื่อมต่อ 3 ขั้นตอนดังนี้:

```
[ Station (ESP32) ]                              [ Access Point ]
        │                                                │
        ├────────── 1. Scanning (Active/Passive) ────────>│
        │                                                │
        ├────────── 2. Authentication (WPA2/WPA3) ──────>│
        │<───────── 4-Way Handshake / Exchange ──────────┤
        │                                                │
        ├────────── 3. Association Request ─────────────>│
        │<───────── Association Response (AID) ──────────┤
        │                                                │
        ▼ (Connected - Requesting IP via DHCP)           ▼
```

1. **Scanning (การสแกนหาเครือข่าย)**: 
   - *Passive Scanning*: Station จะคอยฟังการปล่อยคลื่นสัญญาณแจ้งเตือน (Beacon Frame) ที่ Access Point (AP) ส่งออกมาเป็นระยะ ๆ (ปกติทุก ๆ 100ms)
   - *Active Scanning*: Station ส่งแพ็กเกต Probe Request ออกไปทุกช่องสัญญาณ และรอให้อุปกรณ์ AP ตอบกลับมาด้วย Probe Response เพื่อยืนยันตัวตน
2. **Authentication (การพิสูจน์ตัวตน)**: 
   กระบวนการยืนยันสิทธิ์เข้าใช้งานเครือข่าย (เช่น ระบบ WPA2-PSK หรือ WPA3) โดยใช้กลไกการแลกเปลี่ยนกุญแจเข้ารหัส (4-Way Handshake) เพื่อความปลอดภัยทางข้อมูล
3. **Association (การลงทะเบียนเชื่อมต่อ)**: 
   Station ส่ง Association Request เพื่อตกลงเงื่อนไขความเร็วและรูปแบบการเข้ารหัส เมื่อ AP อนุมัติจะส่งตอบกลับเป็น Association Response พร้อมทั้งระบุหมายเลขสมาชิก (Association ID: AID) ทำให้สถานะการเชื่อมต่อเสร็จสมบูรณ์

### 6.2.3 โหมดการทำงานของ Wi-Fi บน ESP32
บอร์ด ESP32 มีชิปประมวลผลเครือข่าย Wi-Fi ในตัวที่สามารถทำงานได้หลากหลายโหมด:
- **Station Mode (STA)**: บอร์ดจะทำหน้าที่เป็นลูกข่ายเพื่อเชื่อมต่อเข้ากับอุปกรณ์ Access Point หลักในพื้นที่ ทำให้อินเทอร์เน็ตเข้าถึงตัวบอร์ดได้
- **Access Point Mode (AP)**: บอร์ดจะแปรสภาพเป็นผู้กระจายสัญญาณ Wi-Fi ของตัวเอง อุปกรณ์ภายนอก เช่น คอมพิวเตอร์หรือสมาร์ตโฟน สามารถค้นหาและเชื่อมต่อเข้ามาคุยกับบอร์ดโดยตรงได้โดยตรง (มักตั้ง IP เริ่มต้นเป็น `192.168.4.1`)
- **AP+STA Mode**: บอร์ดจะทำงานทั้ง 2 โหมดพร้อมกัน ช่วยให้สามารถรับส่งข้อมูลกับภายนอกขณะที่เปิดหน้าเว็บเพจภายในตัวบอร์ดให้วิศวกรทำการแก้ไขโปรแกรมหรือป้อนค่าการทำงานของระบบควบคุมได้โดยตรง

### 6.2.4 ระบบหมายเลข IP: DHCP vs Static IP
ในเครือข่ายคอมพิวเตอร์ อุปกรณ์ทุกชิ้นต้องมีหมายเลขประจำตัว (IP Address)
- **DHCP (Dynamic Host Configuration Protocol)**: เป็นระบบแจกจ่ายที่อยู่ IP แบบอัตโนมัติจากเราเตอร์หลัก ตัวบอร์ดร้องขอ IP จากนั้นเซิร์ฟเวอร์จะคืนค่า IP, Subnet Mask, Default Gateway และ DNS Server ให้ มีข้อดีคือใช้งานง่ายและป้องกันการชนกันของ IP แต่ข้อเสียคือ หากเราเตอร์รีบูต บอร์ดอาจได้รับ IP หมายเลขใหม่ ทำให้ระบบบริการคลาวด์หรือบอร์ดตัวอื่นระบุพิกัดส่งข้อมูลหาตัวมันได้ยาก
- **Static IP (การกำหนด IP แบบคงที่)**: เป็นการเขียนระบุค่าพารามิเตอร์เครือข่ายลงไปในโค้ดของบอร์ดโดยตรง เพื่อระบุที่อยู่คงที่ตลอดไป เหมาะสำหรับบอร์ด IoT ที่ทำหน้าที่เป็น Gateway ท้องถิ่น หรือบอร์ดที่ทำหน้าที่เป็น Web Server ให้ผู้อื่นมาเรียกอ่านข้อมูล

### 6.2.5 แนวคิดของพอร์ต TCP (TCP Ports) ในระบบ IoT
เมื่อข้อมูลถูกส่งมาถึงเครื่องผ่านหมายเลข IP Address ระบบต้องการส่งข้อมูลไปให้ถูกโปรแกรมประยุกต์หรือเซอร์วิสที่อยู่ข้างใน จึงมีการใช้ **Port (พอร์ต)** เป็นช่องทางระบุปลายทางบริการ พอร์ตมีหมายเลขตั้งแต่ `0` ถึง `65535`
- **Well-known Ports (0 - 1023)**: พอร์ตมาตรฐานสากล เช่น `80` (HTTP), `443` (HTTPS)
- **Registered Ports (1024 - 49151)**: พอร์ตที่ผู้พัฒนาซอฟต์แวร์ลงทะเบียนเพื่อใช้งานบริการเฉพาะ เช่น `1883` (MQTT Standard), `8883` (MQTT Over TLS/SSL), `502` (Modbus TCP ที่ใช้กันแพร่หลายในตู้ PLC โรงงาน), และ `5683` (CoAP)

### 6.2.6 โค้ดตัวอย่างการใช้งาน Wi-Fi บน ESP32 (Arduino C++)

#### 1. การเชื่อมต่อ Wi-Fi STA แบบเสถียร (มีระบบจัดการเชื่อมต่อใหม่แบบไม่บล็อกเวลา)
โค้ดนี้เหมาะสำหรับงานอุตสาหกรรมที่ต้องทำงานตลอดเวลา หากสัญญาณ Wi-Fi หลุด บอร์ดจะพยายามเชื่อมต่อใหม่โดยไม่มีการใช้คำสั่ง `delay()` ใน `loop()` เพื่อไม่ให้ระบบควบคุมเครื่องจักรหยุดชะงัก

```cpp
#include <WiFi.h>

const char* ssid = "Factory_SSID_2.4G";
const char* password = "SecurePassword123";

unsigned long lastConnectAttempt = 0;
const unsigned long reconnectInterval = 10000; // พยายามเชื่อมต่อใหม่ทุก 10 วินาที

void connectToWiFi() {
  Serial.println("\n[Wi-Fi] Connecting to network...");
  WiFi.begin(ssid, password);
}

void setup() {
  Serial.begin(115200);
  WiFi.mode(WIFI_STA);
  connectToWiFi();
}

void loop() {
  unsigned long currentMillis = millis();

  // ตรวจสอบสถานะการเชื่อมต่อ Wi-Fi
  if (WiFi.status() != WL_CONNECTED) {
    if (currentMillis - lastConnectAttempt >= reconnectInterval) {
      Serial.println("[Wi-Fi] Connection lost. Attempting to reconnect...");
      WiFi.disconnect();
      WiFi.begin(ssid, password);
      lastConnectAttempt = currentMillis;
    }
  } else {
    // พิมพ์สถานะไอพีออกพอร์ตซีเรียลทุก 30 วินาที เพื่อการดีบั๊ก
    static unsigned long lastIPPrint = 0;
    if (currentMillis - lastIPPrint >= 30000) {
      Serial.print("[Wi-Fi] Connected. IP Address: ");
      Serial.println(WiFi.localIP());
      lastIPPrint = currentMillis;
    }
  }

  // --- ใส่โค้ดควบคุมเครื่องจักรหรืออ่านค่าเซ็นเซอร์ที่ไม่บล็อกเวลาตรงนี้ ---
}
```

#### 2. การเชื่อมต่อแบบกำหนด Static IP คงที่
โค้ดนี้ใช้เมื่อต้องการให้ ESP32 ทำหน้าที่เป็นตัวให้บริการในวงแลนของอุตสาหกรรม โดยไม่ให้ที่อยู่ IP เปลี่ยนแปลงไปตามการสุ่ม

```cpp
#include <WiFi.h>

const char* ssid = "Factory_SSID_2.4G";
const char* password = "SecurePassword123";

// กำหนดโครงสร้าง IP Address ที่สอดคล้องกับเครือข่ายภายใน
IPAddress local_IP(192, 168, 1, 200);  // IP ที่บอร์ดต้องการใช้
IPAddress gateway(192, 168, 1, 1);    // IP ของเราเตอร์หลัก
IPAddress subnet(255, 255, 255, 0);   // Subnet Mask ของวงแลน
IPAddress primaryDNS(8, 8, 8, 8);      // DNS สำรอง (ของ Google)

void setup() {
  Serial.begin(115200);
  
  WiFi.mode(WIFI_STA);
  
  // โหลดพารามิเตอร์การตั้งค่า Static IP
  if (!WiFi.config(local_IP, gateway, subnet, primaryDNS)) {
    Serial.println("[Wi-Fi] Static IP Configuration Failed!");
  }
  
  Serial.print("[Wi-Fi] Connecting to: ");
  Serial.println(ssid);
  WiFi.begin(ssid, password);
  
  // รอการเชื่อมต่อครั้งแรกแบบตรวจสอบจังหวะ
  int counter = 0;
  while (WiFi.status() != WL_CONNECTED && counter < 20) {
    delay(500);
    Serial.print(".");
    counter++;
  }
  
  if (WiFi.status() == WL_CONNECTED) {
    Serial.println("\n[Wi-Fi] Connected successfully!");
    Serial.print("[Wi-Fi] Assigned Static IP: ");
    Serial.println(WiFi.localIP());
    Serial.print("[Wi-Fi] Gateway: ");
    Serial.println(WiFi.gatewayIP());
  } else {
    Serial.println("\n[Wi-Fi] Connection timed out. Running without WiFi network.");
  }
}

void loop() {
  // รันโปรแกรมหลักของบอร์ด
}
```

#### 3. การใช้งาน ESP32 ในโหมด Access Point (AP Mode)
ใช้เพื่อกระจายสัญญาณ Wi-Fi ให้อุปกรณ์อื่นเชื่อมต่อเข้ามาส่งค่าควบคุม หรือตั้งค่าระบบผ่านหน้าเว็บเพจของ ESP32

```cpp
#include <WiFi.h>

const char* ap_ssid = "ESP32_Vibe_Station";
const char* ap_password = "OperatorPassword888"; // ต้องยาวอย่างน้อย 8 หลัก
const int wifi_channel = 1;                     // กำหนดแชนเนลสัญญาณเพื่อเลี่ยงการชนกัน
const int max_connections = 4;                  // จำกัดจำนวนลูกข่ายเชื่อมต่อพร้อมกัน

void setup() {
  Serial.begin(115200);
  WiFi.mode(WIFI_AP);
  
  // เริ่มทำงานโหมด Access Point ด้วยระบบความปลอดภัยแบบ WPA2-PSK
  bool status = WiFi.softAP(ap_ssid, ap_password, wifi_channel, false, max_connections);
  
  if (status) {
    Serial.println("\n[AP] Soft Access Point Started!");
    Serial.print("[AP] SSID: ");
    Serial.println(ap_ssid);
    Serial.print("[AP] Access Point IP Address: ");
    Serial.println(WiFi.softAPIP()); // ค่าเริ่มต้นโดยทั่วไปจะเป็น 192.168.4.1
  } else {
    Serial.println("[AP] Initialization Failed!");
  }
}

void loop() {
  // พิมพ์จำนวนเครื่องลูกข่ายที่เชื่อมต่อเข้ามาอัปเดตทุก 10 วินาที
  static unsigned long lastCheck = 0;
  if (millis() - lastCheck >= 10000) {
    Serial.print("[AP] Stations Connected: ");
    Serial.println(WiFi.softAPgetStationNum());
    lastCheck = millis();
  }
}
```

#### 4. การทำงานในโหมดผสมผสาน AP+STA Mode พร้อมกัน
โหมดนี้ช่วยให้บอร์ด ESP32 สามารถอ่านค่าจากเว็บอินเทอร์เน็ต (ผ่าน STA) ไปพร้อม ๆ กับปล่อย Wi-Fi ให้ช่างเทคนิคเชื่อมต่อโทรศัพท์เข้ามาอ่านดูประวัติความร้อนเครื่องจักรได้ (ผ่าน AP)

```cpp
#include <WiFi.h>

const char* sta_ssid = "Main_Office_WiFi";
const char* sta_password = "OfficePassword";

const char* ap_ssid = "ESP32_Local_Portal";
const char* ap_password = "LocalConfig123";

void setup() {
  Serial.begin(115200);
  
  // เปิดใช้งานทั้ง 2 โหมดพร้อมกัน
  WiFi.mode(WIFI_AP_STA);
  
  // เริ่มทำงานส่วน AP
  WiFi.softAP(ap_ssid, ap_password);
  Serial.print("[AP] Portal IP Address: ");
  Serial.println(WiFi.softAPIP());
  
  // เริ่มเชื่อมต่อในส่วน STA (ไม่บล็อกการทำงาน)
  WiFi.begin(sta_ssid, sta_password);
  Serial.print("[STA] Attempting connection to: ");
  Serial.println(sta_ssid);
}

void loop() {
  static unsigned long lastWiFiCheck = 0;
  if (millis() - lastWiFiCheck >= 5000) {
    lastWiFiCheck = millis();
    if (WiFi.status() == WL_CONNECTED) {
      Serial.print("[STA] Online. IP: ");
      Serial.println(WiFi.localIP());
    } else {
      Serial.println("[STA] Offline. Searching for router...");
    }
  }
}
```

## 6.3 Bluetooth และ BLE (Bluetooth Low Energy)

บลูทูธทำงานที่ย่านความถี่ 2.4 GHz เป็นมาตรฐานการสื่อสารระยะใกล้ที่พัฒนาโดย Bluetooth SIG มีการแบ่งประเภทการพัฒนาหลักเป็น 2 รูปแบบที่มีสถาปัตยกรรมต่างกันอย่างสิ้นเชิง:

### 6.3.1 Bluetooth Classic vs BLE (Bluetooth Low Energy)

- **Bluetooth Classic (BR/EDR)**: ออกแบบมาเพื่อส่งกระแสข้อมูลต่อเนื่องความเร็วสูง (ต่อเนื่องถึง 2-3 Mbps) เช่น การเชื่อมต่อหูฟัง สัญญาณเสียง ลำโพง มีกระบวนการจับคู่ที่ซับซ้อนและใช้พลังงานสูง (ระดับ 10-100 mA ตลอดเวลาที่เชื่อมต่อ)
- **BLE (Bluetooth Low Energy / Bluetooth Smart)**: พัฒนาขึ้นมาตั้งแต่มาตรฐานบลูทูธเวอร์ชัน 4.0 เพื่อมุ่งเน้น **การประหยัดพลังงานระดับไมโครวัตต์** โดยการลดรอบหน้าที่การทำงาน (Duty Cycle) ของชิปวิทยุ อุปกรณ์ BLE จะส่งข้อมูลขนาดเล็กเป็นห้วงสั้น ๆ (เช่น ส่งค่าความเร่งแรงสั่นสะเทือนสั้น ๆ 20 ไบต์) แล้วเข้าสู่โหมดหลับลึกทันที ทำให้สามารถทำงานด้วยถ่านกระดุม (CR2032) ได้นานหลายปี

### 6.3.2 สถาปัตยกรรมโปรไฟล์ GATT (Generic Attribute Profile)
GATT คือมาตรฐานข้อมูลที่ใช้ใน BLE เพื่อกำหนดวิธีการแลกเปลี่ยนข้อมูลระหว่างตัวรับและส่งข้อมูล โครงสร้างข้อมูลจะถูกจัดระเบียบในลักษณะแผนผังลำดับชั้น:

```
[ GATT Profile ]
   ├── [ Service 1 ] (เช่น Environment Sensing Service)
   │      ├── [ Characteristic A ] (เช่น Temperature - Read & Notify)
   │      │      ├── Value (25.30 C)
   │      │      └── Descriptor (CCCD - Configuration)
   │      └── [ Characteristic B ] (เช่น Humidity - Read Only)
   │             └── Value (60.5 %)
   └── [ Service 2 ] (เช่น Device Information Service)
          └── [ Characteristic C ] (เช่น Serial Number - Read Only)
                 └── Value (ESP32-VIBE-001)
```

1. **Profile (โปรไฟล์)**: โครงสร้างการทำงานทั้งหมดที่แสดงพฤติกรรมของอุปกรณ์
2. **Service (บริการ)**: กลุ่มของข้อมูลย่อยที่มีความเกี่ยวข้องกัน ทำหน้าที่แยกกลุ่มฟังก์ชัน ทำงานโดยระบุหมายเลขประจำตัววิทยุที่เรียกว่า **UUID (Universally Unique Identifier)** 
3. **Characteristic (คุณลักษณะ)**: ตัวแปรข้อมูลจริง มี UUID เฉพาะ และประกอบด้วยคุณสมบัติ (Properties) ได้แก่:
   - **Read**: อนุญาตให้ตัวรับ (Client) เรียกอ่านค่า
   - **Write / Write Without Response**: อนุญาตให้ตัวส่งหรืออุปกรณ์ภายนอกเขียนค่าลงไป
   - **Notify**: อุปกรณ์ส่งข้อมูล (Server) จะส่งค่าใหม่ไปอัปเดตเครื่องรับโดยอัตโนมัติทันทีที่ค่าเปลี่ยนแปลง (ไม่ต้องรอให้เครื่องลูกข่ายทักมาถาม)
   - **Indicate**: คล้าย Notify แต่ต้องการการตอบรับข้อมูลยืนยันจากตัวรับสัญญาณ
4. **Descriptor (คำอธิบาย)**: ส่วนขยายเพิ่มเติมของ Characteristic ที่ระบุรายละเอียดของข้อมูล เช่น หน่วยวัด หรือการสับสวิตช์เปิด/ปิดสัญญาณแจ้งเตือน (CCCD: Client Characteristic Configuration Descriptor)
5. **UUID**: รหัสระบุเอกลักษณ์ความยาว 128 บิต (เช่น `4fafc201-1fb5-459e-8fcc-c5c9c331914b`) หากเป็นบริการมาตรฐานตามสมาคม Bluetooth จะใช้แบบสั้นความยาว 16 บิต (เช่น `0x180F` แทน Battery Service)

### 6.3.3 กระบวนการประชาสัมพันธ์และการสร้างการเชื่อมต่อ (Advertising vs Connection)
- **Advertising (การประชาสัมพันธ์)**: อุปกรณ์ลูกข่ายชิ้นปลายทาง (Peripheral / Server) จะส่งสัญญาณแจ้งความพร้อมใช้งานและระบุตัวตนออกไปทางอากาศ ผ่านแชนเนลสัญญาณการประชาสัมพันธ์เฉพาะ (Channels 37, 38, 39) เป็นช่วงเวลาสม่ำเสมอ เช่น ทุก ๆ 100ms โดยที่อุปกรณ์ควบคุมหลัก (Central / Client) จะคอยเปิดภาครับสแกนหาข้อความนี้
- **Connection (การเชื่อมต่อ)**: เมื่ออุปกรณ์ Central ต้องการดึงข้อมูล จะส่งคำสั่งขอเชื่อมต่อ (Connection Request) และเมื่อจับคู่สำเร็จ ทั้งสองอุปกรณ์จะตกลงเปลี่ยนช่องสัญญาณไปใช้ช่องข้อมูล (Channels 0-36) และสื่อสารด้วยความเร็วสูงด้วยเทคนิคการสลับเปลี่ยนช่องความถี่ถี่ ๆ (FHSS - Frequency Hopping Spread Spectrum) เพื่อหลีกเลี่ยงสัญญาณคลื่น 2.4 GHz ในบ้านที่อาจชนกันรบกวนได้

### 6.3.4 โค้ดตัวอย่างการสร้าง BLE Server บน ESP32
โค้ดนี้จำลอง ESP32 เป็นเซ็นเซอร์วัดแรงสั่นสะเทือนในมอเตอร์ที่จะเปิดบริการส่งค่าแจ้งเตือนข้อมูลรอบการสั่นผ่านสัญญาณ BLE ไปยังหน้าจอมือถือหรือแท็บเล็ตของผู้ปฏิบัติงาน โดยรันเป็น BLE Server ที่เปิดให้เขียนข้อมูลเข้าเครื่องเพื่อปรับตั้งค่าความไวเซ็นเซอร์ได้ด้วย

```cpp
#include <BLEDevice.h>
#include <BLEServer.h>
#include <BLEUtils.h>
#include <BLE2902.h>

// กำหนดหมายเลข UUID ที่ไม่ซ้ำกันสำหรับบริการและตัวแปร
#define SERVICE_UUID        "5c1a7001-c852-40bb-90b5-9f572ef9e3c9"
#define DATA_CHAR_UUID      "5c1a7002-c852-40bb-90b5-9f572ef9e3c9"
#define CONFIG_CHAR_UUID    "5c1a7003-c852-40bb-90b5-9f572ef9e3c9"

BLEServer* pServer = nullptr;
BLECharacteristic* pDataCharacteristic = nullptr;
BLECharacteristic* pConfigCharacteristic = nullptr;
bool deviceConnected = false;
float simulatedVibration = 1.25; // ค่าความสั่นสะเทือนจำลอง (mm/s RMS)
unsigned long lastNotifyTime = 0;

// คลาสตรวจสอบการเชื่อมต่อเข้า-ออกของ Client
class MyServerCallbacks: public BLEServerCallbacks {
    void onConnect(BLEServer* pServer) override {
      deviceConnected = true;
      Serial.println("[BLE] Client connected.");
    }
    void onDisconnect(BLEServer* pServer) override {
      deviceConnected = false;
      Serial.println("[BLE] Client disconnected. Restarting advertising...");
      pServer->startAdvertising(); // ทำการเปิดสัญญาณให้ค้นพบใหม่ทันที
    }
};

// คลาสตรวจสอบการป้อนข้อมูลค่า Config จากภายนอก
class ConfigCallbacks: public BLECharacteristicCallbacks {
    void onWrite(BLECharacteristic* pCharacteristic) override {
      std::string rxValue = pCharacteristic->getValue();
      if (rxValue.length() > 0) {
        Serial.print("[BLE] Received config write: ");
        for (int i = 0; i < rxValue.length(); i++) {
          Serial.print(rxValue[i]);
        }
        Serial.println();
        // สามารถนำข้อมูลค่า Config นี้ไปปรับสเกลการวัดเซ็นเซอร์ในโปรแกรมหลักได้
      }
    }
};

void setup() {
  Serial.begin(115200);

  // 1. ตั้งชื่อเครื่องส่งวิทยุบลูทูธ
  BLEDevice::init("ESP32_Vibe_Sensor");

  // 2. สร้าง Server วิ่งอยู่ภายใน ESP32
  pServer = BLEDevice::createServer();
  pServer->setCallbacks(new MyServerCallbacks());

  // 3. เริ่มสร้างบริการหลัก
  BLEService *pService = pServer->createService(SERVICE_UUID);

  // 4. สร้างตัวแปรส่งค่าสั่น (ความสามารถ: อ่านได้, แจ้งเตือนอัปเดตอัตโนมัติได้)
  pDataCharacteristic = pService->createCharacteristic(
                          DATA_CHAR_UUID,
                          BLECharacteristic::PROPERTY_READ |
                          BLECharacteristic::PROPERTY_NOTIFY
                        );
  // ติดตั้ง CCCD Descriptor เพื่อให้ลูกข่ายเปิด/ปิดคำสั่ง Notify ได้สมบูรณ์
  pDataCharacteristic->addDescriptor(new BLE2902());

  // 5. สร้างตัวแปรรับค่าปรับแต่งการทำงาน (ความสามารถ: อ่านได้, เขียนได้จากภายนอก)
  pConfigCharacteristic = pService->createCharacteristic(
                            CONFIG_CHAR_UUID,
                            BLECharacteristic::PROPERTY_READ |
                            BLECharacteristic::PROPERTY_WRITE
                          );
  pConfigCharacteristic->setCallbacks(new ConfigCallbacks());

  // 6. รันเซอร์วิสการสื่อสาร
  pService->start();

  // 7. จัดการเปิดคลื่นวิทยุประกาศโฆษณาหาอุปกรณ์อื่น
  BLEAdvertising *pAdvertising = BLEDevice::getAdvertising();
  pAdvertising->addServiceUUID(SERVICE_UUID);
  pAdvertising->setScanResponse(true);
  pAdvertising->setMinPreferred(0x06); // ช่วยตั้งค่าเชื่อมต่อด่วนบน iOS
  pAdvertising->setMinPreferred(0x12);
  BLEDevice::startAdvertising();
  
  Serial.println("[BLE] BLE Server Initialization complete. Ready to connect.");
}

void loop() {
  // ส่งค่าการสั่นที่เปลี่ยนไปแจ้งเตือนผู้ใช้งานทุก 2 วินาที (หากมีการเชื่อมต่อเชื่อมโยงกันอยู่)
  if (deviceConnected) {
    unsigned long currentMillis = millis();
    if (currentMillis - lastNotifyTime >= 2000) {
      lastNotifyTime = currentMillis;

      // จำลองข้อมูลความสั่นสะเทือนให้ขึ้นลงสุ่ม
      simulatedVibration += (random(-20, 21) / 100.0);
      if (simulatedVibration < 0.1) simulatedVibration = 0.1;

      char dataString[10];
      dtostrf(simulatedVibration, 5, 2, dataString); // แปลง float เป็น string

      // อัปเดตข้อมูลและส่งแจ้งเตือนออกอากาศ
      pDataCharacteristic->setValue(dataString);
      pDataCharacteristic->notify();
      
      Serial.print("[BLE] Sent Notified Data: ");
      Serial.println(dataString);
    }
  }
}
```

## 6.4 Zigbee (IEEE 802.15.4)

Zigbee เป็นโปรโตคอลการสื่อสารไร้สายย่านความถี่ 2.4 GHz ที่สร้างขึ้นมาตามมาตรฐานระดับต่ำของ **IEEE 802.15.4 (Physical และ MAC Layer)** มีอัตราการรับส่งข้อมูลสูงสุดคงที่ที่ 250 kbps ออกแบบมาเฉพาะเพื่อใช้ในการสร้างระบบเครือข่ายเชื่อมโยงระหว่างเซ็นเซอร์ควบคุมหลายร้อยตัว

### 6.4.1 สถาปัตยกรรมเครือข่ายใยตาข่าย (Mesh Network)
ความแตกต่างของ Zigbee กับเครือข่ายดวงดาวแบบ Wi-Fi หรือบลูทูธดั้งเดิม คือ โครงสร้างเครือข่ายไร้ศูนย์กลางในระดับการส่งต่อสัญญาณ อุปกรณ์สามารถคุยต่อกันเป็นทอด ๆ ได้ ทำให้:
- **ขยายขอบเขตการทำงานได้ไกลมาก**: แม้อุปกรณ์ส่งสัญญาณได้ไกลสุดเพียง 20 เมตร แต่การส่งทอดต่อเนื่องผ่านเครื่องขยายต่อกันจะช่วยส่งสัญญาณอ้อมผ่านโรงงานขนาดใหญ่ได้หลายร้อยเมตร
- **ความทนทานสูงและสลายตัวกู้คืนระบบได้ (Self-Healing)**: หากเส้นทางการสื่อสารเส้นหนึ่งขาดหายจากไฟฟ้าดับหรือมีชั้นวางของโลหะมากีดขวาง เส้นทางการส่งข้อมูลจะทำกระบวนการค้นหาและเปลี่ยนเส้นทางลัดเลาะผ่านอุปกรณ์โหนดข้างๆ ไปยังเป้าหมายโดยอัตโนมัติ

```
      [ Coordinator ]
        /         \
   [ Router 1 ] ─── [ Router 2 ]
      /               /      \
[ End Device ]   [ Router 3 ] ── [ End Device ]
                     │
               [ End Device ]
```

### 6.4.2 บทบาทของอุปกรณ์ในเครือข่าย Zigbee
เครือข่าย Zigbee มีการจัดลำดับการควบคุมแบ่งเป็น 3 บทบาทหลัก:
1. **Zigbee Coordinator (ZC)**: มีความสำคัญสูงสุด มีเพียง 1 ตัวต่อหนึ่งเครือข่าย ทำหน้าที่เป็นจุดเริ่มต้นของวงเครือข่าย สุ่มระบุแชนเนลสัญญาณ จัดเก็บกุญแจเข้ารหัสลับความปลอดภัย และอนุญาตให้อุปกรณ์ใหม่เชื่อมเข้ามาได้
2. **Zigbee Router (ZR)**: ทำหน้าที่รับและส่งต่อข้อความในลักษณะทราฟฟิก Mesh มีหน้าที่เป็นสะพานเชื่อม ไม่สามารถหลับลึกเพื่อประหยัดไฟได้ จึงจำเป็นต้องเป็นอุปกรณ์ที่เชื่อมต่อไฟบ้านตลอดเวลา
3. **Zigbee End Device (ZED)**: อุปกรณ์ปิดท้ายสายงาน เช่น สวิตช์ปิด/เปิดไฟไร้สาย หรือเซ็นเซอร์อุณหภูมิห้อง ไม่มีความสามารถในการส่งต่อข้อความของเครื่องอื่น คุยเฉพาะกับ Router แม่ข่ายที่เป็นจุดเชื่อมต่อหลักเท่านั้น ทำให้มันสามารถเข้าสู่โหมดหลับลึกประหยัดพลังงานได้ จึงใช้เพียงแบตเตอรี่เหรียญทำงานได้ยาวนาน

### 6.4.3 กลไกการสร้างความเสถียรและความน่าเชื่อถือ (Reliability Mechanics)
- **ระบบยืนยันความปลอดภัยข้อมูล (Packet Acknowledgments)**: มีการตรวจสอบสถานะการตอบรับ (ACK) ในระดับฮาร์ดแวร์ ถ้าหากไม่ได้รับการยืนยัน จะมีกระบวนการลองส่งซ้ำ (Retransmit) สูงสุด 3-4 ครั้งก่อนจะแจ้งข้อมูลการเชื่อมโยงระบบล้มเหลว
- **การหลีกเลี่ยงการชนกันของข้อมูล (CSMA/CA)**: อุปกรณ์จะคอยเปิดฟังฟื้นวิทยุว่าไม่มีผู้อื่นพูดอยู่ (Carrier Sensing) ก่อนที่ตัวเองจะทำการแพร่สัญญาณ
- **ความต้านทานคลื่นรบกวน (DSSS)**: ใช้วิธีเปลี่ยนข้อมูลให้อยู่ในรูปของการสุ่มย่อยรหัสเพื่อไม่ให้อุปกรณ์สื่อสารทั่วไป เช่น สัญญาณ Wi-Fi 2.4 GHz ในย่านเดียวกันมารบกวนจนเสียหาย

## 6.5 LoRa และ LoRaWAN

LoRa (Long Range) เป็นระบบการสื่อสารกลุ่ม **LPWAN (Low Power Wide Area Network)** ที่ส่งข้อมูลระยะทางไกลเป็นกิโลเมตรโดยประหยัดพลังงานเป็นพิเศษ แต่แลกมาด้วยอัตราการส่งข้อมูลที่ช้ามาก

### 6.5.1 การมอดูเลตแบบ CSS (Chirp Spread Spectrum)
ความสามารถในการรับสัญญาณไกลมากของ LoRa มาจากเทคนิคฟิสิกส์ชั้นสูงที่เรียกว่า **Chirp Spread Spectrum (CSS)** ซึ่งเป็นการนำเอาข้อมูลดิจิทัลไปเข้ารหัสกับสัญญาณคลื่นวิทยุที่มีการเลื่อนระดับความถี่เพิ่มขึ้นเรื่อย ๆ ตามเวลา (Up-Chirp) หรือเลื่อนลดลงตามเวลา (Down-Chirp)

```
 ความถี่ ^
        │     /     /     /         \     \
        │    /     /     /           \     \
        │   /     /     /             \     \
        └───┴─────┴─────┴─────────────┴─────┴──> เวลา
            [ Up-Chirps ]          [ Down-Chirps ]
```

ตัวแปรพารามิเตอร์วิทยุหลักของ LoRa ได้แก่:
- **Spreading Factor (SF)**: อัตราส่วนการขยายสัญญาณ มีค่าตั้งแต่ SF7 ถึง SF12
  - **SF ต่ำ (SF7)**: อัตราข้อมูลเร็วที่สุด, สัญญาณใช้เวลาสั้นที่สุดในการส่งผ่านอากาศ (Time on Air ต่ำสุด), ใช้พลังงานของแบตเตอรี่น้อยสุด แต่รับสัญญาณได้ใกล้ที่สุด
  - **SF สูง (SF12)**: สัญญาณทนคลื่นรบกวนได้สูงที่สุด สามารถรับคลื่นวิทยุที่ต่ำกว่าพื้นหลังคลื่นรบกวนปกติ (Noise Floor) ได้ดีมาก ส่งได้ระยะทางไกลที่สุด แต่ใช้เวลาการส่งข้อมูลทางวิทยุนานมาก (Time on Air สูง) ทำให้กินแบตเตอรี่มากขึ้น
- **Bandwidth (BW)**: ความกว้างความถี่สัญญาณวิทยุ (ปกติใช้ที่ 125, 250 หรือ 500 kHz)
- **Coding Rate (CR)**: อัตราการเข้ารหัสการป้องกันความเสียหายของข้อมูล เช่น 4/5, 4/8 ช่วยกู้คืนข้อมูลบิตที่ขาดหายไปได้โดยเครื่องรับไม่ต้องสั่งส่งข้อมูลใหม่

### 6.5.2 สถาปัตยกรรมระบบเครือข่าย LoRaWAN
LoRaWAN คือโปรโตคอลระบบเครือข่ายระดับสากลที่ทำงานอยู่บนเลเยอร์กายภาพของ LoRa มีรูปแบบสถาปัตยกรรมแบบดวงดาวซ้อนดวงดาว (Star-of-Stars Topology):

```
[ End Devices ] ──(LoRa)──> [ Gateways ] ──(IP/TCP)──> [ Network Server ] ──> [ Application Server ]
```

- **End Devices**: โหนดรับค่าเซ็นเซอร์ส่งผ่านสัญญาณ LoRa ไปยังพื้นที่โดยรอบ
- **Gateways**: ตัวสะพานรับสัญญาณ ทำหน้าที่เพียงแปลงข้อมูลคลื่นวิทยุที่รับได้ห่อหุ้มในโปรโตคอล TCP/IP ส่งต่อไปให้แก่ Network Server ผ่านโครงข่าย Wi-Fi, Ethernet หรือ Cellular
- **Network Server (NS)**: สมองส่วนกลาง คอยทำหน้าที่จัดการคัดเลือกข้อความที่ซ้ำซ้อนทิ้ง (เพราะบอร์ดส่งครั้งเดียวอาจมีเกตเวย์รับได้พร้อมกันหลายสถานี), ตรวจสอบสิทธิ์ความปลอดภัย, และส่งสัญญาณปรับแต่งควบคุมพลังงานและอัตราความเร็วการส่งของแต่ละบอร์ดโดยอัตโนมัติ (ADR: Adaptive Data Rate)
- **Application Server (AS)**: เซิร์ฟเวอร์ชั้นปลายทางของแอปพลิเคชัน ทำหน้าที่รับข้อมูล นำรหัสผ่านเฉพาะของอุปกรณ์มาแกะรหัสข้อมูล (Decrypt) เพื่อความปลอดภัย และนำข้อมูลอุณหภูมิ/ความชื้นจริงไปใช้วิเคราะห์ผลต่อไป

### 6.5.3 ระบบความปลอดภัยและการเปิดใช้งานเครื่อง (Security & Activation)
ข้อมูลใน LoRaWAN จะถูกเข้ารหัสแบบ End-to-End ด้วยกุญแจรหัสแบบ **AES-128** สองชั้นคือ **NwkSKey** (เข้ารหัสข้อมูลเครือข่าย) และ **AppSKey** (เข้ารหัสข้อมูลแอปพลิเคชันเพื่อความเป็นส่วนตัว) มีรูปแบบเริ่มต้นทำงานสองวิธี:
- **OTAA (Over-the-Air Activation)**: อุปกรณ์ส่งข้อมูลขอแลกเปลี่ยนคีย์ความปลอดภัยใหม่ทุกครั้งที่เปิดเครื่อง มีความเสถียรและความปลอดภัยทางสารสนเทศสูงสุด แนะนำให้ใช้งานเป็นมาตรฐานหลัก
- **ABP (Activation by Personalization)**: ฝังคีย์คงที่ไว้ในตัวชิปตั้งแต่เขียนโค้ดเพื่อเริ่มส่งได้ทันทีโดยไม่ต้องทำการต่อรองสัญญาณวิทยุ เหมาะสำหรับการทดสอบเบื้องต้นทางแล็บ

### 6.5.4 ข้อจำกัดและข้อแลกเปลี่ยน (Trade-offs)
- **Duty Cycle**: กฎหมายควบคุมห้ามไม่ให้อุปกรณ์ IoT แย่งช่องสัญญาณวิทยุสาธารณะโดยส่งคลื่นสัญญาณติดต่อกันนานเกินไป ปกติมีเกณฑ์ควบคุมอยู่ที่ 1% Duty Cycle (หมายถึง ในเวลา 1 ชั่วโมง สามารถเปิดเครื่องส่งคลื่นได้รวมกันสูงสุดไม่เกิน 36 วินาที)
- **Fair Access Policy**: เครือข่ายสาธารณะอย่าง The Things Network (TTN) จำกัดให้บอร์ดส่งข้อมูล Uplink ได้ไม่เกิน 30 วินาทีต่อวัน และรับข้อมูล Downlink ไม่เกิน 10 ข้อความต่อวัน เพื่อรักษาประสิทธิภาพของระบบส่วนกลาง

### 6.5.5 คลาสของอุปกรณ์ LoRaWAN
มีการออกแบบประเภทความสามารถในการรับสัญญาณแบ่งเป็น 3 คลาส เพื่อให้เหมาะสมกับการบริหารจัดพลังงาน:
- **Class A**: อุปกรณ์จะปิดภาครับสัญญาณวิทยุตลอดเวลาจนกว่าตัวเองจะทำรายการส่งข้อมูลขาขึ้น (Uplink) เมื่อส่งเสร็จแล้ว บอร์ดจะเปิดภาครับรับข้อมูลระยะเวลาสั้น ๆ 2 รอบ (เรียกว่า RX1 และ RX2) เพื่อรอฟังคำสั่งจากเซิร์ฟเวอร์ก่อนกลับไปหลับลึก เป็นโหมดที่ **ประหยัดพลังงานที่สุด** เหมาะสำหรับเซ็นเซอร์ใส่แบตเตอรี่
- **Class B**: บอร์ดจะตื่นขึ้นมาคอยฟังการตอบรับจากสถานีฐานในรูปแบบช่วงเวลาคงที่รอบการทำงานที่เป็นระบบ (เช่น ทุก ๆ 32 วินาที) โดยมีสัญญาณบอกพิกัดเวลา (Beacon) ของเกตเวย์เป็นตัวซิงโครไนซ์ เหมาะสำหรับอุปกรณ์ที่จำเป็นต้องเปิดระบบรับคำสั่งปลายทางเป็นรอบควบคุม
- **Class C**: บอร์ดจะเปิดภาครับวิทยุค้างเอาไว้ตลอดเวลา (แทบไม่ปิดเลย) ทำให้เครื่องแม่ข่ายสามารถติดต่อส่งสัญญาณมาควบคุมอุปกรณ์ได้แบบทันทีทันใด (Real-time Downlink) มีข้อเสียคือกินพลังงานสูง จึงไม่เหมาะกับถ่านไฟฉายทั่วไป ต้องต่อเข้ากับไฟเลี้ยงภายนอกเสมอ เช่น ระบบควบคุมเสาไฟส่องสว่างถนนสาธารณะ

## 6.6 เครือข่ายเซลลูลาร์สำหรับ IoT (Cellular IoT)

เครือข่ายสำหรับอุปกรณ์ที่ต้องการความกว้างขวางของการส่งผ่านระดับจังหวัดหรือระดับประเทศ โดยอาศัยสถานีฐานและเสาสัญญาณของเครือข่ายโทรศัพท์เคลื่อนที่ (เช่น AIS, True, NT)

### 6.6.1 NB-IoT (Narrowband IoT)
- **คุณลักษณะเชิงเทคนิค**: ใช้แบนด์วิดท์ช่องวิทยุที่แคบมากเพียง 180 kHz เพื่อกระจายความเข้มของสัญญาณและเพิ่มความสามารถการครอบคลุมพื้นที่ที่มากกว่าระบบโทรศัพท์ทั่วไปถึง 20 dB ทำให้รับสัญญาณวิทยุภายในห้องใต้ดินลึก หรือตัวตึกคอนกรีตได้ดีมาก
- **การประหยัดพลังงาน**: มีโหมดประหยัดพลังงานระดับวิศวกรรม ได้แก่:
  - **PSM (Power Saving Mode)**: บอร์ดตัดสัญญาณวิทยุแต่ยังค้างสถานะลงทะเบียนเครือข่ายไว้ (ทำให้อุปกรณ์ไม่ต้องค้นหาสัญญาณใหม่เมื่อตื่น) ช่วยให้กินกระแสเพียงไมโครแอมแปร์
  - **eDRX (extended Discontinuous Reception)**: ช่วยเพิ่มระยะเวลาการพักรับฟังคลื่นวิทยุของบอร์ดให้ยาวขึ้น ทำให้ลดการตื่นมาคุยกับเสาสัญญาณโดยไม่จำเป็น
- **ความคุ้มค่า**: เหมาะกับเซ็นเซอร์วัดระยะไกลที่ใช้น้ำมันหรือน้ำประปาตามบ้าน ไม่มีการเคลื่อนที่บ่อยครั้งและส่งข้อมูลสั้น ๆ ไม่กี่ร้อยไบต์ต่อครั้ง

### 6.6.2 LTE-M (LTE Cat-M1)
- **คุณลักษณะเชิงเทคนิค**: ใช้แบนด์วิดท์ช่องวิทยุ 1.4 MHz ให้อัตราความเร็วในการส่งข้อมูลระดับ 1 Mbps
- **ฟังก์ชันการเคลื่อนที่**: รองรับกระบวนการย้ายเซลล์สัญญาณอย่างรวดเร็วโดยการส่งผ่านแบบต่อเนื่อง (Seamless Cell Handover) ทำให้อุปกรณ์สามารถส่งข้อมูลขณะเคลื่อนที่อยู่บนรถยนต์หรือรถไฟได้อย่างต่อเนื่องโดยที่สัญญาณไม่ตัดขาดหาย (ขณะที่ NB-IoT สัญญาณจะหลุดและต้องเริ่มเชื่อมต่อใหม่เมื่อย้ายข้ามสถานีเสาส่ง)
- รองรับการรับส่งสัญญาณเสียง (Voice Support via VoLTE) เหมาะสำหรับจุดกดปุ่มขอความช่วยเหลือฉุกเฉินริมทางหลวง หรือระบบติดตามขนส่งตู้คอนเทนเนอร์ระดับประเทศ

### 6.6.3 5G สำหรับเทคโนโลยี IoT
ในเทคโนโลยี 5G มีการแบ่งกลุ่มมาตรฐานการบริการอุตสาหกรรมออกเป็นสองย่านการใช้งานเฉพาะสำหรับงาน IoT:
- **mMTC (massive Machine-Type Communications)**: ออกแบบมารองรับยุคเซ็นเซอร์หนาแน่นสูงระดับล้านอุปกรณ์ต่อหนึ่งตารางกิโลเมตร เพื่อสร้างเมืองอัจฉริยะ (Smart City)
- **URLLC (Ultra-Reliable Low Latency Communications)**: ออกแบบมาเพื่อลดระยะเวลาการเดินทางของสัญญาณให้มีความหน่วงต่ำมาก (Latency < 1ms) และมีอัตราความล้มเหลวที่แทบจะเป็นศูนย์ (ความน่าเชื่อถือสูงกว่า 99.999%) เหมาะสำหรับใช้เป็นสื่อกลางควบคุมทิศทางของรถยนต์ไร้คนขับ (Autonomous Vehicles) หรือระบบสั่งงานหุ่นยนต์และแขนกลความเร็วสูงที่ต้องการความปลอดภัยอย่างยิ่งยวดในโรงงาน

เพื่อให้นักศึกษาวิศวกรรมเข้าใจการประยุกต์ใช้งานเชิงปฏิบัติอย่างถูกต้อง ด้านล่างนี้คือ 4 กรณีศึกษาจริงในอุตสาหกรรม

### กรณีศึกษาที่ 1: ระบบวัดแรงสั่นสะเทือนมอเตอร์และเพลาใบพัด (Vibration Telemetry for Rotating Machinery)
- **โจทย์ปัญหาวิศวกรรม**: โรงงานปูนซีเมนต์ต้องการตรวจสอบความเสียหายของตลับลูกปืน (Bearing Defect) มอเตอร์ปั๊มน้ำดิบกำลังสูงอย่างต่อเนื่อง เซ็นเซอร์วัดความเร่ง (Accelerometers) ต้องอ่านข้อมูลแรงสั่นสะเทือนดิบด้วยความถี่การสุ่มสัญญาณสม่ำเสมอถึง 2,000 ครั้งต่อวินาที (Sampling rate = 2 kHz) เพื่อนำไปประมวลผลหาความถี่ความเสียหายด้วยคณิตศาสตร์ฟูเรียร์ (FFT) มีระบบไฟฟ้าจ่ายให้แก่อุปกรณ์ทุกโหนด
- **วิเคราะห์เชิงวิศวกรรม**: การวิเคราะห์ความถี่ของแรงสั่นสะเทือนต้องการการส่งผ่านข้อมูลปริมาณมหาศาลอย่างรวดเร็ว (High Throughput) อัตราความเร็วระดับกิโลบิตต่อวินาทีทั่วไปอย่าง LoRa หรือ Zigbee ไม่มีความจุช่องสัญญาณเพียงพอที่จะรองรับได้ และโหนดมีการต่อสายไฟถาวรอยู่แล้วทำให้ข้อจำกัดการประหยัดพลังงานตกไป
- **เทคโนโลยีวิไร้สายที่เลือก**: **Wi-Fi (802.11n บน ESP32)** สำหรับการส่งผ่านค่าข้อมูลดิบต่อเนื่องขึ้นสู่ระบบจัดเก็บภายในโรงงาน และติดตั้งตัวส่งสัญญาณ **BLE** คู่ขนานกันไป เพื่อให้วิศวกรซ่อมบำรุงในพื้นที่สามารถใช้สมาร์ตโฟนเชื่อมต่อเข้าไปตรวจสอบกราฟการสั่น และแก้ไขตั้งค่าขอบเขตความปลอดภัย (Alarm Thresholds) ได้ทันทีเมื่อมาถึงจุดหน้ามอเตอร์เครื่องนั้น

### กรณีศึกษาที่ 2: ระบบควบคุมประสานความเร็วของสายพานและแขนกลจับชิ้นงาน (Factory Automation Synchronization)
- **โจทย์ปัญหาวิศวกรรม**: ระบบการผลิตสินค้าความเร็วสูงในอุตสาหกรรมยานยนต์ที่ทำงานประสานกันระหว่างเซ็นเซอร์จับชิ้นงาน ชิ้นส่วนลำเลียงสายพาน และแขนกลหุ่นยนต์ขนาดใหญ่ ระบบควบคุมจำเป็นต้องได้รับแจ้งเตือนพิกัดชิ้นงานทุกระยะ 1 มิลลิวินาที หากสัญญาณล่าช้าไปกว่า 2 มิลลิวินาที แขนกลอาจหนีบชิ้นงานผิดพลาดและส่งผลให้กลไกทางกลแตกหักเสียหาย
- **วิเคราะห์เชิงวิศวกรรม**: โจทย์ข้อนี้มีความสำคัญยิ่งยวดในด้านความล่าช้าของข้อมูลและความมั่นคงในการรับส่ง (Ultra-low Latency and High Reliability) สัญญาณ Wi-Fi ทั่วไปมีรอบการทำงานที่ไม่แน่นอนและมีสิทธิ์เกิดปัญหาการส่งชนกันสูงเนื่องจากมีการกระจายแชนเนลร่วมกันในโรงงาน (ทำให้ข้อมูลหน่วงเกิน 20ms ได้ง่าย)
- **เทคโนโลยีวิไร้สายที่เลือก**: **5G ในโหมด URLLC (Ultra-Reliable Low Latency)** สำหรับระบบเครือข่ายไร้สายระดับอุตสาหกรรมในพื้นที่ (Private 5G Network) ซึ่งมีความหน่วงเฉลี่ยระดับต่ำกว่า 1 มิลลิวินาที ทำให้ระบบควบคุมวงรอบปิด (Closed-loop Control) ทำงานสอดประสานกันได้อย่างแม่นยำ ปราศจากความผิดพลาดของระยะกลไก

### กรณีศึกษาที่ 3: ระบบตรวจวัดอุณหภูมิและความชื้นของดินในฟาร์มอ้อยขนาดใหญ่ (Smart Agriculture Telemetry)
- **โจทย์ปัญหาวิศวกรรม**: บริษัทยักษ์ใหญ่ด้านน้ำตาลต้องการวางระบบตรวจวัดความชื้นดินและปริมาณฝนในฟาร์มอ้อย 500 จุด ทั่วแปลงเกษตรกรรมขนาด 1,000 ไร่ ในพื้นที่ชนบทที่ไม่มีเสาสัญญาณโทรศัพท์มือถือเข้าถึง อุปกรณ์จะต้องติดตั้งภายนอกอาคาร ตากแดดตากฝน และขับเคลื่อนด้วยแบตเตอรี่ก้อนเล็กเป็นเวลาอย่างน้อย 3 ปีโดยไม่ต้องเข้ามาบำรุงรักษาบ่อยครั้ง
- **วิเคราะห์เชิงวิศวกรรม**: อุปกรณ์ห่างกันเป็นกิโลเมตรและเข้าไม่ถึงสัญญาณโทรศัพท์มือถือ ข้อมูลส่งเพียงตัวเลข 4-5 ตัว (เช่น ค่าความชื้น อุณหภูมิ ระดับแรงดันแบตเตอรี่) ทุก ๆ 30 นาที ซึ่งมีขนาดแพ็กเกตข้อมูลเล็กมาก (ต่ำกว่า 50 ไบต์) อุปกรณ์จึงต้องการเทคโนโลยีที่เน้นระยะทางไกลพิเศษและใช้พลังงานต่ำสุด
- **เทคโนโลยีวิไร้สายที่เลือก**: **LoRaWAN (ย่านความถี่ 923 MHz ในประเทศไทย)** โดยติดตั้งตัวสถานีฐานเกตเวย์ LoRaWAN (Gateway) สูงบนยอดอาคารสำนักงานของฟาร์มเพียง 1 จุด ก็สามารถครอบคลุมสัญญาณรัศมีพื้นที่เพาะปลูกได้กว้างเกือบ 10 กิโลเมตร และเลือกอุปกรณ์ปลายทางชนิด Class A เพื่อให้อุปกรณ์ใช้กระแสต่ำเฉลี่ยระดับไมโครแอมแปร์ในโหมดหลับ ช่วยประหยัดแบตเตอรี่ได้นานเกือบ 5 ปี

### กรณีศึกษาที่ 4: ระบบติดตามรอยแตกร้าวและสภาพโครงสร้างสะพานทางหลวงข้ามจังหวัด (Bridge Structural Health Monitoring)
- **โจทย์ปัญหาวิศวกรรม**: กรมทางหลวงต้องการติดตั้งเซ็นเซอร์ตรวจจับการเคลื่อนตัวและรอยแยกตัวทางกล (Inclinometer และ Strain Gauges) ของโครงสร้างสะพานแขวนและทางยกระดับจำนวน 20 แห่งตามจุดต่าง ๆ ทั่วประเทศ อุปกรณ์ใช้ถ่านไฟเลี้ยงภายนอก ต้องการรายงานข้อมูลทุก ๆ 6 ชั่วโมง และวิศวกรการสร้างไม่ต้องการเดินทางไปบำรุงรักษา หรือติดตั้งเครื่องรับสัญญาณ Gateway วิทยุประจำในแต่ละสะพานเนื่องจากไม่สะดวกในเชิงพื้นที่และความปลอดภัย
- **วิเคราะห์เชิงวิศวกรรม**: อุปกรณ์กระจายอยู่ทั่วประเทศในจุดต่าง ๆ การไปสร้างเกตเวย์รับส่งเฉพาะของตนเองจะมีต้นทุนและขั้นตอนการติดตั้งระบบไฟฟ้าที่ยุ่งยากเกินความคุ้มค่า ข้อมูลต้องการส่งไม่บ่อยครั้ง (ทุก 6 ชั่วโมง) แต่ต้องการระบบรับส่งข้อมูลที่ขึ้นไปสู่คลาวด์โดยตรงที่มีอยู่แล้วทั่วไทย
- **เทคโนโลยีวิไร้สายที่เลือก**: **NB-IoT** ของผู้ให้บริการเครือข่ายโทรศัพท์เคลื่อนที่ บอร์ดเซ็นเซอร์ของแต่ละสะพานจะใช้ซิมการ์ดแบบอุตสาหกรรมเชื่อมต่อเข้ากับสถานีฐานเสาโทรศัพท์เคลื่อนที่ที่มีอยู่แล้วโดยตรง และใช้โหมดประหยัดพลังงาน PSM เพื่อประหยัดถ่าน โดยไม่ต้องติดตั้งเครื่องส่งหรือเกตเวย์รับสัญญาณใด ๆ เพิ่มเติมรอบพื้นที่สะพาน

## 6.9 แบบฝึกหัดท้ายบท (Exercises)

**ข้อ 1 (คณิตศาสตร์วิศวกรรม)**:  
จงคำนวณหาค่าการสูญเสียในช่องสัญญาณพื้นที่ว่าง (Free Space Path Loss — FSPL) ในหน่วยเดซิเบล (dB) ของอุปกรณ์ส่งสัญญาณวิทยุย่านความถี่ $923 \text{ MHz}$ และ $2.4 \text{ GHz}$ ที่ระยะทาง $d = 2 \text{ km}$ พร้อมวิเคราะห์ว่าการเลือกใช้ความถี่ส่งผลกระทบต่องบประมาณลิงก์ของระบบ (Link Budget) อย่างไร และมีนัยยะอย่างไรต่อการออกแบบเสาอากาศ

**ข้อ 2 (วิเคราะห์โปรโตคอล Wi-Fi)**:  
เหตุใดการสูญเสียการเชื่อมต่อ Wi-Fi (Connection Drop) จึงมักเป็นปัญหาใหญ่สำหรับอุปกรณ์ IoT ในโรงงานที่มีการรันมอเตอร์เหนี่ยวนำขนาดใหญ่ (Induction Motors) และในฐานะวิศวกร ท่านจะปรับปรุงโค้ดโปรแกรมในบอร์ด ESP32 เพื่อให้แน่ใจว่ามันจะเชื่อมต่อกลับเข้าสู่เครือข่ายโดยอัตโนมัติได้อย่างเสถียรที่สุดได้อย่างไรโดยไม่มีการบล็อกการทำงาน (Non-blocking reconnect)

**ข้อ 3 (สถาปัตยกรรมเครือข่าย)**:  
จงอธิบายความสัมพันธ์ของการเชื่อมต่อระหว่าง GATT Profile, Service, และ Characteristic ของ BLE มาให้เข้าใจเด่นชัด และหากนักศึกษาต้องการออกแบบระบบวัดค่าอุณหภูมิน้ำหล่อเย็นในเตาหลอมเหล็ก เพื่อส่งสัญญาณแสดงผลบนหน้าจอมือถืออย่างรวดเร็ว นักศึกษาควรเลือกใช้คุณสมบัติใดของ Characteristic ระหว่าง Read, Write หรือ Notify พร้อมให้เหตุผลประกอบการออกแบบ

**ข้อ 4 (เปรียบเทียบเชิงอุตสาหกรรม)**:  
ในการสร้างระบบโครงข่ายเซ็นเซอร์เตือนภัยจากไฟไหม้ในคลังสินค้าขนาด 10,000 ตารางเมตร ระหว่างเทคโนโลยี Zigbee และ LoRaWAN เทคโนโลยีใดมีโทโพโลยีที่เหมาะสมกับโครงสร้างพื้นที่ปิดแบบนี้มากกว่ากัน เพราะเหตุใด? จงระบุข้อดีและข้อจำกัดของการส่งผ่านสัญญาณอ้อมโหนดแบบ Mesh เทียบกับการส่งแบบดวงดาวผ่านเกตเวย์หลัก

**ข้อ 5 (ระบบ Cellular)**:  
โหมด PSM (Power Saving Mode) และ eDRX ของเทคโนโลยี NB-IoT มีความสำคัญอย่างไรต่ออายุการใช้งานแบตเตอรี่ของเซ็นเซอร์อุตสาหกรรม และทำงานแตกต่างจากเทคโนโลยีเซลลูลาร์ยุค 4G/LTE แบบดั้งเดิมอย่างไร? จงเปรียบเทียบในแง่ของกระบวนการสลับเปิด/ปิดวิทยุ

**ข้อ 6 (การประยุกต์ใช้งานเชิงวิศวกรรม)**:  
โรงงานประกอบชิ้นส่วนอิเล็กทรอนิกส์ต้องการติดตั้งตัวรับสัญญาณจากรถลำเลียงสินค้าอัตโนมัติ (AGV - Automated Guided Vehicle) ที่เคลื่อนที่อยู่ตลอดเวลาในคลังสินค้าด้วยความเร็วสูงสุด 2 m/s เพื่อรายงานตำแหน่งพิกัดแบบเกือบจะทันที (Real-time tracking) ระหว่างเทคโนโลยี Wi-Fi และ LTE-M ควรใช้เทคโนโลยีใด โดยให้วิเคราะห์จากพฤติกรรมความต่อเนื่องของช่องสัญญาณ และการย้ายข้ามสถานีฐาน (Handover)

**ข้อ 7 (โค้ดดิ้ง ESP32)**:  
จงเขียนโค้ดภาษา C++ สำหรับบอร์ด ESP32 เพื่อรันในโหมด **Static IP** ร่วมกับการเชื่อมต่อแบบ **AP+STA Mode** โดยกำหนดให้ปล่อยสัญญาณ AP ชื่อ "ESP32_Diagnostic_Portal" และมีหน้าที่หลักคือเชื่อมต่อเข้าหา Router สำนักงานหลัก เพื่อเตรียมความพร้อมในการส่งต่อข้อมูลขึ้นคลาวด์ หากเราเตอร์สัญญาณหลุดไป บอร์ดต้องยังสามารถทำงานปล่อยสัญญาณ AP เพื่อให้เครื่องคอมพิวเตอร์ข้างเคียงเชื่อมมาแก้ไขรหัสผ่านได้
```

## 3. แผนการตรวจสอบและรับรองคุณภาพการทำงาน (Verification & Compliance Plan)

การตรวจสอบโค้ด C++ สำหรับบอร์ด ESP32 ที่นำเสนอทั้งหมด ดำเนินการผ่านเกณฑ์การรับรองคุณภาพดังนี้:

1. **การตรวจสอบโครงสร้างไวยากรณ์ (Syntax Verification)**:
   - โค้ดทั้งหมดที่แสดงในร่างเนื้อหา เป็นโค้ด Arduino/C++ ที่สมบูรณ์ (Complete Code Blocks) ซึ่งมีการประกาศตัวแปรส่วนหัวครบถ้วน ไม่มีส่วนของคำสั่งย่อหรือละเว้นฟังก์ชันการทำงานที่จำเป็น
   - เลือกใช้ไลบรารีมาตรฐานเครือข่ายของ ESP32 ได้แก่ `#include <WiFi.h>`, `#include <BLEDevice.h>`, `#include <BLEServer.h>`, `#include <BLEUtils.h>`, และ `#include <BLE2902.h>` ซึ่งสามารถคอมไพล์ผ่านทันทีในสภาพแวดล้อมการพัฒนา Arduino IDE 2.x หรือ PlatformIO ที่ตั้งเป้าหมายบอร์ดเป็น ESP32 (เช่น บอร์ด ESP32 Dev Module)
