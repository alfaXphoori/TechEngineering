## 1. จุดประสงค์และกรอบการปรับปรุงเนื้อหา (Objectives & Scope)

จากข้อกำหนดระดับบทเรียนและข้อจำกัดเชิงระบบ เราจะมุ่งขยายเนื้อหาบทที่ 8 ใน 6 ด้านสำคัญ:
1. **เพิ่มความเข้มข้นด้านทฤษฎี Cloud Computing สำหรับ IoT ในงานอุตสาหกรรม (Industrial IoT):** ขยายการเปรียบเทียบระหว่าง Edge Computing และ Cloud Computing พร้อมแนะนำตัวชี้วัดประสิทธิภาพเครือข่าย (Latency, Bandwidth, Jitter, Packet Loss, SLA)
2. **ขยายสถาปัตยกรรมและรายละเอียดทางเทคนิคของ Blynk:** อธิบายกลไกภายในของ Blynk Server การส่งผ่านข้อมูลด้วย Virtual Pin (พินเสมือน), การเขียนโปรแกรมแบบ Event-driven (Callbacks), ไลบรารี ESP32 ที่เจาะลึก และการอธิบายความเสี่ยงเชิงลึกของการใช้ฟังก์ชันบล็อกเวลาอย่าง `delay()`
3. **จัดทำกรณีศึกษาเชิงวิศวกรรมเครื่องกล (Mechanical/Industrial Case Study):** ออกแบบระบบตรวจวัดและควบคุมหม้อไอน้ำอุตสาหกรรม (Factory Steam Boiler Monitoring) ที่มีทั้งการตรวจวัดตัวแปรทางฟิสิกส์และการควบคุมระดับฮาร์ดแวร์/คลาวด์
4. **ลงลึกรายละเอียดโปรโตคอลการรับเข้าข้อมูล (Data Ingestion) และสถาปัตยกรรมฐานข้อมูล (Databases):** เปรียบเทียบ MQTT, HTTP, CoAP อย่างละเอียดด้วยตารางตัวชี้วัด และอธิบายเหตุผลทางสถาปัตยกรรมที่ Time-series Database (TSDB) เหมาะสมกับ IoT มากกว่า SQL/Relational Database
5. **ขยายเนื้อหาความปลอดภัย IoT (IoT Security):** เพิ่มความรู้เรื่องการเก็บรักษา Auth Token, การเข้ารหัส SSL/TLS (MQTTS/HTTPS) บน ESP32 และความปลอดภัยในการอัปเดตเฟิร์มแวร์ผ่านอากาศ (OTA Security)
6. **ออกแบบแบบฝึกหัดและกิจกรรมปฏิบัติ Wokwi ใหม่:** เพิ่มคำถามคำนวณ Bandwidth และออกแบบโครงงานจำลองหม้อไอน้ำที่มีระบบขัดจังหวะความปลอดภัยในระดับท้องถิ่น (Local Edge Safety Override)

## 2. รายละเอียดการขยายเนื้อหาแต่ละหัวข้อ (Detailed Expansion Plan)

### หัวข้อที่ 1: คลาวด์คอมพิวติงสำหรับงานวิศวกรรมเครื่องกลและอุตสาหกรรม (8.1 - 8.3)
**เนื้อหาที่จะเพิ่ม:**
*   **การแบ่งขอบเขตระหว่าง Edge Computing และ Cloud Computing (Edge vs Cloud Architecture):**
    *   *Edge Computing:* ทำหน้าที่ประมวลผลข้อมูลที่มีความถี่สูงและต้องการการตอบสนองที่รวดเร็ว (Sub-millisecond Latency) เช่น การวิเคราะห์การสั่นสะเทือนของแบริ่งมอเตอร์ด้วย FFT (Fast Fourier Transform), ลูปการตัดระบบไฟฟ้าฉุกเฉิน (Emergency Safety Trip) 
    *   *Cloud Computing:* ทำหน้าที่วิเคราะห์ข้อมูลในภาพรวมระยะยาว (Long-term Analytics), การทำนายความเสียหายล่วงหน้า (Predictive Maintenance) ด้วย Machine Learning และการรวมข้อมูลจากหลายโรงงาน
*   **ตัวชี้วัดประสิทธิภาพระบบเครือข่ายสำหรับวิศวกร (Network Quality of Service - QoS Metrics):**
    *   *Latency (ความหน่วงเวลา) / RTT (Round Trip Time):* นิยามและผลกระทบต่อลูปการควบคุมเชิงกล
    *   *Bandwidth (แบนด์วิดท์) & Throughput (ปริมาณงาน):* การประเมินขนาดข้อมูลเซนเซอร์ในโรงงาน
    *   *Jitter (ความแปรปรวนของความหน่วง):* ปัญหาที่ส่งผลต่อระบบควบคุมแบบเรียลไทม์
    *   *Packet Loss (อัตราสูญหายของแพ็กเกจ):* การจัดการของโปรโตคอลที่ระดับ Transport (TCP vs UDP)
    *   *Availability & Service Level Agreement (SLA):* การคำนวณเวลาที่ระบบหยุดทำงาน (Downtime) ในหนึ่งปี (เช่น 99.9% หรือ "Three Nines" ยอมให้ระบบล่มได้ไม่เกิน 8.76 ชั่วโมงต่อปี)

### หัวข้อที่ 2: สถาปัตยกรรมและรายละเอียดทางเทคนิคของ Blynk (8.6)
**เนื้อหาที่จะเพิ่ม:**
*   **กลไกสถาปัตยกรรม Blynk:**
    *   แผนผังการไหลของข้อมูล: `ESP32 Board` $\leftrightarrow$ `Blynk Cloud Server (TCP Port 80/443 หรือ 8080/443)` $\leftrightarrow$ `Mobile App/Web Console`
    *   การทำงานของ Blynk Protocol (ฐาน Binary ประสิทธิภาพสูงเพื่อลด Overhead ของแพ็กเกจข้อมูล)
*   **พินเสมือน (Virtual Pins - V0 ถึง V255):**
    *   อธิบายแนวคิดการจัดสรรข้อมูลโดยทำหน้าที่เป็น "Data Channels" หรือตัวแปรหน่วยความจำเสมือนที่ตั้งอยู่บนคลาวด์ ซึ่งช่วยแยกตรรกะของฮาร์ดแวร์ (Hardware Abstraction) ออกจากอินเทอร์เฟซผู้ใช้ (UI Dashboard)
    *   ตัวอย่างชนิดข้อมูลที่ Virtual Pins รองรับ: `Integer`, `Float`, `String` และข้อมูลรูปแบบอาร์เรย์ (Array/CSV) หรือ JSON
*   **ความแตกต่างระหว่างการเขียนโค้ดแบบเดิมและการเขียนโค้ดแบบ Event-driven (Callbacks):**
    *   การใช้ `BLYNK_WRITE(V_PIN)`: อธิบายกลไกเมื่อผู้ใช้กดปุ่มบนแอป คลาวด์เซิร์ฟเวอร์จะส่งแพ็กเกจมายัง ESP32 เพื่อกระตุ้นฟังก์ชัน Callนี้ จากนั้นใช้ `param.asInt()` หรือ `param.asFloat()` เพื่อดึงค่าออกมาใช้งาน
    *   การใช้ `BLYNK_READ(V_PIN)`: ฟังก์ชัน Call เมื่อคลาวด์ร้องขอข้อมูลจากอุปกรณ์
    *   การใช้ `BLYNK_CONNECTED()`: สำหรับการเรียกใช้ฟังก์ชัน `Blynk.syncAll()` เพื่อประสานสถานะ (Sync) ล่าสุดจากคลาวด์กลับคืนสู่ฮาร์ดแวร์หลังจากไฟดับหรือเน็ตหลุด
*   **ทำไมฟังก์ชัน `delay()` จึงเป็นภัยคุกคักร้ายแรงในระบบ IoT:**
    *   อธิบายการทำงานของบอร์ด ESP32 ที่ทำงานแบบ Single Thread เป็นหลัก (แม้จะมี Dual Core แต่เฟรมเวิร์ก Arduino มักรันบน Core 1 เป็นหลัก)
    *   เมื่อเรียกใช้ `delay(ms)` บอร์ดจะหยุดรอโดยไม่ทำงานอื่น ซึ่งรวมถึงการตอบสนองต่อสแต็กเครือข่ายและการเรียกฟังก์ชัน `Blynk.run()`
    *   Blynk Server จะมีระบบส่งแพ็กเกจตรวจสอบสถานะ (Ping/Heartbeat) ทุกๆ 10 วินาที หาก ESP32 ถูกบล็อกด้วย `delay()` และไม่ส่งข้อมูลกลับภายในช่วงเวลา (Heartbeat Timeout) เซิร์ฟเวอร์จะตัดการเชื่อมต่อทันที ส่งผลให้เกิดวงจร "หลุด-ต่อใหม่-หลุด" วนเวียนไม่สิ้นสุด
*   **เจาะลึกสถาปัตยกรรมของ `BlynkTimer`:**
    *   พัฒนาต่อยอดมาจากไลบรารี `SimpleTimer`
    *   ใช้กลไกการเช็คเวลาปัจจุบันผ่านระบบสะสมเวลา `millis()` บนฮาร์ดแวร์ โดยการวนลูปเช็คอย่างรวดเร็ว (Non-blocking Software Polling) 
    *   แสดงตัวอย่างการใช้งานแบบมีพารามิเตอร์และการผูกหลายฟังก์ชันจับเวลาเข้ากับตัวจับเวลาเดียว

### หัวข้อที่ 3: กรณีศึกษาเชิงวิศวกรรมเครื่องกล: ระบบตรวจวัดและควบคุมหม้อไอน้ำอุตสาหกรรม (Steam Boiler)
**เนื้อหาที่จะเพิ่ม:**
*   **รายละเอียดระบบประยุกต์เชิงกล:**
    *   *วัตถุประสงค์:* ตรวจสอบความปลอดภัยและประสิทธิภาพของหม้อต้มไอน้ำในโรงงานประกอบรถยนต์
    *   *ตัวแปรทางกายภาพที่วัด:* อุณหภูมิในห้องต้ม (Temperature - °C), ความดันไอ (Steam Pressure - MPa), ระดับน้ำในถัง (Water Level - % หรือ Liters)
    *   *อุปกรณ์ควบคุม (Actuators):* วาล์วจ่ายน้ำเข้าระบบ (Feedwater Pump/Solenoid Valve - รีเลย์), วาล์วควบคุมหัวเผาไหม้ (Burner Control Valve - มอเตอร์เซอร์โว)
*   **ตารางการจับคู่พินเสมือน (Virtual Pin Mapping Table):**
    
    | พินเสมือน | ชนิดข้อมูล | ทิศทางการส่ง | อุปกรณ์แสดงผล/ควบคุมบนแดชบอร์ด | คำอธิบายการใช้งาน |
    | :--- | :--- | :--- | :--- | :--- |
    | `V0` | Float | ESP32 $\rightarrow$ Blynk | Gauge (มาตรวัด) | อุณหภูมิห้องต้มห้องต้ม (0.0 - 250.0 °C) |
    | `V1` | Float | ESP32 $\rightarrow$ Blynk | Super Chart (กราฟเส้น) | ความดันไอน้ำในถังต้ม (0.00 - 2.00 MPa) |
    | `V2` | Integer | ESP32 $\rightarrow$ Blynk | Value Display (ตัวเลข) | ระดับน้ำหม้อต้ม (0 - 100%) |
    | `V3` | Integer | ESP32 $\rightarrow$ Blynk | LED (ดวงไฟแสดงผล) | สถานะความปลอดภัย (Green = Normal, Red = Alarm) |
    | `V4` | Integer | Blynk $\rightarrow$ ESP32 | Button (ปุ่มกดล็อกสถานะ) | ปุ่มเปิด-ปิดระบบหัวเผาไหม้แบบ Manual Override |
    | `V5` | String | ESP32 $\rightarrow$ Blynk | Terminal (หน้าต่างข้อความ) | ล็อกประวัติการทำงานและข้อผิดพลาด (Event Logging) |

*   **โครงสร้างสถาปัตยกรรมการควบคุมแบบผสมผสาน (Edge-Cloud Hybrid Control):**
    *   *ตรรกะความปลอดภัยที่ Edge (Local Safety Interlock):* หากความดันไอน้ำเซนเซอร์วัดได้เกิน 1.5 MPa บอร์ด ESP32 จะต้องสั่ง **ปิดวาล์วหัวเผาทันทีในระดับโค้ดฮาร์ดแวร์** โดยไม่รอการตอบสนองจากคลาวด์เพื่อความปลอดภัยสูงสุด จากนั้นจึงส่งสัญญาณเตือนภัย (Alarm) ขึ้นคลาวด์ผ่าน `V3` และส่งข้อความพุชโนติฟิเคชันไปยังมือถือ
    *   *ตรรกะระดับคลาวด์ (Cloud Control):* วิศวกรสามารถควบคุมการปิดเปิดปั๊มน้ำป้อนได้ผ่านปุ่มบนมือถือ แต่โค้ด ESP32 จะตรวจสอบก่อนว่าความดันในหม้อต้มไม่ได้อยู่ในระดับอันตราย จึงจะอนุญาตให้ปั๊มน้ำเปิดทำงาน

### หัวข้อที่ 4: การรับเข้าข้อมูล (Data Ingestion) และฐานข้อมูลอนุกรมเวลา (Time-series Databases) (8.4 & 8.7)
**เนื้อหาที่จะเพิ่ม:**
*   **การวิเคราะห์โปรโตคอลการรับเข้าข้อมูล (Data Ingestion Protocols):**
    *   *MQTT:* ส่งข้อมูลแบบ Publisher/Subscriber ทำงานบน TCP/IP มีขนาด Header ต่ำสุดเพียง 2 Bytes มีระบบ Quality of Service (QoS 0, QoS 1, QoS 2) และ Last Will and Testament (LWT) เหมาะสำหรับระบบที่ข้อมูลส่งบ่อย แบนด์วิดท์จำกัด
    *   *HTTP:* ส่งข้อมูลแบบ Request/Response ทำงานบน TCP/IP มีขนาด Header ที่ใหญ่มาก (หลักร้อยถึงพันไบต์เนื่องจากใช้ข้อความ Text) เหมาะสำหรับการดึงข้อมูลนานๆ ครั้ง หรือการส่งภาพนิ่ง
    *   *CoAP:* ส่งข้อมูลแบบ Request/Response แต่ทำงานบน UDP ประหยัดพลังงานเพราะไม่มีขั้นตอนการทำ Handshake ที่ซับซ้อน เหมาะสำหรับเซนเซอร์ใช้แบตเตอรี่ที่ต้องการส่งข้อมูลด่วนและไม่ห่วงแพ็กเกจหล่นหายเล็กน้อย
*   **ตารางเปรียบเทียบโปรโตคอลสำหรับการใช้งานจริง:**
    
    | คุณลักษณะ | MQTT | HTTP (REST API) | CoAP |
    | :--- | :--- | :--- | :--- |
    | **ชั้นการขนส่ง (Transport)** | TCP | TCP | UDP |
    | **สถาปัตยกรรม** | Publish / Subscribe | Request / Response | Request / Response |
    | **การเชื่อมต่อ** | ต่อเนื่อง (Keep-Alive Connection) | เชื่อมต่อใหม่ทุกครั้งที่ร้องขอ (Stateless) | ไม่ต่อเนื่อง (Connectionless) |
    | **ขนาดส่วนหัวขั้นต่ำ** | 2 Bytes | ~200 Bytes - 1 KB | 4 Bytes |
    | **ระดับความปลอดภัยมาตรฐาน** | MQTTS (TLS Port 8883) | HTTPS (TLS Port 443) | DTLS (Port 5684) |
    | **การประยุกต์ใช้ในอุตสาหกรรม** | ระบบเทเลเมทรีเครื่องจักรส่งข้อมูลบ่อย | ระบบส่งรายงานการผลิตประจำวัน | เซนเซอร์ขนาดเล็กในไร่เกษตรกรรมอัจฉริยะ |

*   **การเจาะลึกโครงสร้างฐานข้อมูลอนุกรมเวลา (Time-series Database vs SQL Database):**
    *   *โครงสร้าง SQL:* การเก็บข้อมูลแบบ Relational Database (เช่น PostgreSQL, MySQL) มีดัชนีเป็นแบบ B-Tree เมื่อมีการเขียนข้อมูลนับหมื่นแถวจากเซนเซอร์ทุกวินาที ฐานข้อมูลจะต้องเรียงลำดับดัชนีใหม่ตลอดเวลา เกิดปัญหาคอขวดเขียนไม่ทัน และการดึงข้อมูลย้อนหลังตามขอบเขตเวลา (Range Queries) ต้องสแกนข้อมูลจำนวนมหาศาลทำให้ประสิทธิภาพตกลงเรื่อยๆ
    *   *โครงสร้าง TSDB:* (เช่น InfluxDB, TimescaleDB) การเก็บข้อมูลจะเน้นเป็นชุดข้อมูลเวลา (Append-only write ด้วยโครงสร้าง Log-Structured Merge-tree) ข้อมูลจะบันทึกต่อท้ายและจัดเก็บแยกพาร์ทิชันตามช่วงเวลาโดยอัตโนมัติ ทำให้ประสิทธิภาพการเขียนคงที่แม้ระบบจะมีขนาดใหญ่ขึ้น และมีความสามารถในการรวบรวมข้อมูลย้อนหลัง (Downsampling) เช่น การย่นย่อข้อมูลอุณหภูมิความถี่วินาทีในอดีตให้เหลือค่าเฉลี่ยทุกๆ 1 ชั่วโมงโดยอัตโนมัติเพื่อประหยัดพื้นที่จัดเก็บข้อมูล

### หัวข้อที่ 5: ความปลอดภัยในระบบ IoT (8.8)
**เนื้อหาที่จะเพิ่ม:**
*   **แนวทางการรักษาความลับของ Auth Token:**
    *   อธิบายความเสี่ยงเมื่อเอา Auth Token ไปอัปโหลดขึ้น GitHub สาธารณะ แฮกเกอร์สามารถใช้สแกนเนอร์บอทค้นหา Token และแย่งการควบคุมฮาร์ดแวร์
    *   แนวทางการออกแบบ: แนะนำการบันทึกโทเค็นไว้ใน EEPROM หรือ Non-Volatile Storage (NVS) ของ ESP32 ผ่านการ Provisioning หน้าเว็บ (เช่น Blynk Edgent / WiFi Manager) แทนการระบุตรงในโค้ดดิบ
*   **การเข้ารหัสและการเชื่อมต่อที่ปลอดภัย (SSL/TLS Encryption):**
    *   ความแตกต่างระหว่างการส่งข้อมูลแบบ HTTP/Plain MQTT (ข้อความวิ่งผ่านเครือข่ายเป็น Plaintext แฮกเกอร์ใช้ Wireshark ดักจับได้ง่าย) กับ HTTPS/MQTTS (ข้อมูลจะถูกเข้ารหัสก่อนส่งออก)
    *   การนำมาใช้งานบน ESP32: อธิบายไลบรารี `WiFiClientSecure` และการโหลดใบรับรองความปลอดภัยระดับราก (Root CA Certificate) ลงบนหน่วยความจำแฟลช
    *   *ข้อจำกัดและข้อควรระวัง:* การเข้ารหัส SSL/TLS บน ESP32 ส่งผลให้ชิปต้องประมวลผลการคำนวณคณิตศาสตร์เข้ารหัสที่ซับซ้อน (RSA/ECC) ทำให้ใช้หน่วยความจำ RAM เพิ่มขึ้นประมาณ 20-30 KB และเวลาในการเริ่มเชื่อมต่ออาจนานขึ้น 1-2 วินาที
*   **สถาปัตยกรรมความปลอดภัยในการอัปเดตเฟิร์มแวร์ผ่านอากาศ (Over-The-Air: OTA Security):**
    *   อธิบายความเสี่ยงของการแฮกแบบ Firmware Injection (ผู้ประสงค์ร้ายจำลองตนเองเป็นคลาวด์ส่งไฟล์เฟิร์มแวร์ที่มีมัลแวร์ไปเขียนทับอุปกรณ์)
    *   แนวทางป้องกัน: การลงลายมือชื่อดิจิทัลบนเฟิร์มแวร์ (Firmware Signing ด้วย Public/Private Keys) และการตรวจสอบความถูกต้องด้วยค่าแฮช (SHA-256 Checksum) เพื่อให้ตัวโหลดบูตของ ESP32 ตรวจสอบก่อนอนุญาตให้รันโค้ดใหม่

### หัวข้อที่ 6: แบบฝึกหัดและกิจกรรมปฏิบัติ Wokwi ใหม่ (แบบฝึกหัดท้ายบท)
**เนื้อหาที่จะเพิ่ม:**

*   **คำถามทบทวนความเข้าใจเชิงคำนวณและทฤษฎีเพิ่มเติม:**
    1.  *คำนวณการใช้ข้อมูล (Bandwidth Calculation):* "โรงงานติดตั้งระบบวัดแรงสั่นสะเทือนมอเตอร์ 50 เครื่อง แต่ละเครื่องส่งข้อมูลขนาด 32 ไบต์ ทุกๆ 100 มิลลิวินาที จงเปรียบเทียบปริมาณข้อมูลต่อชั่วโมงหากใช้ระบบ MQTT (ขนาด Header 2 ไบต์) เทียบกับ HTTP GET (ขนาด Header เฉลี่ย 350 ไบต์) พร้อมทั้งวิเคราะห์ข้อจำกัดในการรองรับสเกลแบนด์วิดท์"
    2.  *วิเคราะห์ความปลอดภัยเครือข่าย:* "ให้อธิบายแผนภาพความเสี่ยงหากอุปกรณ์ ESP32 สื่อสารกับ Blynk Cloud ผ่าน WiFi ที่ไม่เข้ารหัสแบบไม่มีรหัสผ่าน (Open Hotspot) และมีคนประสงค์ร้ายต้องการขโมยค่าอุณหภูมิการผลิตหม้อไอน้ำ จะสามารถโจมตีด้วยวิธีใดบ้างและมีแนวทางแก้ไขอย่างไรที่ตัวอุปกรณ์"
    3.  *วิเคราะห์เชิงสถาปัตยกรรมระบบควบคุม:* "เพราะเหตุใดฟังก์ชันสั่งเปิดวาล์วลดแรงดันฉุกเฉินของหม้อไอน้ำ (Pressure Relief Valve) จึงต้องถูกตัดสินใจและสั่งงานที่ระดับ Edge Controller (ESP32 ในพื้นที่) ทันที มากกว่าการรอประมวลผลบนคลาวด์แล้วสั่งงานกลับลงมา อภิปรายผลกระทบหากเครือข่ายอินเทอร์เน็ตขัดข้องเป็นเวลา 15 วินาที"

*   **กิจกรรมฝึกปฏิบัติบน Wokwi 2 ภารกิจใหม่ (พร้อมแผนภาพวงจรและโค้ดนำทาง):**
    
    #### ภารกิจที่ 1: ระบบตรวจวัดหม้อต้มอัจฉริยะและการตัดระบบความปลอดภัยด้วยตนเอง (Local Interlock & Cloud Control)
    *   **วัตถุประสงค์:** ฝึกออกแบบลูปความปลอดภัยในระดับ Edge และการยอมรับการควบคุมแบบสั่งงานด้วยคน (Manual Override) ผ่านคลาวด์
    *   **อุปกรณ์ในโปรแกรมจำลอง Wokwi:**
        *   บอร์ด ESP32
        *   โพเทนชิโอมิเตอร์ 2 ตัว (ตัวที่ 1 จำลองแรงดันไอ `0.00-2.00 MPa` ต่อเข้าพิน A0, ตัวที่ 2 จำลองอุณหภูมิหม้อต้ม `0-250 °C` ต่อเข้าพิน A1)
        *   รีเลย์ 1 ตัวพร้อมต่อกับพัดลมหรือไฟ LED (จำลองโซลินอยด์จ่ายแก๊สของหัวเผาต้มน้ำ)
        *   ไฟ LED สีแดง 1 ดวง (จำลองไฟแจ้งเตือนอันตราย Local Warning)
    *   **ข้อกำหนดการเขียนโปรแกรม (C++):**
        1. ใช้ `BlynkTimer` เพื่อสุ่มอ่านค่าตัวต้านทานปรับค่าได้ทั้งสองตัว และอัปโหลดขึ้น Blynk ทุกๆ 2.5 วินาที โดยเชื่อมโยงกับพินเสมือน `V0` (อุณหภูมิ) และ `V1` (แรงดัน)
        2. ออกแบบระบบตัดระบบฉุกเฉิน (Edge Interlock): หากแรงดันไอ превышает (เกิน) `1.60 MPa` หรืออุณหภูมิเกิน `200 °C` ให้โปรแกรมสั่งปิดรีเลย์หัวเผา (ขาเอาต์พุตเชื่อมโยงเป็น 0) ทันที และส่งสถานะสัญญาณเตือนอันตรายไปยัง `V3` (Blynk LED widget) และส่งแจ้งเตือนข้อความฉุกเฉินไปยังมือถือ
        3. อนุญาตให้หัวหน้าวิศวกรส่งคำสั่งเปิด-ปิดรีเลย์แบบ Manual ผ่านปุ่มกดบนหน้าแดชบอร์ด (`V4`) **แต่จะสามารถเปิดระบบได้ต่อเมื่ออุณหภูมิและแรงดันอยู่ในเกณฑ์ปลอดภัยเท่านั้น** หากยังอยู่ในช่วงอันตราย บอร์ด ESP32 จะต้องปฏิเสธคำสั่งของคลาวด์และส่งข้อความเตือนกลับไปยัง `V5` (Terminal Widget)
    
    #### ภารกิจที่ 2: ระบบตรวจวัดข้อมูลทนทานต่อเครือข่ายขัดข้อง (Network Resilient Telemetry)
    *   **วัตถุประสงค์:** ฝึกให้นักศึกษารับมือกับสถานการณ์จริงเมื่ออินเทอร์เน็ตล่ม โดยให้บอร์ดจัดเก็บข้อมูลลงหน่วยความจำชั่วคราวแล้วส่งย้อนหลังเมื่อกลับมาเชื่อมต่อใหม่
    *   **อุปกรณ์ใน Wokwi:** บอร์ด ESP32 + เซนเซอร์ DHT22 (วัดอุณหภูมิและชื้นรอบเครื่องจักร)
    *   **ข้อกำหนดการเขียนโปรแกรม (C++):**
        1. ออกแบบโครงสร้างข้อมูลอาศัยการเช็คสถานะการเชื่อมต่อ `WiFi.status()` และ `Blynk.connected()` 
        2. หากเชื่อมต่อได้ปกติ ให้ส่งอุณหภูมิจาก DHT22 ขึ้น `V1` ทุกๆ 5 วินาที
        3. หากเน็ตหลุด (สามารถจำลองโดยเขียนโค้ดสั่งปลดการเชื่อมต่อ หรือปรับฟังก์ชันตรวจสอบ) ห้ามตัวโปรแกรมหยุดรัน ให้ ESP32 บันทึกค่าข้อมูลลงในหน่วยความจำ SPIFFS (แฟลชของบอร์ด) หรือจัดเก็บในรูปแบบอาร์เรย์แคช (Ring Buffer ในหน่วยความจำ RAM) พร้อมบันทึกดัชนีเวลาจำลอง
        4. เมื่อระบบตรวจสอบว่าการเชื่อมต่อกลับคืนมาปกติ (ตรวจพบเครือข่ายอีกครั้ง) ให้ทำการอ่านข้อมูลที่แคชไว้ในหน่วยความจำและดึงมาส่งคืนย้อนหลังเข้าสู่คลาวด์ Blynk เพื่อให้ระบบบันทึกแนวโน้มโดยข้อมูลไม่สูญหายไปในช่วงเวลาที่อินเทอร์เน็ตล่ม

## 3. รายละเอียดตัวอย่างโค้ดที่จะนำมาเสริมในบทเรียน (Proposed Code Snippets)

เพื่อความชัดเจน จะมีการเตรียมโค้ดตัวอย่างที่สมบูรณ์และถูกต้องสำหรับกรณีศึกษาหม้อไอน้ำ (Boiler System) ไว้สำหรับนำไปใส่ในบทที่ 8 โดยตรง ดังโครงสร้างตัวอย่างด้านล่างนี้:

```cpp
#define BLYNK_TEMPLATE_ID "TMPL000000"
#define BLYNK_TEMPLATE_NAME "BoilerMonitor"
#define BLYNK_AUTH_TOKEN "Your_Boiler_Token"

#define BLYNK_PRINT Serial
#include <WiFi.h>
#include <WiFiClient.h>
#include <BlynkSimpleEsp32.h>

// กำหนดขาฮาร์ดแวร์ ESP32
#define PIN_PRESSURE_ADC 34  // รับสัญญาณแอนะล็อกความดัน
#define PIN_TEMP_ADC 35      // รับสัญญาณแอนะล็อกอุณหภูมิ
#define PIN_BURNER_RELAY 25  // ควบคุมรีเลย์หัวเผา
#define PIN_ALARM_LED 26     // หลอดไฟเตือนอันตรายที่บอร์ด

// เกณฑ์ความปลอดภัยเชิงกล
const float LIMIT_MAX_PRESSURE = 1.50; // เมกะปาสกาล (MPa)
const float LIMIT_MAX_TEMP = 180.0;    // องศาเซลเซียส (°C)

char ssid[] = "Wokwi-GUEST";
char pass[] = "";

BlynkTimer timer;
bool systemEnabled = true; // สถานะการทำงานของระบบเผาต้มน้ำ

// ฟังก์ชันจำลองการดึงค่าแรงดันเชิงฟิสิกส์ (แปลงค่า ADC 12 บิต)
float readPressure() {
  int rawAdc = analogRead(PIN_PRESSURE_ADC);
  // แปลงจาก 0-4095 เป็นแรงดันไฟฟ้า 0-3.3V และแปลงเป็นแรงดัน MPa 0.0 - 2.0 MPa
  float voltage = (rawAdc / 4095.0) * 3.3;
  return (voltage / 3.3) * 2.0; 
}

// ฟังก์ชันจำลองการแปลงค่าอุณหภูมิเชิงฟิสิกส์
float readTemperature() {
  int rawAdc = analogRead(PIN_TEMP_ADC);
  // แปลงช่วงอุณหภูมิ 0 - 250 องศาเซลเซียส
  return (rawAdc / 4095.0) * 250.0;
}

// ฟังก์ชันหลักในการทำงานของ Edge Control loop และส่งค่าขึ้นคลาวด์
void monitorAndControlSystem() {
  float pressure = readPressure();
  float temperature = readTemperature();
  
  Serial.print("Boiler Pressure: "); Serial.print(pressure); Serial.print(" MPa | ");
  Serial.print("Boiler Temp: "); Serial.print(temperature); Serial.println(" C");

  // ส่งข้อมูลเข้าพินเสมือนของ Blynk
  Blynk.virtualWrite(V0, temperature);
  Blynk.virtualWrite(V1, pressure);

  // --- ตรรกะระดับ Edge (Local Interlock Safety Trip) ---
  if (pressure > LIMIT_MAX_PRESSURE || temperature > LIMIT_MAX_TEMP) {
    // ภัยคุกคามทางกายภาพ: ตัดการทำงานระดับฮาร์ดแวร์ทันทีโดยไม่พึ่งพาระบบเครือข่าย
    digitalWrite(PIN_BURNER_RELAY, LOW); 
    digitalWrite(PIN_ALARM_LED, HIGH);
    systemEnabled = false;
    
    // แจ้งเตือนข้อความเตือนภัยและเปิดไฟเตือนบน Blynk Dashboard
    Blynk.virtualWrite(V3, 255); // สั่งเปิดวิดเจ็ต LED ของ Blynk เป็นสีแดงสว่าง
    Blynk.virtualWrite(V5, "ALARM: Overpressure or Overtemperature! Boiler automatically tripped.");
    Blynk.logEvent("boiler_emergency", "Safety alert: Boiler limits exceeded!"); // สั่งให้ส่ง Push Notification
  } else {
    // อยู่ในเกณฑ์ความปลอดภัย ไฟเตือนบนคลาวด์ปิด
    Blynk.virtualWrite(V3, 0); 
    digitalWrite(PIN_ALARM_LED, LOW);
  }
}

// --- ตรรกะรับคำสั่งควบคุมระยะไกลจาก Blynk App (Cloud Control) ---
BLYNK_WRITE(V4) {
  int remoteCmd = param.asInt(); // รับคำสั่งเปิด/ปิดจากปุ่มบนแอป
  
  float currentPressure = readPressure();
  float currentTemp = readTemperature();

  // ตรวจสอบความปลอดภัยระดับเครื่องกลก่อนรันคำสั่งสั่งการระยะไกล
  if (remoteCmd == 1) {
    if (currentPressure < LIMIT_MAX_PRESSURE && currentTemp < LIMIT_MAX_TEMP) {
      digitalWrite(PIN_BURNER_RELAY, HIGH);
      systemEnabled = true;
      Blynk.virtualWrite(V5, "System started manually via Cloud.");
    } else {
      // ปฏิเสธคำสั่งเปิดระบบเนื่องจากฮาร์ดแวร์ยังอยู่ในภาวะอันตราย
      Blynk.virtualWrite(V4, 0); // คืนค่าปุ่มบนแอปให้กลับเป็นปิด (Sync UI)
      Blynk.virtualWrite(V5, "ERROR: Command Rejected. Boiler state is still unsafe.");
    }
  } else {
    digitalWrite(PIN_BURNER_RELAY, LOW);
    systemEnabled = false;
    Blynk.virtualWrite(V5, "System stopped manually via Cloud.");
  }
}

// จัดเก็บพินและพารามิเตอร์เริ่มต้นเมื่อเชื่อมต่อคลาวด์สำเร็จ
BLYNK_CONNECTED() {
  Blynk.syncAll(); // ตรวจสอบและดึงค่าสถานะ V4 จากปุ่มบน Blynk Cloud กลับมารันบนบอร์ด
}

void setup() {
  Serial.begin(115200);
  
  pinMode(PIN_BURNER_RELAY, OUTPUT);
  pinMode(PIN_ALARM_LED, OUTPUT);
  digitalWrite(PIN_BURNER_RELAY, LOW); // เริ่มต้นระบบความปลอดภัยด้วยการปิดรีเลย์ไว้ก่อน
  digitalWrite(PIN_ALARM_LED, LOW);

  Blynk.begin(BLYNK_AUTH_TOKEN, ssid, pass);
  
  // ให้โปรแกรมทำงานทุกๆ 2000 ms (2 วินาที)
  timer.setInterval(2000L, monitorAndControlSystem);
}

void loop() {
  Blynk.run();
  timer.run();
}
```
