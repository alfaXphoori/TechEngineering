## 1. การเพิ่มความเข้มข้นของเนื้อหา Cloud Computing ในงานวิศวกรรมเครื่องกลและอุตสาหกรรม (Rigorous Industrial IoT)

เนื้อหาเดิมอธิบายแนวคิด Cloud Computing ทั่วไป แต่ยังขาดความเชื่อมโยงกับสถาปัตยกรรมอุตสาหกรรมจริง สิ่งที่ต้องเพิ่มเข้ามาคือ:

### 1.1 การเปรียบเทียบระหว่าง Edge Computing และ Cloud Computing (Edge vs. Cloud Trade-offs)
*   **คำอธิบายหลักการ:** อธิบายว่าระบบควบคุมเครื่องจักรกลหลายประเภทต้องการเวลาตอบสนองในระดับมิลลิวินาที (Real-time control loop) เช่น ระบบควบคุมเซอร์โวมอเตอร์ หรือระบบไฮดรอลิกส์ ซึ่งไม่สามารถรอให้ข้อมูลเดินทางขึ้นไปประมวลผลบนคลาวด์แล้วส่งกลับลงมาได้ (เนื่องจากปัญหา Network Latency)
*   **การแบ่งบทบาท:**
    *   **Edge Computing (การประมวลผลที่ขอบเครือข่าย):** ทำหน้าที่ประมวลผลงานควบคุมที่วิกฤตด้านเวลา (Safety-critical & Real-time loops), การแปลงและกรองข้อมูลขั้นต้น (Preprocessing)
    *   **Cloud Computing (การประมวลผลบนคลาวด์):** ทำหน้าที่ประมวลผลวิเคราะห์ข้อมูลระยะยาว (Long-term data analytics), การทำนายความเสียหายล่วงหน้า (Predictive Maintenance), การปรับปรุงค่ากำหนดเชิงเปรียบเทียบข้ามเครื่องจักร (Fleet Optimization) และการเก็บประวัติย้อนหลัง (Historical Logging)

### 1.2 การลดความถี่และการดึงคุณลักษณะเฉพาะของข้อมูลที่ขอบ (Data Downsampling & Feature Extraction)
*   **ความจำเป็นในงานเครื่องกล:** ในการตรวจวัดความสั่นสะเทือนของแบริ่งมอเตอร์ (Vibration Monitoring) เราใช้เซนเซอร์วัดความเร่ง (Accelerometer) ที่ความถี่การสุ่มตัวอย่างสูงมาก (เช่น 1 kHz - 10 kHz) หากส่งข้อมูลดิบ (Raw Data) ทั้งหมดขึ้นคลาวด์ จะทำให้เกิดคอขวดของแบนด์วิดท์เครือข่ายและค่าใช้จ่ายคลาวด์ที่สูงมาก
*   **แนวทางการแก้ปัญหา:** อธิบายวิธีการทำงานของตัวประมวลผล Edge (เช่น ESP32) ในการรวบรวมข้อมูลความสั่นสะเทือนแบบดิบ แล้วคำนวณหาค่าสถิติเชิงคุณลักษณะ เช่น:
    *   **RMS (Root Mean Square):** เพื่อดูระดับพลังงานความสั่นสะเทือนรวม
    *   **Peak-to-Peak:** เพื่อตรวจจับการกระแทกที่รุนแรง
    *   **FFT (Fast Fourier Transform):** เพื่อวิเคราะห์ความถี่หลัก
    *   จากนั้นจึงส่งเพียงค่าดัชนีเหล่านี้ (เช่น ค่า RMS ทุกๆ 5 วินาที) ขึ้นสู่คลาวด์ แทนที่จะส่งข้อมูลดิบแสนรายการ

### 1.3 กลไกการป้องกันข้อมูลสูญหายและการส่งต่อข้อมูลหลังเชื่อมต่อใหม่ (Store-and-Forward / Offline Buffering)
*   **ปัญหาเครือข่ายในโรงงานจริง:** สัญญาณ Wi-Fi หรือ Cellular ในโรงงานอุตสาหกรรมมักมีการรบกวนทางคลื่นแม่เหล็กไฟฟ้าสูงจากมอเตอร์ขนาดใหญ่ ทำให้เกิดปัญหาการตัดการเชื่อมต่อชั่วคราว
*   **กลไกการแก้ไข:** อธิบายแนวคิดการจัดเก็บข้อมูลสำรองลงในหน่วยความจำชั่วคราวของบอร์ด (เช่น Internal Flash Memory ของ ESP32 ผ่าน SPIFFS/LittleFS หรือ SD Card Shield) เมื่อขาดการเชื่อมต่อกับคลาวด์ และเมื่อเครือข่ายกลับมาใช้งานได้ อุปกรณ์จะทยอยส่งข้อมูลที่ค้างอยู่ขึ้นไป (Store-and-Forward) เพื่อป้องกันไม่ให้ข้อมูลบันทึกขาดหาย (Data Loss)

## 2. การลงลึกกลไกและสถาปัตยกรรมของ Blynk (Blynk Architecture & Code Mechanics)

ส่วนนี้ควรขยายรายละเอียดเพื่อให้เห็นโครงสร้างเบื้องหลังการทำงาน และการเขียนโค้ดที่ถูกต้องตามหลักการพัฒนาซอฟต์แวร์สมังฝังตัว (Embedded Software Best Practices):

### 2.1 สถาปัตยกรรมการสื่อสารของ Blynk (Blynk Communication Model)
*   **โปรโตคอลเบื้องหลัง:** อธิบายว่า Blynk ใช้โปรโตคอลการสื่อสารระดับแอปพลิเคชัน (Application Layer) ที่เป็นกรรมสิทธิ์เฉพาะ (Proprietary Binary Protocol) ซึ่งทำงานอยู่บน TCP/IP หรือ WebSockets ทำให้มีขนาดแพ็กเก็ตข้อมูลเล็กมากและตอบสนองได้เร็ว (Latency ต่ำ)
*   **แผนผังการไหลของข้อมูล:**
    ```
    [ESP32 Hardware] <---(Blynk Protocol over TCP)---> [Blynk Cloud Server] <---(WebSockets/REST)---> [Blynk App / Web Dashboard]
    ```

### 2.2 โครงสร้างกลไกพินเสมือน (Virtual Pins Mechanics)
*   **คำนิยามทางเทคนิค:** พินเสมือน (Virtual Pin) ไม่ใช่พินทางกายภาพ แต่เป็นช่องหน่วยความจำ (Memory Address) บน Blynk Server ที่เป็นตัวกลางเก็บค่าของตัวแปร ตัวแปรนี้จะเชื่อมโยง (Mapped) ระหว่างอุปกรณ์ฮาร์ดแวร์และแอปพลิเคชัน
*   **กลไกการรับส่งข้อมูล:**
    *   **การส่งข้อมูลจากฮาร์ดแวร์ไปคลาวด์ (Push):** ใช้ `Blynk.virtualWrite(V_PIN, Value)` 
    *   **การรับคำสั่งจากคลาวด์มาฮาร์ดแวร์ (Pull/Callback):** ใช้มาโครคำสั่ง `BLYNK_WRITE(V_PIN)` ซึ่งเปรียบเสมือนฟังก์ชัน Interrupt หรือ Callback ที่จะถูกเรียกใช้โดยอัตโนมัติเมื่อค่าของพินเสมือนนั้นถูกเปลี่ยนแปลงจากฝั่ง Dashboard (เช่น ผู้ใช้งานเลื่อนสไลเดอร์หรือกดปุ่ม)
*   **ตัวอย่างโครงสร้างคำสั่งและการแปลงชนิดข้อมูล:**
    ```cpp
    BLYNK_WRITE(V2) {
      // ดึงค่าข้อมูลจาก Blynk server และแปลงชนิดข้อมูล
      int buttonState = param.asInt();        // แปลงเป็นจํานวนเต็ม (0 หรือ 1)
      float speedSetPoint = param.asFloat();   // แปลงเป็นทศนิยม
      String textMsg = param.asStr();        // แปลงเป็นสตริง

      if (buttonState == 1) {
        digitalWrite(RELAY_PIN, HIGH);
      } else {
        digitalWrite(RELAY_PIN, LOW);
      }
    }
    ```

### 2.3 วงจรชีวิตการทำงานของไลบรารี Blynk (ESP32 & Blynk Library Lifecycle)
*   **การเริ่มระบบเชื่อมต่อ (Connection Setup):**
    *   `Blynk.begin(auth, ssid, pass)`: ทำงานแบบ **Blocking** คือโค้ดจะหยุดรอที่บรรทัดนี้จนกว่าจะเชื่อมต่อ Wi-Fi และเซิร์ฟเวอร์ Blynk สำเร็จ หากไม่มีเราเตอร์ Wi-Fi หรือ Blynk Server ล่ม อุปกรณ์จะไม่สามารถเริ่มการทำงานต่อไปได้ ซึ่งเป็นอันตรายต่อการควบคุมเครื่องจักรกลจริง
    *   `Blynk.config(auth)` ร่วมกับ `Blynk.connectWiFi(ssid, pass)` หรือการจัดการ Wi-Fi แยกต่างหาก: เป็นแบบ **Non-blocking** ทำให้อุปกรณ์สามารถทำงานควบคุมระดับท้องถิ่น (Local loop) ได้แม้ไม่มีสัญญาณอินเทอร์เน็ต
*   **การควบคุมลูปหลัก (Main Loop Execution):**
    *   `Blynk.run()`: ทำหน้าที่รักษาการเชื่อมต่อ (Keep-alive ping/heartbeat) และตรวจจับเพื่อเรียกใช้ callback ฟังก์ชันต่างๆ (เช่น `BLYNK_WRITE`) ในลูปหลักจะต้องเรียกใช้อย่างรวดเร็วและไม่มี `delay()` มาขวาง
    *   `timer.run()`: ทำหน้าที่จัดการกับช่วงเวลาของ BlynkTimer เพื่อเรียกฟังก์ชันที่ตั้งไว้ตามกำหนด

### 2.4 เหตุใดคำสั่ง `delay()` จึงเป็นอันตรายในงาน IoT และกลไกของ BlynkTimer
*   **ทำไม `delay()` จึงทำงานไม่ได้กับ Blynk:** 
    *   คำสั่ง `delay(ms)` ใน Arduino เป็นแบบ Busy-waiting ซึ่งจะยึดครอง CPU และหยุดการประมวลผลโค้ดทั้งหมด
    *   หาก CPU หยุดทำงานนานเกินค่าเวลาคงชีพการเชื่อมต่อ (Heartbeat Timeout - ปกติประมาณ 10-15 วินาทีของ Blynk) เซิร์ฟเวอร์ Blynk จะคิดว่าอุปกรณ์ออฟไลน์และตัดการเชื่อมต่อ TCP
    *   เมื่อผ่านพ้นช่วง delay อุปกรณ์จะต้องเริ่มต้นทำ TLS/TCP Handshake ใหม่ทั้งหมด ทำให้เปลืองพลังงาน แบนด์วิดท์ และทำให้ระบบขาดความเสถียร
*   **การแก้ปัญหาด้วย BlynkTimer:**
    *   BlynkTimer ทำงานโดยอาศัยฟังก์ชัน `millis()` ของตัวเก็บเวลาภายในบอร์ด (Hardware timer ticker) 
    *   โครงสร้างเป็นแบบ Non-blocking polling: ทุกครั้งที่ `timer.run()` ถูกเรียกในลูปหลัก มันจะเปรียบเทียบเวลาปัจจุบันกับเวลาที่ตั้งจองไว้ หากยังไม่ถึงเวลาก็จะผ่านไปทันทีโดยไม่รั้งการทำงานของ CPU ทำให้ลูปยังคงรัน `Blynk.run()` ได้นับแสนครั้งต่อวินาที
*   **ข้อจำกัดการใช้งาน BlynkTimer:** แจ้งเตือนเรื่องจำนวนของ Event สูงสุดที่รองรับต่อหนึ่งออบเจกต์ (จำกัดที่ 16 ตัว) และการตั้งระยะเวลาของไทม์เมอร์ไม่ให้ชนกันจนทำให้เกิดความหน่วง

### 2.5 การควบคุมอัตราการส่งข้อมูลและ Flood Error (Rate Limiting)
*   **กลไก Rate Limiting:** Blynk server (โดยเฉพาะ Free Cloud Plan) มีการตั้งระบบคัดกรองสแปมและการใช้งานแบนด์วิดท์เกิน โดยจำกัดการส่งคำสั่ง `Blynk.virtualWrite()` ไม่เกิน 10-20 ครั้งต่อวินาที
*   **ผลกระทบ:** หากเขียนส่งข้อมูลในลูปหลักโดยไม่มีการหน่วงเวลา อุปกรณ์จะถูกบล็อกและตัดการเชื่อมต่อเนื่องจากข้อผิดพลาด **"Flood Error"**
*   **การออกแบบที่ถูกต้อง:** การกระจายจังหวะส่งค่าเซนเซอร์ที่ต่างชนิดกัน (เช่น อุณหภูมิส่งทุก 5 วินาที ความดันส่งทุก 2 วินาที) โดยใช้ BlynkTimer แยกรอบการทำงานกัน หรือรวมข้อมูลส่งเป็นชุด

## 3. ตัวอย่างการใช้งานจริงและกรณีศึกษาเชิงวิศวกรรมเครื่องกล (Mechanical/Industrial Case Studies)

เพื่อดึงความสนใจของนักศึกษาวิศวกรรมเครื่องกล ควรเพิ่มกรณีศึกษาที่เป็นรูปธรรมและครอบคลุมฟิสิกส์ของการวัด:

### กรณีศึกษาที่ 1: ระบบตรวจติดตามความปลอดภัยของหม้อไอน้ำอุตสาหกรรม (Factory Steam Boiler Safety Monitoring)
*   **ความสำคัญเชิงวิศวกรรม:** หม้อไอน้ำ (Boiler) มีความดันสูง หากระบบควบคุมแรงดันล้มเหลวอาจเกิดการระเบิดครั้งใหญ่ (BLEVE) วิศวกรจึงต้องมีการตรวจวัดพารามิเตอร์แบบซ้ำซ้อน (Redundancy) และดูข้อมูลระยะไกลได้
*   **พารามิเตอร์และการเชื่อมโยงข้อมูล:**
    1.  **ความดันไอน้ำ (Steam Pressure):** ส่งเข้าพิน `V1` แสดงผลด้วยมาตรวัดความดัน (Gauge) เพื่อให้เจ้าหน้าที่เห็นสถานะความปลอดภัยทันที
    2.  **อุณหภูมิผนังเตา (Shell Temperature):** ส่งเข้าพิน `V2` แสดงผลในรูปแบบกราฟประวัติย้อนหลัง (SuperChart) เพื่อวิเคราะห์ประสิทธิภาพการถ่ายเทความร้อน
    3.  **ระดับน้ำในหม้อต้ม (Water Level):** ส่งเข้าพิน `V3` แสดงผลแบบแถบระดับ (Level Widget) เพื่อป้องกันปัญหาการแห้งขอดของหม้อน้ำ (Dry boiling)
    4.  **ปุ่มเปิดวาล์วระบายแรงดันฉุกเฉิน (Emergency Steam Release Valve):** ใช้พินเสมือน `V4` เชื่อมกับปุ่มกดบนมือถือ (Button Widget) เพื่อสั่งเปิดโซลินอยด์วาล์วระยะไกลกรณีแรงดันสูงเกินขีดอันตราย
*   **ระบบความปลอดภัยระดับฮาร์ดแวร์ (Safety Interlock):** ถึงแม้จะมีการควบคุมทางไกลผ่านคลาวด์ แต่การออกแบบต้องกำหนดให้มีคำสั่งความปลอดภัยภายในบอร์ด ESP32 เองด้วย (เช่น หากอ่านแรงดันได้เกิน 8 bar ให้สั่งตัดระบบทำความร้อนและเปิดวาล์วระบายแรงดันทันทีโดยไม่ต้องรอคำสั่งจากคลาวด์)

### กรณีศึกษาที่ 2: ระบบตรวจวัดสุขภาพแบริ่งแกนหมุนของเครื่องจักร CNC (CNC Spindle Bearing Health Monitoring)
*   **ความสำคัญเชิงวิศวกรรม:** การตรวจจับแบริ่งเสียหายในระยะแรกเริ่มผ่านความสั่นสะเทือนและการสะสมความร้อน ช่วยป้องกันเครื่องจักรหยุดทำงานโดยไม่ได้วางแผน (Unplanned Downtime)
*   **พารามิเตอร์และการเชื่อมโยงข้อมูล:**
    1.  **ความสั่นสะเทือน RMS (Vibration RMS):** คำนวณค่าจากเซนเซอร์ความเร่งที่ขอบเครือข่าย แล้วส่งค่าเฉลี่ยทุก 5 วินาทีเข้าพิน `V5`
    2.  **อุณหภูมิผิวตลับลูกปืน (Bearing Temperature):** ส่งเข้าพิน `V6` 
    3.  **ความเร็วรอบแกนหมุน (Spindle Speed - RPM):** ส่งเข้าพิน `V7`
*   **การเชื่อมโยงข้อมูลคลาวด์:** การใช้กราฟแสดงผลแบบสองแกน (X-Y Plot หรือคู่กราฟร่วมเวลา) เพื่อดูว่าความสั่นสะเทือนเพิ่มสูงขึ้นเฉพาะในช่วงความเร็วรอบวิกฤต (Critical speed / Resonance) หรือเกิดความสั่นสะเทือนจากตลับลูกปืนเสียหายจริง

### กรณีศึกษาที่ 3: ระบบตรวจจับจุดลมอัดรั่วไหลในโรงงาน (Pneumatic System Air Leak Detection)
*   **ความสำคัญเชิงวิศวกรรม:** ระบบลมอัด (Compressed Air) สิ้นเปลืองพลังงานไฟฟ้าสูงมาก จุดลมรั่วเล็กๆ ทั่วโรงงานรวมกันทำให้เกิดความสูญเสียทางเศรษฐกิจมหาศาล
*   **พารามิเตอร์และการเชื่อมโยงข้อมูล:**
    1.  **อัตราการไหลของลม (Air Flow Rate - L/min):** ส่งเข้าพิน `V8`
    2.  **ความดันของระบบ (System Pressure - bar):** ส่งเข้าพิน `V9`
    3.  **การเปรียบเทียบในเวลากลางคืน:** เขียนโปรแกรมให้วิเคราะห์บนคลาวด์หรือ ESP32 ว่า ในช่วงเวลาที่หยุดการผลิต (ชั่วโมงที่ไม่มีการทำงาน) หากยังมีอัตราการไหลของลมค้างอยู่และมีการสตาร์ทของคอมเพรสเซอร์ลม แสดงว่ามีจุดลมรั่วในระบบ และแจ้งเตือนผ่านคลาวด์ส่งอีเมลไปยังผู้ดูแลอาคาร

## 4. โครงสร้างรายละเอียดสำหรับ Data Ingestion Protocols และฐานข้อมูลเปรียบเทียบ

การเปลี่ยนผ่านจากแพลตฟอร์มสำเร็จรูปไปสู่สถาปัตยกรรมระดับอุตสาหกรรม ต้องอาศัยการทำความเข้าใจโปรโตคอลหลักและโครงสร้างการเก็บข้อมูล:

### 4.1 รายละเอียดโปรโตคอลการรับเข้าข้อมูลสำหรับ IoT (IoT Data Ingestion Protocols)

วิศวกรควรเลือกโพรโทคอลให้เหมาะกับปริมาณแบนด์วิดท์ พลังงาน และระดับความเสถียรของเครือข่าย:

*   **MQTT (Message Queuing Telemetry Transport):**
    *   **สถาปัตยกรรม:** ทำงานแบบ Publish/Subscribe ผ่านตัวกลางที่เรียกว่า MQTT Broker (เช่น Mosquitto, HiveMQ)
    *   **จุดเด่น:** มีขนาดส่วนหัวข้อความ (Header Overhead) ต่ำมากเพียง 2 ไพต์เป็นค่าเริ่มต้น ทำให้ประหยัดแบนด์วิดท์ เหมาะกับการส่งข้อมูลแบบสตรีมมิ่งตลอดเวลา
    *   **ความเสถียร (QoS - Quality of Service):** รองรับระดับความมั่นใจในการส่งข้อมูล 3 ระดับ (QoS 0: ส่งครั้งเดียวไม่สนผลลัพธ์, QoS 1: ส่งอย่างน้อยหนึ่งครั้งถึงปลายทางชัวร์, QoS 2: ส่งเพียงครั้งเดียวและถึงปลายทางถูกต้องแน่นอน)
    *   **ฟังก์ชันตรวจสอบการเชื่อมต่อ:** มีคุณลักษณะ "Last Will and Testament (LWT)" ที่โบรกเกอร์จะแจ้งให้อุปกรณ์อื่นรู้ทันทีหากตัวส่งสัญญาณขาดหายไปอย่างกะทันหัน
*   **HTTP/HTTPS (REST API):**
    *   **สถาปัตยกรรม:** Request/Response โดยใช้วิธี POST, GET, PUT ผ่านพอร์ตมาตรฐาน 80/443
    *   **จุดเด่น:** ใช้งานง่าย ทำงานเข้ากับเว็บเซิร์ฟเวอร์แบบดั้งเดิมได้ทันที เหมาะกับการส่งข้อมูลขนาดใหญ่เป็นรอบๆ (Batch processing) หรือการอัปเดตไฟล์เฟิร์มแวร์
    *   **ข้อจำกัด:** มีค่าหัวเอกสาร (HTTP Header) ที่เป็นตัวอักษรขนาดใหญ่ (หลายร้อยไพต์ขึ้นไป) ไม่เหมาะกับการส่งข้อมูลความถี่สูงจากอุปกรณ์แบตเตอรี่ต่ำ
*   **CoAP (Constrained Application Protocol):**
    *   **สถาปัตยกรรม:** ออกแบบมาตามแนวคิด REST (Request/Response) เหมือน HTTP แต่ทำงานบน UDP แทน TCP เพื่อลดทอนภาระของการทำ Handshake
    *   **จุดเด่น:** มีความเบามาก ใช้ข้อมูลหัวของแพ็กเก็ตคงที่เพียง 4 ไพต์ รองรับการสื่อสารแบบกระจายเสียงกลุ่ม (Multicast) ทำให้ค้นหาอุปกรณ์ในระบบเครือข่ายท้องถิ่นได้ง่าย เหมาะสำหรับอุปกรณ์ขนาดเล็กมากที่ใช้พลังงานจากแบตเตอรี่/โซลาร์เซลล์

#### ตารางที่ 4: ตารางเปรียบเทียบคุณลักษณะของโปรโตคอลหลัก IoT

| หัวข้อเปรียบเทียบ | MQTT | HTTP (REST) | CoAP |
| :--- | :--- | :--- | :--- |
| **สถาปัตยกรรมสื่อสาร** | Publish / Subscribe | Request / Response | Request / Response (REST over UDP) |
| **โปรโตคอลขนส่ง (Transport)** | TCP | TCP | UDP |
| **ขนาด Overhead ขั้นต่ำ** | ~2 Bytes | >100 Bytes | ~4 Bytes |
| **การใช้พลังงาน/แบนด์วิดท์** | ต่ำมาก | สูง | ต่ำมาก |
| **การจัดการสถานะการต่อ** | มีการเชื่อมโยงค้างไว้ (Connection-oriented) | สิ้นสุดในครั้งเดียว (Stateless) | ส่งแบบเดี่ยว (Connectionless) |
| **ความเหมาะสมหลัก** | การวัดผลส่งค่าอย่างต่อเนื่องเรียลไทม์ | การดึงไฟล์ อัปเกรดซอฟต์แวร์ อัปโหลดรายชั่วโมง | อุปกรณ์เซนเซอร์ไร้สายกินไฟต่ำที่หลับใหลได้ (Deep Sleep) |

### 4.2 การเปรียบเทียบฐานข้อมูลอนุกรมเวลา (Time-Series Database) และฐานข้อมูลเชิงสัมพันธ์ (SQL)

ขยายความเปรียบเทียบทางเทคนิคในการจัดการข้อมูล IoT ที่เก็บสะสมเป็นเวลานาน:

*   **ACID vs Append-Only:** SQL เน้นคุณสมบัติ ACID เพื่อความปลอดภัยในการทำธุรกรรมทางการเงินและระบบคลังสินค้า ทำให้ต้องล็อกแถวข้อมูลเพื่อตรวจสอบความขัดแย้งขณะบันทึก ส่วน TSDB ออกแบบมาสำหรับการบันทึกอย่างเดียว (Append-Only) ข้อมูลเป็นแบบไม่มีโครงสร้างซับซ้อน จึงเขียนลงดิสก์ได้เร็วมากโดยไม่มีภาระตรวจสอบความสัมพันธ์
*   **การจัดการพื้นที่จัดเก็บและบีบอัดข้อมูล (Data Compression):** ข้อมูลความสั่นสะเทือนหรืออุณหภูมิที่เกิดขึ้นทุกวินาทีจะมีค่าที่ใกล้เคียงกันมาก TSDB นำอัลกอริทึมการบีบอัดเฉพาะ เช่น Double-Delta Compression (สำหรับเวลาที่เพิ่มขึ้นทีละวินาทีอย่างคงที่) และ XOR Compression (สำหรับค่าทศนิยมของตัวแปรที่เปลี่ยนทีละน้อย) ทำให้ใช้พื้นที่จัดเก็บบนคลาวด์ลดลงกว่า 90% เมื่อเทียบกับการบันทึกในตาราง SQL แบบดิบ
*   **นโยบายการจัดเก็บข้อมูลและการลดความละเอียด (Retention Policies & Downsampling):** TSDB มีความสามารถในตัวสำหรับการดึงข้อมูลเก่าย้อนหลังมาคำนวณและลดความถี่ลงโดยอัตโนมัติ เช่น ข้อมูลอุณหภูมิรายวินาทีที่มีอายุเกิน 30 วัน จะถูกนำมาเฉลี่ยเป็นข้อมูลรายชั่วโมง และลบข้อมูลดิบระดับวินาทีทิ้งเพื่อประหยัดพื้นที่จัดเก็บ ซึ่งการจัดการนี้ทำได้ยากและช้ามากใน SQL

## 5. การขยายเนื้อหาความปลอดภัยและความเป็นส่วนตัวของ IoT (Advanced IoT Security)

ในบทเรียนระดับวิศวกรรม ต้องเปลี่ยนผ่านจากการกล่าวเตือนลอยๆ ไปสู่แนวปฏิบัติทางเทคนิคที่จับต้องได้:

### 5.1 การจัดการความปลอดภัยของ Auth Token
*   **ความเสี่ยงของการ Hardcode:** การเขียน Token ไว้ในโค้ดทำให้เสี่ยงต่อการหลุดรอดเมื่ออัปโหลดโค้ดขึ้น GitHub หรือแบ่งปันไฟล์โค้ด
*   **แนวทางการป้องกัน:**
    1.  **การแยกเก็บไฟล์ภายนอก:** การเก็บรหัสในไฟล์คอนฟิกแยกต่างหาก (เช่น `secrets.h`) และใช้ไฟล์ `.gitignore` บล็อกไม่ให้อัปโหลดขึ้นอินเทอร์เน็ต
    2.  **Blynk.Inject (Dynamic Provisioning):** อธิบายแนวทางมาตรฐานระดับอุตสาหกรรม โดยเริ่มแรก ESP32 จะเปิดโหมด Wi-Fi Access Point (AP) ส่วนตัว เพื่อให้ผู้ใช้ล็อกอินผ่านสมาร์ทโฟนแล้วส่งค่า SSID, Password ของบ้าน และ Auth Token เข้าไปบันทึกไว้ในหน่วยความจำถาวร (EEPROM/NVS) ของบอร์ดโดยตรง โค้ดที่เขียนจึงไม่มีข้อความรหัสผ่านอยู่เลย ทำให้ปลอดภัยและยืดหยุ่นสูง

### 5.2 การเข้ารหัสข้อมูลที่ระดับการขนส่ง (Transport Layer Security - TLS/SSL)
*   **การเข้ารหัสในการเชื่อมต่อ:** อธิบายความแตกต่างระหว่าง HTTP/MQTT (ส่งข้อมูลแบบข้อความเปล่าดักฟังง่าย) กับ HTTPS/MQTTS (เข้ารหัสข้อมูลแบบกุญแจสมมาตรและไม่สมมาตรด้วย TLS)
*   **การใช้งานจริงบน ESP32:** การเรียกใช้งานไลบรารี `WiFiClientSecure` แทน `WiFiClient` ปกติ และการนำเข้า **Root CA Certificate** ของเซิร์ฟเวอร์คลาวด์มาใส่ไว้ในหน่วยความจำของ ESP32 เพื่อให้อุปกรณ์สามารถยืนยันตัวตนได้ว่ากำลังคุยกับ Blynk Cloud ตัวจริง ไม่ใช่เซิร์ฟเวอร์ปลอมที่มาสวมรอย
*   **ผลกระทบต่อทรัพยากร (Resource Constraints):** อธิบายว่าการทำ TLS Handshake ต้องใช้กำลังประมวลผลของไมโครคอนโทรลเลอร์สูง และต้องกันพื้นที่ RAM ประมาณ 20-30 KB สำหรับเก็บกุญแจเข้ารหัส ซึ่งเป็นสิ่งที่วิศวกรต้องคำนึงถึงในการเลือกสเปกไมโครคอนโทรลเลอร์

### 5.3 ความปลอดภัยของการอัปเดตเฟิร์มแวร์ผ่านอากาศ (Secure OTA Updates)
*   **ภัยคุกคามจากการอัปเดตระยะไกล:** หากระบบรับไฟล์อัพเดทโค้ดโดยไม่มีระบบป้องกัน แฮกเกอร์อาจส่งโค้ดประสงค์ร้ายเข้ามาแฟลชทับเพื่อควบคุมเครื่องจักร หรือทำให้อุปกรณ์พังเสียหาย (Bricking)
*   **กลไกความปลอดภัยในการทำ OTA:**
    1.  **HTTPS-based OTA:** การดาวน์โหลดไฟล์เฟิร์มแวร์ผ่านช่องทางที่เข้ารหัสปลอดภัย
    2.  **การใช้ลายเซ็นดิจิทัลและ Secure Boot (Firmware Signing):** อุปกรณ์จะตรวจสอบคีย์สาธารณะ (Public Key) ที่ฝังไว้ในฮาร์ดแวร์ เพื่อเช็กว่าไฟล์เฟิร์มแวร์ใหม่ถูกเซ็นชื่อยืนยันจากวิศวกรผู้พัฒนาจริง (ใช้ Private Key ส่วนตัวเซ็นมา) หากลายเซ็นดิจิทัลไม่ถูกต้อง อุปกรณ์จะปฏิเสธการแฟลชโค้ดทันที

## 6. แบบฝึกหัดและกิจกรรมเชิงปฏิบัติการผ่าน Wokwi ที่ปรับปรุงใหม่ (New Exercises & Lab Tasks)

ปรับเปลี่ยนระดับความยากและสร้างงานแบบปฏิบัติการเชิงลึกที่สอดรับกับความต้องการในงานวิศวกรรม:

### 6.1 แบบฝึกหัดเชิงทฤษฎีและการคำนวณ (Theoretical & Quantitative Problems)

1.  **การคำนวณปริมาณการส่งผ่านข้อมูลและแบนด์วิดท์ (Bandwidth & Payload Math):**
    *   *โจทย์:* สถานีตรวจวัดสภาพอากาศในอู่ต่อเรือ มีการติดตั้งเซนเซอร์ 5 ตัว ส่งข้อมูลทุกๆ 10 วินาที โดยข้อมูลดิบมีขนาด 60 ไพต์ 
        *   หากส่งด้วย **HTTP POST** มี Overhead ของส่วนหัวโปรโตคอลเฉลี่ย 350 ไพต์ต่อการส่งหนึ่งครั้ง
        *   หากส่งด้วย **MQTT Publish** มี Overhead รวมเฉลี่ย 20 ไพต์ต่อการส่งหนึ่งครั้ง (รวมการเชื่อมต่อคงค้าง)
        *   จงคำนวณและเปรียบเทียบปริมาณข้อมูลอินเทอร์เน็ตที่อุปกรณ์ใช้ในระยะเวลา 1 เดือน (30 วัน) ทั้งสองกรณีในหน่วย Megabytes (MB)
2.  **การวิเคราะห์เหตุการณ์คอขวดเชิงเวลา (Execution Time Analysis):**
    *   *โจทย์:* นักศึกษาคนหนึ่งเขียนโค้ดระบบเฝ้าระวังมอเตอร์ โดยใส่คำสั่ง `delay(5000)` ไว้ในลูปหลักของ ESP32 เพื่อจำลองการรออ่านค่าอุณหภูมิ และใช้ Blynk ในการควบคุมไฟแสดงผล จงวิเคราะห์ว่า:
        *   จะเกิดอะไรขึ้นกับการตอบสนองของปุ่มกดเปิด-ปิดไฟบน Blynk Dashboard? (อธิบายความหน่วงที่เกิดขึ้นทางทฤษฎี)
        *   การเชื่อมต่อระหว่างบอร์ดและ Blynk Server จะยังเสถียรอยู่หรือไม่? เพราะเหตุใด? จงระบุกราฟการเชื่อมต่อที่นักศึกษาจะสังเกตเห็นจาก Serial Monitor

### 6.2 แบบฝึกหัดภาคปฏิบัติบนโปรแกรมจำลอง Wokwi (Practical Tasks)

#### ใบงานที่ 8.1: ระบบตัดการทำงานอัตโนมัติของหม้อไอน้ำร่วมกับระบบแจ้งเตือนแบบเรียลไทม์ (Boiler Overpressure Safety Interlock)
*   **เป้าหมาย:** สร้างระบบควบคุมแบบไฮบริดที่สามารถควบคุมอัจฉริยะจากคลาวด์ แต่มีระบบป้องกันภัยเชิงกลคอยขัดจังหวะความปลอดภัยในระดับท้องถิ่น (Local Fail-safe Override)
*   **อุปกรณ์และวงจรใน Wokwi:**
    *   ESP32
    *   สไลด์โพเทนชิโอมิเตอร์ (Potentiometer) จำลองแรงดันไอน้ำ (Pressure Sensor) ต่อเข้าพินอนาล็อก `ADC34`
    *   ไฟ LED สีแดง จำลองโซลินอยด์วาล์วระบายไอน้ำ (Solenoid Relief Valve) ต่อเข้าพิน `GPIO14`
    *   ไฟ LED สีเขียว จำลองรีเลย์ปั๊มน้ำป้อนระบบ (Water Feed Pump Relay) ต่อเข้าพิน `GPIO12`
    *   Blynk IoT Cloud Dashboard (สร้างแม่แบบที่มี Virtual Pins)
*   **การเชื่อมโยง Virtual Pins:**
    *   `V1` (Data In - Gauge/Chart): ส่งค่าแรงดันไอน้ำจำลอง (หน่วย: bar)
    *   `V2` (Data In - LED): ส่งสถานะของวาล์วระบายไอน้ำ (LED สีแดง)
    *   `V3` (Data Out - Button): ปุ่มเปิด-ปิดวาล์วด้วยการสั่งการจากระยะไกล (Manual Override)
*   **ข้อกำหนดของโค้ดและการทำงาน:**
    1.  **Non-blocking sampling:** ใช้ `BlynkTimer` รันการอ่านค่าโพเทนชิโอมิเตอร์และอัปโหลดค่าแรงดันขึ้น `V1` ทุกๆ 1 วินาที ห้ามใช้ `delay()` ในโค้ด
    2.  **Local Safety Logic (Interlock):** หากแรงดันไอน้ำเกิน **8.0 bar** ระบบในบอร์ด ESP32 ต้องสั่งตัดการทำงานปั๊มน้ำทันที (LED สีเขียวดับ) และสั่งเปิดวาล์วระบายแรงดัน (LED สีแดงติด) โดยทันทีในระดับฮาร์ดแวร์เพื่อความปลอดภัยสูงสุด โดยไม่ต้องรอการเชื่อมต่อคลาวด์ พร้อมทั้งเปลี่ยนสถานะบนวิดเจ็ตไฟบนหน้าจอ `V2` เป็นสีแดง
    3.  **Manual Cloud Override:** หากสถานะแรงดันปกติ (< 8.0 bar) ผู้ใช้ต้องสามารถกดปุ่มที่หน้าจอ `V3` เพื่อควบคุมเปิด-ปิด วาล์วระบายไอน้ำ (LED สีแดง) แมนนวลได้ แต่ถ้าหากแรงดันเกิน 8.0 bar ปุ่มแมนนวลจะต้องไม่มีผลควบคุมเพื่อความปลอดภัย

#### ใบงานที่ 8.2: ระบบส่งข้อมูลเซนเซอร์แบบแปรผันตามระดับสัญญาณอันตราย (Dynamic Telemetry Rate Controller)
*   **เป้าหมาย:** เรียนรู้การจัดการแบนด์วิดท์อย่างชาญฉลาดและการหลีกเลี่ยง Rate limit (Flood) ของคลาวด์ โดยปรับเปลี่ยนช่วงความถี่การส่งค่าตามสภาพแวดล้อม
*   **อุปกรณ์ใน Wokwi:**
    *   ESP32
    *   เซนเซอร์ DHT22 (วัดอุณหภูมิและความชื้นห้องควบคุมเครื่องจักร) ต่อเข้าพิน `GPIO15`
    *   ปุ่มกดกายภาพ (Push Button) สำหรับขออัปเดตข้อมูลฉุกเฉิน ต่อเข้าพิน `GPIO13`
*   **การเชื่อมโยง Virtual Pins:**
    *   `V10` (Gauge): อุณหภูมิ (°C)
    *   `V11` (Gauge): ความชื้นสัมพัทธ์ (%)
    *   `V12` (Value Display): ความเร็วในการส่งข้อมูลในขณะนั้น (วินาทีต่อครั้ง)
*   **ข้อกำหนดของโค้ดและการทำงาน:**
    1.  **Normal State:** ในภาวะปกติที่อุณหภูมิไม่เกิน **40 °C** ให้ทำการส่งข้อมูลอุณหภูมิและความชื้นขึ้น Blynk ทุกๆ **10 วินาที** เพื่อประหยัดพื้นที่จัดเก็บและปริมาณการสื่อสาร
    2.  **Critical State:** หากอุณหภูมิเพิ่มสูงขึ้นเกิน **40 °C** (จำลองความร้อนของเครื่องจักรที่เริ่มมีปัญหา) โค้ดจะต้องปรับแก้เวลาการส่งข้อมูลของ `BlynkTimer` โดยอัตโนมัติให้ส่งถี่ขึ้นเป็นทุกๆ **2 วินาที** ทันที เพื่อให้ผู้ดูแลระบบมองเห็นพฤติกรรมการเพิ่มขึ้นของอุณหภูมิได้อย่างละเอียดแบบเรียลไทม์ และเปลี่ยนกลับเป็น 10 วินาทีเมื่ออุณหภูมิลดลงต่ำกว่า 38 °C (ป้องกันอาการแกว่งรอบขีดจำกัดด้วยกลไก Hysteresis)
    3.  **On-Demand Request:** หากผู้ใช้งานกดปุ่มกายภาพที่ตัวบอร์ด (พิน 13) บอร์ดจะต้องทำฟังก์ชันส่งค่าปัจจุบันในทันที (Manual force push) 1 ครั้ง แต่ต้องมีระบบเขียนโค้ดเพื่อกรองสัญญาณและดักจับไม่ให้ส่งถี่เกิน 1 ครั้งต่อวินาที (Rate Limiter / Debouncing) เพื่อป้องกัน Flood Error
