## 1. การเพิ่มความเข้มข้นทางทฤษฎีใน Cloud Computing สำหรับงานวิศวกรรมเครื่องกลและอุตสาหกรรม (Rigorous Cloud Theory for Mechanical/Industrial IoT)

ในบทเรียนเดิม การอธิบายระบบคลาวด์ยังอยู่ในระดับทั่วไป แผนการปรับปรุงจะเน้นการเชื่อมโยงทฤษฎีคอมพิวเตอร์เข้ากับข้อจำกัดทางกายภาพและข้อกำหนดทางวิศวกรรมเครื่องกลจริง โดยเพิ่มเติมหัวข้อดังนี้:

### 1.1 การเปรียบเทียบและการตัดสินใจเลือกระหว่าง Edge Computing และ Cloud Computing (Edge vs. Cloud Trade-offs)
ให้นักศึกษาเข้าใจเหตุผลทางวิศวกรรมในการเลือกตำแหน่งการประมวลผลข้อมูลผ่าน 4 ปัจจัยหลัก:
*   **แบนด์วิดท์และปริมาณข้อมูล (Bandwidth & Data Volume):** ยกตัวอย่าง เซนเซอร์วัดการสั่นสะเทือน (Vibration Sensor) เช่น Accelerometer ที่ต้องสุ่มวัดความถี่สูง (เช่น 5 kHz) จะสร้างข้อมูลดิบขนาดประมาณ 30 KB ต่อวินาที หรือ 2.6 GB ต่อวันต่อเซนเซอร์ตัวเดียว หากมีเครื่องจักร 100 เครื่องในโรงงาน การส่งข้อมูลดิบทั้งหมดขึ้นคลาวด์โดยตรงจะทำให้เกิดปัญหาคอขวดของเครือข่ายและค่าใช้จ่ายคลาวด์มหาศาล ทางออกคือการทำ **Edge Processing** เพื่อคำนวณหาค่าเฉลี่ยหรือค่า RMS (Root Mean Square) บนบอร์ด ESP32 แล้วจึงส่งเฉพาะข้อมูลสรุปขึ้นคลาวด์ทุกๆ 5 วินาที
*   **ความหน่วงเวลา (Latency):** สำหรับลูปควบคุมความปลอดภัย (Safety Control Loop) เช่น ระบบตัดการทำงานฉุกเฉินของหม้อไอน้ำ (Boiler Emergency Shutdown) หากใช้ระบบคลาวด์ในการตัดสินใจ ความหน่วงจากการรับ-ส่งข้อมูลผ่านอินเทอร์เน็ต (Round-trip time) ซึ่งอาจสูงถึง 100–500 ms และความเสี่ยงในการขาดการเชื่อมต่อ จะทำให้ระบบไม่ปลอดภัยอย่างรุนแรง การควบคุมในลักษณะนี้จำเป็นต้องตัดสินใจที่ **Edge Device (ESP32)** ในระดับมิลลิวินาที ส่วนระบบคลาวด์จะใช้สำหรับมอนิเตอร์และเก็บประวัติเท่านั้น
*   **ความน่าเชื่อถือเมื่อขาดการเชื่อมต่อ (Reliability & Offline Operation):** ระบบคลาวด์ขึ้นอยู่กับเครือข่ายอินเทอร์เน็ต หากสายสัญญาณเสียหาย อุปกรณ์ที่ Edge จะต้องสามารถรันระบบควบคุมพื้นฐานและบันทึกข้อมูลเก็บไว้ในหน่วยความจำชั่วคราว (เช่น SD Card หรือ Flash Memory) ได้โดยไม่ทำงานล้มเหลว
*   **ขีดความสามารถการคำนวณ (Computational Capacity):** ระบบคลาวด์มีความสามารถประมวลผลสูงมาก เหมาะสำหรับการทำนายการเสื่อมสภาพของเครื่องจักรด้วย Machine Learning หรือการวิเคราะห์ข้อมูลย้อนหลังหลายปี ซึ่งไม่สามารถทำได้บนหน่วยประมวลผลขนาดเล็กของบอร์ดไมโครคอนโทรลเลอร์

### 1.2 สถาปัตยกรรมการส่งข้อมูลคลาวด์ในอุตสาหกรรม (Industrial Cloud Telemetry Pipeline)
อธิบายสถาปัตยกรรมระดับระบบเพื่อให้เห็นภาพรวมการเดินทางของข้อมูล:
$$\text{Physical Sensor} \rightarrow \text{Edge MCU (ESP32)} \rightarrow \text{Local Gateway (เช่น Raspberry Pi)} \rightarrow \text{Cloud IoT Core/Broker} \rightarrow \text{Stream Processor} \rightarrow \text{Time-Series Database} \rightarrow \text{Dashboard}$$
นอกจากนี้จะเพิ่มคำอธิบายของแนวคิด **Digital Twin (Device Shadow)** ซึ่งเป็นการเก็บจำลองสถานะล่าสุดของอุปกรณ์ไว้บนคลาวด์ ทำให้อุปกรณ์สามารถสื่อสารแลกเปลี่ยนคำสั่งได้แม้ว่าในช่วงเวลานั้นตัวบอร์ดจริงจะปิดการทำงานอยู่ (Offline)

## 2. รายละเอียดสถาปัตยกรรมและการเขียนโปรแกรม Blynk (Blynk Architecture & Programming Details)

ปรับปรุงส่วนคู่มือการเขียนโค้ดและพัฒนาแอปพลิเคชัน Blynk ให้มีความเป็นระบบและอธิบายกลไกการทำงานเบื้องหลังอย่างละเอียด เพื่อแก้ปัญหาการใช้งานแบบลอกโค้ดโดยไม่เข้าใจ:

### 2.1 สถาปัตยกรรมการทำงานของ Blynk (Blynk Architecture Under the Hood)
*   **องค์ประกอบ 3 ส่วน:** 
    1.  **Blynk Library (Client):** รันอยู่บนบอร์ด ESP32 ทำหน้าที่สร้างการเชื่อมต่อแบบ TCP/IP Socket และจัดรูปแบบข้อมูลตาม Blynk Protocol
    2.  **Blynk Server (Cloud Server):** ตัวกลางทำหน้าที่จัดการสิทธิ์การเข้าถึง เส้นทางของข้อมูล (Data Routing) และการจัดเก็บบันทึกข้อมูลลงฐานข้อมูลอนุกรมเวลา
    3.  **Blynk App/Console (User Interface):** แสดงผลข้อมูลทางฝั่งผู้ใช้และส่งคำสั่งควบคุมกลับไปยังคลาวด์
*   **กลไก Keepalive:** อุปกรณ์และเซิร์ฟเวอร์จะมีการแลกเปลี่ยนแพ็กเก็ตสั้นๆ (Ping/Pong) ทุกๆ 10 วินาที เพื่อยืนยันว่าสถานะการเชื่อมต่อยังคงออนไลน์อยู่

### 2.2 พินเสมือน (Virtual Pins) และการสื่อสารสองทาง (Bidirectional Communication)
*   **คอนเซปต์ของ Virtual Pins:** อธิบายการแยกส่วนทางฮาร์ดแวร์และซอฟต์แวร์ออกจากกัน (Abstraction Layer) เพื่อแก้ปัญหาความไม่ยืดหยุ่นของการระบุขาฮาร์ดแวร์ตรงๆ บนแดชบอร์ด โดย Virtual Pins (V0 ถึง V255) เปรียบเสมือนกล่องรับจดหมายบนระบบคลาวด์
*   **ทิศทางการสื่อสารแบบละเอียด:**
    *   **Hardware to Cloud (Telemetry):** การเขียนค่าจากไมโครคอนโทรลเลอร์โดยใช้คำสั่ง `Blynk.virtualWrite(Vpin, value)`
    *   **Cloud to Hardware (Command & Control):** การรับค่าจาก Widget บนโทรศัพท์มือถือหรือหน้าเว็บ (เช่น ปุ่มกด, สไลเดอร์) โดยใช้ Macro `BLYNK_WRITE(Vpin)` ซึ่งเปรียบเสมือนฟังก์ชันขัดจังหวะการทำงาน (Callback function) เมื่อข้อมูลบนเซิร์ฟเวอร์ได้รับการอัปเดต พร้อมตัวอย่างการแปลงข้อมูลที่ได้รับด้วยเมทอด เช่น `param.asInt()`, `param.asFloat()`, และ `param.asString()`

```cpp
// ตัวอย่างการรับค่าจากสวิตช์บนแดชบอร์ดเพื่อควบคุมรีเลย์
BLYNK_WRITE(V0) {
  int relayState = param.asInt(); // แปลงข้อมูลจากคลาวด์เป็นเลขจำนวนเต็ม (0 หรือ 1)
  digitalWrite(RELAY_PIN, relayState); // สั่งงานขา GPIO จริงของ ESP32
  Serial.print("Relay state changed by user to: ");
  Serial.println(relayState);
}
```

### 2.3 รายละเอียดการใช้ไลบรารี ESP32 และรูปแบบการเชื่อมต่อ (Blynk ESP32 Library Configuration)
*   **การเชื่อมต่อแบบ Blocking (`Blynk.begin`):** อธิบายว่าการเรียกใช้ `Blynk.begin(auth, ssid, pass)` ในหน้า `setup()` จะเป็นกระบวนการแบบ Blocking หากเราจ่ายไฟให้อุปกรณ์แต่ไม่มีสัญญาณ Wi-Fi หรือ Blynk Server ขัดข้อง โค้ดจะหยุดรันอยู่ที่บรรทัดนี้ทั้งหมดและไม่สามารถรันลูปถัดไปได้
*   **การเชื่อมต่อแบบ Non-blocking (`Blynk.config` + `Blynk.connect`):** นำเสนอทางเลือกในการต่อแบบไม่บล็อกเพื่อใช้ในระบบอุตสาหกรรมจริง เพื่อให้เครื่องจักรยังสามารถประมวลผลลูปควบคุมความปลอดภัยในตัวบอร์ดต่อได้ แม้ว่าเครือข่ายอินเทอร์เน็ตจะตัดขาด โดยใช้ตัวอย่างโค้ดดังนี้:

```cpp
void setup() {
  Serial.begin(115200);
  WiFi.begin(ssid, pass);
  
  // กำหนดค่าคอนฟิกให้กับ Blynk โดยไม่ต้องบล็อกการรันโค้ด
  Blynk.config(BLYNK_AUTH_TOKEN);
  // พยายามเชื่อมต่อแบบ Async
  Blynk.connect();
}

void loop() {
  // สั่งงาน Blynk.run() เฉพาะเมื่อเครื่องเชื่อมต่ออยู่เท่านั้น
  if (Blynk.connected()) {
    Blynk.run();
  }
  
  // ลูปควบคุมเครื่องจักรและตรวจวัดความปลอดภัยสามารถทำงานต่อตรงนี้ได้ตลอดเวลา
  runSafetyChecks();
}
```

### 2.4 กลไกการทำงานของ BlynkTimer
*   **หลักการของ Cooperative Multitasking:** อธิบายว่า `BlynkTimer` รันงานอยู่บน Thread เดียวกันของซีพียู โดยจะคำนวณและตรวจสอบจังหวะเวลาผ่านฟังก์ชัน `timer.run()` ในลูปหลักอย่างสม่ำเสมอ หากถึงช่วงเวลาที่ระบุ ระบบจะสั่งเรียกฟังก์ชันย่อย (Callback function) เข้ามาประมวลผล
*   **การป้องกัน Timer Collision:** แนะนำวิธีการหลีกเลี่ยงการจัดตารางเวลาของ Timer หลายตัวให้ตรงมิลลิวินาทีเดียวกัน เพื่อไม่ให้เกิดภาวะข้อมูลซ้อนทับกันเกินกำลังส่งของ Wi-Fi โดยใช้วิธี Offset เวลา (เช่น ตั้งค่าส่งตัวที่หนึ่ง 1000L มิลลิวินาที และตั้งค่าส่งตัวที่สอง 1010L มิลลิวินาที)
*   **ขีดจำกัด:** `BlynkTimer` สามารถสร้าง Task ย่อยได้สูงสุด 16 ตัวต่อหนึ่งออบเจกต์

### 2.5 การจำกัดอัตราการส่งข้อมูลและเหตุผลทางเครือข่าย (Rate Limiting & Flood Prevention)
*   **Blynk Flood Error:** การส่งคำสั่ง `Blynk.virtualWrite()` บ่อยเกินไป (เช่น ส่งทุกๆ ลูปของ `void loop()` หรือส่งเร็วกว่า 10 ครั้งต่อวินาที) จะส่งผลให้ Blynk Server ตัดการเชื่อมต่อของอุปกรณ์โดยอัตโนมัติ เพื่อป้องกันไม่ให้อุปกรณ์ดึงทรัพยากรของเซิร์ฟเวอร์จนเกินไป
*   **แนวทางแก้ไขที่ถูกต้อง:** ออกแบบระบบให้ส่งข้อมูลเฉพาะเมื่อค่าของเซนเซอร์เปลี่ยนไปอย่างมีนัยสำคัญ (Deadband filtering) หรือส่งตามเวลาที่กำหนดไว้ด้วย `BlynkTimer` (เช่น 2-5 วินาทีต่อครั้ง)

### 2.6 ทำไมการใช้ `delay()` ในโค้ด IoT จึงไม่ปลอดภัย (Why delay() is Unsafe)
*   **กลไกการเกิด Timeout:** อุปกรณ์ ESP32 เชื่อมต่อกับ Blynk Server ผ่านโปรโตคอล TCP ซึ่งต้องการให้อุปกรณ์เรียกเมทอด `Blynk.run()` บ่อยๆ เพื่อตอบกลับคำขอยืนยันตัวตน (Keepalive ping) จากฝั่งเซิร์ฟเวอร์
*   **ผลของการใช้ delay():** เมื่อใส่โค้ด `delay(5000);` ลงไป ซีพียูจะหยุดประมวลผลการทำงานอื่นๆ ทั้งหมด รวมถึงการประมวลผลโปรโตคอลเครือข่ายใน `Blynk.run()` ด้วย ทำให้ ESP32 ไม่สามารถตอบกลับ Ping จาก Blynk Server ได้ทันเวลา เมื่อพ้นช่วง Timeout (ปกติประมาณ 10-15 วินาที) เซิร์ฟเวอร์จะปิดช่องการเชื่อมต่อทันที อุปกรณ์จึงออฟไลน์ และเมื่อพ้นการหน่วงเวลา อุปกรณ์จะต้องมาทำกระบวนการทำ Handshake เพื่อเชื่อมต่อใหม่เสมอ ซึ่งทำให้สิ้นเปลืองแบนด์วิดท์ เกิดความหน่วงในการตอบสนอง และสูญเสียข้อมูล

## 3. กรณีศึกษาทางวิศวกรรมเครื่องกล (Mechanical Engineering Case Studies)

เพื่อให้สอดคล้องกับข้อกำหนดเรื่องการนำไปประยุกต์ใช้กับงานจริง จะเพิ่มเนื้อหากรณีศึกษาในเชิงลึก 3 รูปแบบ:

### กรณีศึกษาที่ 1: ระบบตรวจวัดอุณหภูมิและความปลอดภัยของหม้อไอน้ำในโรงงาน (Factory Steam Boiler Monitoring System)
*   **ความสำคัญเชิงวิศวกรรม:** หม้อไอน้ำ (Boiler) มีความเสี่ยงในการระเบิดหากมีความดันสูงเกิน หรืออุณหภูมิแห้งตัวของน้ำตกต่ำกว่าระดับที่ปลอดภัย
*   **การเชื่อมต่อเซนเซอร์และอุปกรณ์:**
    *   วัดอุณหภูมิห้องเผาไหม้ (สูงถึง 400°C) โดยใช้ **K-Type Thermocouple** เชื่อมต่อผ่านบอร์ดขับสัญญาณ **MAX6675** (ใช้โปรโตคอล SPI)
    *   วัดความดันไอน้ำในระบบด้วย **Pressure Transducer** (สัญญาณเอาต์พุตแบบอนาล็อก 0-5V แปลงค่าความดันเป็นช่วง 0-10 Bar)
    *   วัดระดับน้ำในหม้อต้มด้วยเซนเซอร์ระดับของเหลวหรือเซนเซอร์อัลตราโซนิกทนความร้อน
    *   ควบคุมวาล์วระบายไอน้ำส่วนเกิน (Relief Valve) ผ่านบอร์ดขับโซเลนอยด์ (Solenoid Valve Controller) หรือรีเลย์ไฟฟ้า
*   **ตรรกะความปลอดภัยที่ Edge และคลาวด์:** 
    *   ที่ Edge (ESP32): อ่านค่าความดันทุกๆ 100 ms หากความดันเกิน 8 Bar ให้สั่งงานรีเลย์เพื่อเปิดโซเลนอยด์วาล์วทันทีโดยไม่ต้องรอรับคำสั่งคลาวด์
    *   ที่ Cloud (Blynk): คอยรับข้อมูลอุณหภูมิและความดันมาแสดงผลบน Dashboard ในแบบเรียลไทม์ และมีสวิตช์ควบคุมแมนนวลให้หัวหน้าวิศวกรส่งคำสั่งเปิดวาล์วลดแรงดันได้จากระยะไกลในสถานการณ์ฉุกเฉิน

### กรณีศึกษาที่ 2: การตรวจสอบสุขภาพและการสั่นสะเทือนของแบริ่งมอเตอร์อุตสาหกรรม (Vibration-Based Predictive Maintenance)
*   **ความสำคัญเชิงวิศวกรรม:** แบริ่ง (Bearing) ของมอเตอร์ปั๊มน้ำขนาดใหญ่ในโรงงานมักเกิดการสึกหรอเมื่อใช้งานไปนานๆ การสั่นสะเทือนจะเพิ่มขึ้นตามความถี่เฉพาะตัวก่อนที่มอเตอร์จะพังเสียหาย
*   **การเชื่อมต่อเซนเซอร์:**
    *   ใช้เซนเซอร์วัดความเร่ง 3 แกนแบบดิจิทัล **ADXL345** (ใช้โปรโตคอล I2C) ติดตั้งแนบกับเสื้อเพลาของมอเตอร์
*   **ตรรกะประมวลผลและการคำนวณที่ Edge:**
    *   ESP32 ทำการสุ่มตัวอย่างความสั่นสะเทือนความเร็วสูง (เช่น สุ่มอ่านค่าความเร่ง $a_x, a_y, a_z$ ที่ความถี่ 100 Hz เป็นเวลา 1 วินาที)
    *   คำนวณหาค่าความสั่นสะเทือนรวมแบบ **Root Mean Square (RMS Vibration Acceleration)**:
        $$a_{RMS} = \sqrt{\frac{1}{N} \sum_{i=1}^{N} (a_{x,i}^2 + a_{y,i}^2 + a_{z,i}^2)}$$
    *   นำค่า $a_{RMS}$ ที่คำนวณได้ไปประเมินระดับความรุนแรงตามมาตรฐานอุตสาหกรรม **ISO 10816**
*   **การส่งข้อมูลคลาวด์:** ส่งค่า $a_{RMS}$ ไปเก็บไว้บนคลาวด์ทุกๆ 5 วินาที (ลดแบนด์วิดท์ลงจากการส่งค่าความถี่ดิบได้กว่า 99.9%) และเมื่อค่าสูงเกินเกณฑ์ขีดจำกัด (เช่น เกินระดับสีเหลือง/เตือนภัย) ให้ส่ง Notification ไปยังแผนกซ่อมบำรุงผ่านฟีเจอร์แจ้งเตือนของ Blynk

### กรณีศึกษาที่ 3: ระบบตรวจสอบพลังงานและปริมาณลมในระบบปรับอากาศส่วนกลาง (Solar-Powered HVAC Air Duct Monitoring)
*   **ความสำคัญเชิงวิศวกรรม:** ระบบจัดการอาคารสีเขียวที่ใช้พลังงานโซลาร์เซลล์ ต้องการตรวจสอบว่าพัดลมส่งลมเย็น (Blower Fan) ทำงานได้เต็มประสิทธิภาพหรือไม่ และประเมินฝุ่นอุดตันในฟิลเตอร์
*   **การเชื่อมต่อเซนเซอร์:**
    *   วัดผลต่างแรงดันก่อน-หลังผ่านตัวกรองอากาศด้วยเซนเซอร์วัดแรงดันต่างอากาศ (Differential Pressure Sensor)
    *   วัดความเร็วลมในท่อส่งลมโดยใช้เซนเซอร์ท่อพิโทต์แบบอิเล็กทรอนิกส์ (Pitot Tube Sensor)
    *   วัดกระแสไฟฟ้าและแรงดันของพัดลมโดยใช้เซนเซอร์กระแสไฟฟ้าแบบไม่สัมผัส **ACS712**
*   **แนวทางการใช้พลังงานอย่างมีประสิทธิภาพ:** เนื่องจากอุปกรณ์ทำงานด้วยระบบไฟพลังงานแสงอาทิตย์ร่วมกับแบตเตอรี่ จึงตั้งค่าให้ ESP32 ทำงานแบบ **Deep Sleep Mode** โดยบอร์ดจะตื่นขึ้นมาอ่านค่าเซนเซอร์ทุกๆ 10 นาที เชื่อมต่อส่งข้อมูลขึ้นคลาวด์ และเข้าสู่สภาวะหลับลึกเพื่อประหยัดแบตเตอรี่

## 4. โปรโตคอลการรับเข้าข้อมูล (Data Ingestion Protocols) และฐานข้อมูล (Database Systems)

อธิบายถึงกลไกการรับเข้าข้อมูลในฝั่งเซิร์ฟเวอร์เพื่อให้ผู้เรียนมีความรู้รอบด้านทางด้านสถาปัตยกรรมระบบ IoT:

### 4.1 เปรียบเทียบโปรโตคอลการส่งข้อมูลอุปกรณ์ IoT เข้าสู่คลาวด์
แสดงการวิเคราะห์เปรียบเทียบในแง่ของโครงสร้างแพ็กเก็ต (Packet Overhead), สถาปัตยกรรมเครือข่าย, และความเหมาะสมกับงานวิศวกรรม:

| โปรโตคอล | สถาปัตยกรรมหลัก | โครงสร้างการเชื่อมต่อ (Transport) | ขนาด Overhead และการใช้พลังงาน | คุณลักษณะเด่นและกลไกหลัก | กรณีการใช้งานที่เหมาะสม |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **MQTT** (Message Queuing Telemetry Transport) | **Publish/Subscribe** ผ่าน Broker ตัวกลาง | TCP (ต้องเปิด Connection ค้างไว้) | **ต่ำมาก** (ตัวหัวแพ็กเก็ตมีขนาดเพียง 2 Bytes เท่านั้น) | มีระดับการันตีข้อมูล (QoS 0, 1, 2) มีฟีเจอร์แจ้งสถานะสูญหาย (Last Will & Testament) และระบบ Keepalive | การรับส่งข้อมูลเรียลไทม์, แบนด์วิดท์อินเทอร์เน็ตจำกัด, อุปกรณ์ต้องคอยรับคำสั่งควบคุมย้อนกลับแบบทันที |
| **HTTP REST API** (Hypertext Transfer Protocol) | **Request/Response** (Client-Server) | TCP (เปิดแล้วปิดการเชื่อมต่อ หรือใช้ Keep-Alive) | **สูง** (มี Header ที่เป็นข้อความขนาดใหญ่อย่างน้อย 200B - 1KB ต่อครั้ง) | ใช้เมทอดมาตรฐาน (`GET`, `POST`, `PUT`) เข้าถึงได้ง่ายด้วยเว็บเบราว์เซอร์ปกติ | การส่งข้อมูลแบบรวมกลุ่มก้อนใหญ่ (Batch upload), การอัปเดตไฟล์เฟิร์มแวร์อุปกรณ์ (OTA), การดึงข้อมูลพยากรณ์อากาศมาใช้ในบอร์ด |
| **CoAP** (Constrained Application Protocol) | **Request/Response** แต่ถูกย่อให้เบาบางลง | **UDP** (ไม่มีการทำ Handshake แบบยาวเพื่อเชื่อมต่อ) | **ต่ำมาก** (มีขนาดหัวแพ็กเก็ตคงที่เพียง 4 Bytes) | มีโมเดลการทำงานแบบ RESTful (GET/POST) เหมือน HTTP แต่ถูกพอร์ตมาทำบน UDP มีการจัดส่งข้อความที่ระบุความน่าเชื่อถือได้ (Confirmable Messages) | อุปกรณ์ขนาดเล็กมากที่ใช้พลังงานจากแบตเตอรี่ก้อนเล็ก, ไม่ต้องการภาระการตรวจสอบความถูกต้องของ TCP |

### 4.2 ฐานข้อมูลอนุกรมเวลา (Time-Series Database) ปะทะ ฐานข้อมูลเชิงสัมพันธ์ (SQL Database)
เพิ่มการเปรียบเทียบเชิงสถาปัตยกรรมระดับล่างของวิธีการประมวลผลข้อมูลและดัชนี (Indexing) ระหว่างฐานข้อมูลทั้งสองประเภท:

*   **โครงสร้างและการทำงานของ SQL (เช่น PostgreSQL, MySQL):**
    *   **สถาปัตยกรรมการเก็บข้อมูล:** เก็บในรูปแบบแถวและคอลัมน์ (Row-oriented) การผูกโยงข้อมูลซับซ้อนด้วยดัชนีแบบ B-Tree
    *   **ปัญหาขวดคอเมื่อนำมาเก็บข้อมูลเซนเซอร์:** ในการเขียนแถวข้อมูลใหม่แต่ละครั้ง ระบบต้องการบันทึกดัชนีและรักษาความเป็นระเบียบ (ACID Properties) ซึ่งใช้เวลานานขึ้นเรื่อยๆ ตามขนาดของฐานข้อมูลที่ขยายตัวขึ้น หากมีอุปกรณ์ 1,000 ตัวเขียนข้อมูลพร้อมกัน ดัชนีจะเกิดความบวมและทำงานล่าช้ามาก
*   **โครงสร้างและการทำงานของ TSDB (เช่น InfluxDB, TimescaleDB, Blynk Cloud Backend):**
    *   **สถาปัตยกรรมการเก็บข้อมูล:** จัดเก็บแบบเรียงตามแนวคอลัมน์ (Column-oriented) และใช้โครงสร้างข้อมูลแบบ Log-Structured Merge (LSM) หรือ Time-Structured Merge (TSM) ซึ่งมองข้ามความสัมพันธ์ที่ซับซ้อน มุ่งเน้นการบันทึกข้อมูลเรียงต่อท้ายต่อคิวเวลาไปเรื่อยๆ (Append-only) 
    *   **คีย์หลักดัชนี:** ใช้ **Timestamp** เป็นดัชนีตัวหลัก ทำให้ความเร็วในการบันทึกข้อมูลคงที่เสมอ ไม่ขึ้นกับปริมาณข้อมูลในเครื่อง
    *   **ฟังก์ชันการรวมข้อมูลและการจัดการข้อมูลเก่า:** มีระบบ **Data Downsampling** อัตโนมัติ (เช่น นำข้อมูลอุณหภูมิที่ละเอียดในระดับวินาทีของเดือนก่อน มารวมเฉลี่ยเป็นข้อมูลรายชั่วโมงเพื่อลดขนาดพื้นที่เก็บลง) และตั้งค่ารอบอายุการทำลายข้อมูล (**Retention Policies**) ได้ง่าย

## 5. การขยายความปลอดภัยในระบบ IoT (IoT Security Extensions)

หัวข้อความปลอดภัยในระบบ IoT จะขยายเนื้อหาให้ลึกซึ้งและครอบคลุมภัยคุกคามในสัญญะปัจจุบัน:

### 5.1 การจัดการกุญแจยืนยันตัวตน (Authentication Token Security)
*   **ทฤษฎีการยืนยันตัวตน:** ตัว Auth Token เป็นคีย์ถอดรหัสแบบสมมาตรที่ใช้ระบุว่าใครเป็นผู้ส่งข้อมูล หากผู้ไม่หวังดีขโมยข้อมูลนี้ไปได้ จะสามารถปลอมแปลงสถานะการส่งข้อมูล หรือสั่งปิด-เปิดการทำงานทางวิศวกรรมจริงได้ทันที
*   **ความเสี่ยงระดับนักพัฒนาและแนวทางแก้ไข:** การใส่ Token ลงในบรรทัด `#define BLYNK_AUTH_TOKEN "..."` ตรงๆ ในโค้ด แล้วอัปโหลดขึ้นบริการแชร์โค้ดสาธารณะ เช่น GitHub จะนำมาซึ่งภัยพิบัติทางข้อมูล วิศวกรจึงต้องเรียนรู้วิธีการเก็บกุญแจในไฟล์การตั้งค่าที่แยกต่างหาก (.env หรือ config) หรือการใช้เทคนิค **Blynk Edgent Dynamic Provisioning** ที่ยอมให้อุปกรณ์เข้าสู่โหมดเชื่อมต่อชั่วคราวแล้วขอรับค่าคีย์และรหัส Wi-Fi มาบันทึกในหน่วยความจำถาวร (EEPROM/NVS) แทนการระบุตายตัวในหน้าเขียนโปรแกรม

### 5.2 ความปลอดภัยของชั้นการขนส่งข้อมูล (Transport Layer Security: HTTPS & MQTTS)
*   **ภัยคุกคาม Man-in-the-Middle (MitM):** อุปกรณ์ส่งข้อมูลอุณหภูมิและความดันหม้อต้มผ่านสัญญาณวิทยุ Wi-Fi ผู้บุกรุกสามารถใช้โปรแกรมดักฟังแพ็กเก็ต (Packet Sniffer) เพื่ออ่านค่าทั้งหมดได้ หากใช้โปรโตคอล HTTP/MQTT แบบไม่มีการเข้ารหัสข้อมูล (Plaintext)
*   **การเข้ารหัสด้วย TLS/SSL:** ข้อมูลก่อนส่งจะถูกเข้ารหัสผ่านกระบวนการ Handshake โดยใช้กุญแจสาธารณะ (Public Key Infrastructure: PKI) ทำให้ข้อมูลกลายเป็นข้อความที่ไม่มีใครอ่านออกยกเว้นตัวเซิร์ฟเวอร์
*   **ข้อจำกัดทางวิศวกรรมฮาร์ดแวร์บน ESP32:** การเข้ารหัส TLS มีข้อเสียคือการคำนวณที่ใช้กำลังไฟฟ้าและพลังประมวลผลสูง (CPU overhead) เนื่องจาก ESP32 จะต้องประมวลผลอัลกอริทึมคณิตศาสตร์ที่ซับซ้อนในขั้นตอนแลกเปลี่ยนกุญแจ (Key Exchange) ซึ่งอาจใช้เวลาทำ Handshake นาน 1-2 วินาที และสูญเสียทรัพยากรหน่วยความจำแรม (RAM footprint) ไปกับตัวจัดเก็บ Buffer การเข้ารหัสมากถึง 20-30 KB การเขียนโปรแกรมจึงต้องบริหารทรัพยากรส่วนนี้ให้ดีเพื่อไม่ให้บอร์ดเกิดปัญหา Memory Overflow

### 5.3 ความปลอดภัยของการอัปเดตระบบผ่านอากาศ (Over-The-Air Update: OTA Security)
*   **สถาปัตยกรรม OTA:** การอัปเดตเฟิร์มแวร์ผ่าน Wi-Fi โดยไม่ต้องแกะกล่องอุปกรณ์อุตสาหกรรมมาต่อสาย USB
*   **ความเสี่ยงและความปลอดภัย:** หากสวิตช์สำหรับการอัปเดต OTA เปิดรับไฟล์แบบไม่มีระบบป้องกัน ผู้บุกรุกบนวงแลนเดียวกันจะสามารถส่งไฟล์เฟิร์มแวร์ที่เป็นภัยคุกคาม (Malicious Firmware) ไปติดตั้งทับในตัวบอร์ดเพื่อยึดอำนาจควบคุมได้
*   **แนวทางการป้องกัน:**
    1.  **HTTPS OTA:** อัปเดตผ่านการดาวน์โหลดไฟล์เฟิร์มแวร์จากเซิร์ฟเวอร์ HTTPS ที่มีใบรับรองความถูกต้อง
    2.  **Code Signing (การลงนามรหัส):** ทำการลงลายมือชื่อดิจิทัลให้กับไฟล์คอมไพล์ (.bin) ด้วยคีย์ส่วนตัวของผู้พัฒนา (Private Key) ก่อนส่งออก โดยบอร์ด ESP32 จะมีคีย์สาธารณะ (Public Key) ติดตั้งไว้อยู่ก่อนหน้า ซึ่งตัวบอร์ดจะทำการตรวจสอบความถูกต้องของลายเซ็นก่อนจะทำการเขียนทับระบบ หากข้อมูลไม่ผ่านการยืนยัน ระบบบูตโหลดเดอร์จะทิ้งไฟล์นั้นทันที

## 6. โครงสร้างแบบฝึกหัดและงานปฏิบัติการจำลอง Wokwi ใหม่ (New Exercises & Wokwi Practical Labs)

เพื่อให้นักศึกษาได้ลงมือปฏิบัติและพิสูจน์ทฤษฎีตามบทเรียนด้วยตนเอง จะเสนอการปรับปรุงแบบฝึกหัดท้ายบทดังนี้:

### 6.1 คำถามทบทวนเชิงคิดวิเคราะห์และแก้ปัญหา (Conceptual & Debugging Exercises)
*   **ข้อที่ 1 (วิเคราะห์หาบั๊กการบล็อกโค้ด):**
    *   *โจทย์:* ให้นักศึกษาวิเคราะห์หาข้อผิดพลาดในโค้ด C++ ต่อไปนี้ที่เชื่อมต่อกับ Blynk:
        ```cpp
        void loop() {
          Blynk.run();
          float val = analogRead(34);
          Blynk.virtualWrite(V1, val);
          delay(5000); // หน่วงเวลา 5 วินาทีเพื่อให้อุปกรณ์ไม่ทำงานหนักเกินไป
        }
        ```
        จงตอบคำถามว่า: (1) โค้ดนี้จะส่งผลอย่างไรต่อสถานะออนไลน์ของอุปกรณ์บน Blynk Console และทำไมจึงเกิดเหตุการณ์นั้นขึ้น (2) จงเขียนโค้ดนี้ใหม่ให้ถูกต้องตามหลักปฏิบัติโดยใช้ `BlynkTimer`
*   **ข้อที่ 2 (การออกแบบโครงสร้าง MQTT Topic):**
    *   *โจทย์:* โรงงานประกอบชิ้นส่วนยานยนต์แบ่งออกเป็น 2 อาคารผลิต (Zone A, Zone B) โดยในแต่ละโซนจะมีสายพานลำเลียง (Conveyor) โซนละ 3 เส้น แต่ละเส้นมีเซนเซอร์วัดรอบความเร็วพัดลม (RPM) และอุณหภูมิมอเตอร์ขับเคลื่อน (Temp) จงออกแบบโครงสร้างหัวข้อ MQTT Topic (Topic Hierarchy) ที่มีประสิทธิภาพ ยืดหยุ่น และง่ายต่อการเขียนโปรแกรมรับ/ส่งข้อมูล
*   **ข้อที่ 3 (เปรียบเทียบการสืบค้นข้อมูล SQL vs TSDB):**
    *   *โจทย์:* หากเราจัดเก็บข้อมูลเซนเซอร์วัดค่าความดันหม้อน้ำอุตสาหกรรมในระบบที่มีจุดวัด 100 จุด สุ่มอ่านค่าทุกๆ 10 วินาที เป็นระยะเวลา 1 ปี จงอธิบายในเชิงทฤษฎีระบบฐานข้อมูลว่า คิวรีดึงข้อมูลประเภทใด (เช่น การหาอุณหภูมิเฉลี่ยย้อนหลัง 24 ชั่วโมงของมอเตอร์ลำดับที่ 5) ที่ทำให้ฐานข้อมูล TSDB ทำงานได้เร็วกว่า SQL ดั้งเดิมอย่างมาก เพราะเหตุใด

### 6.2 งานปฏิบัติการจำลองการสื่อสารและระบบความปลอดภัยบน Wokwi (Boiler Safety Simulation Lab)
*   **ชื่อภารกิจ:** ระบบวัดอุณหภูมิระดับน้ำและความดันหม้อต้มอัจฉริยะ ป้องกันภัยแบบ Real-time ร่วมกับ Blynk
*   **เครื่องมือจำลองใน Wokwi:**
    1.  บอร์ด **ESP32**
    2.  ตัวต้านทานปรับค่าได้ (Slide Potentiometer) แทนการจำลองเซนเซอร์ความดันในระบบ (แปลงระดับแรงดันอนาล็อก 0-3.3V ให้แสดงสเกลเป็นความดันไอน้ำ 0.0 ถึง 15.0 Bar)
    3.  เซนเซอร์ **DHT22** แทนตัววัดอุณหภูมิไอน้ำ
    4.  โมดูล **LED (สีแดง)** ต่อพ่วงผ่าน GPIO เพื่อเลียนแบบสัญญาณเตือนและวาล์วระบายแรงดันฉุกเฉิน (Relay Shield controlling Relief Valve)
*   **ข้อกำหนดการเขียนโปรแกรมและการเชื่อมต่อ:**
    *   เชื่อมต่ออินเทอร์เน็ตของบอร์ดจำลองโดยใช้โครงข่าย `"Wokwi-GUEST"` และพาสเวิร์ดว่าง `""` 
    *   ใช้โครงสร้างแบบ **Non-blocking connection** (`Blynk.config()` และ `Blynk.connect()`) เพื่อป้องกันไม่ให้โค้ดระบบความปลอดภัยหยุดการตรวจสอบหากเครือข่ายอินเทอร์เน็ตมีปัญหา
    *   ใช้ **BlynkTimer** ในการตรวจวัดและอัปเดตข้อมูลขึ้นคลาวด์ทุกๆ 2 วินาที โดยห้ามใช้คำสั่ง `delay()` ในโค้ดเด็ดขาด
    *   ส่งค่าอุณหภูมิขึ้นไปยัง **V1** ส่งค่าความดันขึ้นไปยัง **V2**
    *   **ตรรกะประมวลผลที่ขอบ (Edge Safety Checking Loop):** ในฟังก์ชันความปลอดภัยที่ตรวจจับเป็นอิสระจากคลาวด์ ให้อุปกรณ์อ่านค่าความดัน หากความดันไอน้ำที่สุ่มอ่านได้มีค่า **เกินกว่า 10.0 Bar** ให้ ESP32 สั่งเปิดไฟ LED สีแดง เพื่อส่งสัญญาณระบายวาล์วนิรภัยทันที พร้อมกับส่งการแจ้งเตือนไปยังแอปพลิเคชันมือถือ
    *   **ตรรกะควบคุมย้อนกลับจากคลาวด์ (Cloud Control Loop):** สร้างแดชบอร์ดที่มี Switch Widget ผูกกับพิน **V4** (ตั้งชื่อสวิตช์ว่า "Manual Emergency Release") หากผู้ใช้กดสวิตช์เป็น 1 ให้ ESP32 สั่งเปิด LED ระบายไอน้ำได้ แม้ว่าระดับความดันในตอนนั้นจะยังไม่ถึงเกณฑ์ปลอดภัยก็ตาม เพื่อช่วยอำนวยความสะดวกในการซ่อมบำรุง
*   **ผลงานการส่งงานของนักศึกษา:**
    *   แชร์ลิงก์จำลองผลจากระบบ Wokwi ที่ทำงานได้สมบูรณ์แบบ
    *   ภาพหน้าจอ (Screenshot) ของหน้าจอแดชบอร์ด Blynk (Web Dashboard หรือ App) ที่แสดงผลค่าเป็นรูป Gauge ความดัน รูปกราฟเส้นแสดงแนวโน้มย้อนหลัง และสถานะสวิตช์ควบคุมระบายความดันตามช่วงการเปลี่ยนแปลงของข้อมูลจริง
