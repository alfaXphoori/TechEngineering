## 1. รายละเอียดการขยายเนื้อหา (Proposed Expansion Details)

### หัวข้อที่ 1: เปรียบเทียบสถาปัตยกรรม Microcontroller vs Microprocessor และการทำงานภายใน
ขยายจากตารางเปรียบเทียบในระดับแอปพลิเคชันเดิม ไปสู่ทฤษฎีโครงสร้างฮาร์ดแวร์ระดับล่าง (Hardware Architecture):
* **สถาปัตยกรรมแบบ Von Neumann vs Harvard:**
  * อธิบายโครงสร้างของบัสข้อมูล (Data Bus) และบัสคำสั่ง (Instruction Bus)
  * เปรียบเทียบข้อดี-ข้อเสียในแง่ของความเร็วในการประมวลผลและความยุ่งยากในการออกแบบทางฮาร์ดแวร์
  * แนะนำแนวคิด **Modified Harvard Architecture** ซึ่งเป็นโครงสร้างที่ใช้งานจริงในชิปสมัยใหม่ (เช่น Xtensa, ARM) เพื่อให้อ่านข้อมูลคงที่ (Constant data) จากหน่วยความจำโปรแกรมได้สะดวก
* **ส่วนประกอบภายในของ CPU (Internal CPU Components):**
  * **ALU (Arithmetic Logic Unit):** หน้าที่การคำนวณทางคณิตศาสตร์และตรรกศาสตร์
  * **Registers (รีจิสเตอร์ที่สำคัญ):**
    * *Program Counter (PC):* ชี้ตำแหน่งคำสั่งถัดไปที่จะถูกประมวลผล
    * *Stack Pointer (SP):* จัดการตำแหน่งหน่วยความจำชั่วคราว (Stack) สำหรับการเรียกฟังก์ชัน (Function Call) และ Interrupt
    * *Instruction Register (IR):* พักรหัสคำสั่ง (Opcode) ที่เพิ่ง Fetch มาเพื่อรอการถอดรหัส
    * *Status/Flag Register:* บันทึกสถานะผลลัพธ์ของ ALU (Carry, Zero, Negative, Overflow)
    * *General Purpose Registers:* ใช้สำหรับพักข้อมูลชั่วคราวในการคำนวณ
* **ระบบสัญญาณนาฬิกา (Clock Systems):**
  * แหล่งกำเนิดสัญญาณนาฬิกา (Crystal Oscillator vs Internal RC Oscillator)
  * การใช้ Phase-Locked Loop (PLL) ในการทวีคูณความถี่ (Frequency Multiplication)
  * ตัวหารความถี่ (Prescalers) เพื่อจ่ายสัญญาณให้แก่โมดูลอุปกรณ์ต่อพ่วง (Peripherals)
* **วงรอบการประมวลผลคำสั่ง (Instruction Execution Cycle):**
  * แสดงลำดับขั้นตอน **Fetch-Decode-Execute-Writeback (FDEW)**
  * อธิบายแนวคิดการทำ Pipeline ของคำสั่ง เพื่อเร่งความเร็วในการประมวลผล

### หัวข้อที่ 2: เจาะลึกฮาร์ดแวร์ ESP32 (ESP32 Hardware Details)
ยกระดับเนื้อหาของ ESP32 ให้เป็นระบบวิศวกรรมฝังตัวแบบมืออาชีพ:
* **สถาปัตยกรรม Dual-core Xtensa LX6:**
  * โครงสร้างแบบ 32-bit ของแต่ละ Core (Core 0: PRO_CPU สำหรับโปรโตคอลการสื่อสาร และ Core 1: APP_CPU สำหรับแอปพลิเคชันหลัก)
  * อธิบายกลไกการทำ Symmetric Multiprocessing (SMP) ภายใต้ระบบปฏิบัติการ FreeRTOS และการใช้คำสั่ง `xTaskCreatePinnedToCore()` ในระบบ Arduino เพื่อแยกงานการประมวลผล
* **แผนผังหน่วยความจำ (Memory Map):**
  * โครงสร้างของ SRAM ขนาด 520 KB (แบ่งเป็น SRAM0 สำหรับ IRAM, SRAM1 และ SRAM2 สำหรับ DRAM) และการใช้ DMA (Direct Memory Access)
  * การเชื่อมต่อหน่วยความจำภายนอก (External Flash และ PSRAM) ผ่านอินเทอร์เฟซ SPI โดยใช้ MMU (Memory Management Unit) และระบบ Cache (32 KB ต่อ core) ในการ Map เข้ากับแอดเดรสของ CPU
* **รหัสพินเริ่มต้นระบบ (Boot Strapping Pins):**
  * เจาะลึกขากำหนดโหมดบูต ได้แก่ **GPIO 0, 2, 5, 12, 15**
  * แสดงตารางค่าสถานะลอจิก (HIGH/LOW) ของพินแต่ละตัวในการเข้าสู่โหมดต่าง ๆ เช่น UART Bootloader (สำหรับแฟลชโปรแกรม) และ SPI Boot (รันโปรแกรมปกติ)
  * **คำเตือนทางวิศวกรรม:** ผลกระทบหากเชื่อมต่ออุปกรณ์ภายนอกที่มีวงจรดึงกระแส (Pull-up/Pull-down) เข้ากับพินเหล่านี้ ซึ่งอาจทำให้ชิปบูตไม่ขึ้นหรือไม่สามารถอัปโหลดโค้ดได้
* **การควบคุม GPIO ในระดับรีจิสเตอร์ (GPIO Register Control):**
  * อธิบายกระบวนการเบื้องหลังของฟังก์ชัน `digitalWrite()` และ `digitalRead()` ที่แลกความง่ายด้วยจำนวนรอบสัญญาณนาฬิกาที่สูญเสียไป (Overhead)
  * แนะนำรีจิสเตอร์ควบคุมโดยตรง: `GPIO_OUT_REG` สำหรับเขียนค่า, `GPIO_OUT_W1TS_REG` (Write 1 to Set) และ `GPIO_OUT_W1TC_REG` (Write 1 to Clear) สำหรับแก้ไขสถานะพินแบบอะตอมมิก (Atomic Operation - ป้องกัน Race Condition) และ `GPIO_IN_REG` สำหรับการอ่านค่า
  * แสดงโค้ดเปรียบเทียบระหว่างการเขียนผ่าน Arduino API กับการเข้าถึง Register โดยตรง เพื่อให้เห็นประสิทธิภาพที่ต่างกันอย่างมีนัยสำคัญ

### หัวข้อที่ 3: การออกแบบเฟิร์มแวร์แบบไม่บล็อกการทำงานในภาษา C++ (Non-blocking Firmware Design)
เปลี่ยนกระบวนทัศน์การเขียนโปรแกรม (Programming Paradigm) จากแบบลำดับขั้น (Procedural) ไปสู่สถาปัตยกรรมแบบตอบสนองเหตุการณ์ (Event-driven):
* **ปัญหาของฟังก์ชัน `delay()` ในงานวิศวกรรมจริง:**
  * อธิบายสภาวะ CPU Spinning (การวนลูปเปล่า) ที่ทำให้บล็อกการทำงานอื่น
  * วิเคราะห์ผลกระทบ: การขัดจังหวะการอ่านค่าเซ็นเซอร์ฉุกเฉิน (Emergency Stop), ความล้าหลังของระบบควบคุมแบบป้อนกลับ (PID latency), และการหลุดการเชื่อมต่อของเครือข่าย Wi-Fi/Bluetooth อันเนื่องมาจากไม่ได้เรียกฟังก์ชันรักษาสัญญาณเชื่อมต่อ (Keep-alive) ในส่วนหลังของระบบ
* **การทำงานมัลติทาสกิงแบบร่วมมือด้วย `millis()` (Cooperative Multitasking):**
  * หลักการทำงานของการตรวจสอบเวลาปัจจุบันเทียบกับเวลาอ้างอิงล่าสุด
  * **ประเด็นทางคณิตศาสตร์:** พิสูจน์ว่าทำไมสูตร `currentMillis - previousMillis >= interval` จึงทำงานได้อย่างถูกต้องแม้จะเกิดปัญหาหน่วยความจำตัวนับล้น (32-bit Rollover / Overflow ที่เวลาประมาณ 49.7 วัน) ด้วยคุณสมบัติของเลขคณิตส่วนเติมเต็มสอง (Two's Complement Arithmetic)
* **การสร้างเครื่องจำลองสถานะจำกัด (Finite State Machine - FSM):**
  * การนิยามสถานะระบบด้วย `enum class` เพื่อเพิ่มความปลอดภัยของประเภทข้อมูล (Type Safety)
  * โครงสร้างของสวิตช์และกรณีเลือก (`switch-case`) ในฟังก์ชัน `loop()` เพื่อแยกการทำงานของแต่ละสถานะออกจากกันอย่างเป็นระบบ
  * แนวทางการเขียนเฟิร์มแวร์ให้เปลี่ยนสถานะด้วยเงื่อนไขตามเวลา (Time-based transition) โดยไม่บล็อกการทำงานอื่น

### หัวข้อที่ 4: กรณีศึกษาเชิงวิศวกรรมอุตสาหกรรม (Industrial Engineering Case Study)
เพิ่มตัวอย่างการใช้งานจริงระดับอุตสาหกรรมในส่วนของโค้ดโปรแกรม:
* **ระบบควบคุมปั๊มระบายความร้อนของหม้อต้มอุตสาหกรรม (Boiler Cooling Pump Controller with Safety Interlocks):**
  * **โจทย์:** ควบคุมปั๊มระบายความร้อนโดยรับค่าอุณหภูมิจาก NTC Thermistor (Analog Input) และมีระบบป้องกันความเสียหาย (Failsafe)
  * **สถานะของระบบ (States):**
    1. `IDLE` (ระบบสแตนด์บาย, ปั๊มดับ): คอยวัดอุณหภูมิ หากอุณหภูมิ > 80°C จะเปลี่ยนสถานะไปยัง `PUMPING`
    2. `PUMPING` (ปั๊มทำงาน): สั่งเปิดปั๊ม (Digital Output) และตรวจสอบอุณหภูมิ หากอุณหภูมิลดลงต่ำกว่า 60°C จะเปลี่ยนไป `COOLDOWN_DELAY` แต่หากอุณหภูมิสูงเกิน 95°C หรือเซ็นเซอร์ทำงานผิดพลาด (Open/Short circuit) จะเปลี่ยนไป `FAULT`
    3. `COOLDOWN_DELAY` (ชะลอการดับปั๊ม): ให้ปั๊มทำงานต่อไปอีก 10 วินาทีโดยใช้ `millis()` เพื่อระบายความร้อนสะสมที่ตัวปั๊มและหม้อต้ม ป้องกันระบบเกิดการเปิด-ปิดบ่อยเกินไป (Hysteresis & Post-run cooling) เมื่อครบเวลาจะกลับไปยัง `IDLE`
    4. `FAULT` (ระบบขัดข้อง): สั่งเปิดปั๊มค้างไว้เพื่อความปลอดภัย ส่งสัญญาณไซเรนเตือน (PWM Buzzer) และรอการกดปุ่ม Reset ด้วยมือ (Manual Push Button) เมื่ออุณหภูมิลดลงต่ำกว่าเกณฑ์ปลอดภัย (< 70°C) จึงจะยอมให้กลับไปสแตนด์บาย
  * **ผลลัพธ์ที่ต้องแสดงในเนื้อหา:**
    * แผนภาพการเปลี่ยนสถานะ (State Transition Diagram) ในรูปแบบ ASCII หรือ Mermaid
    * โค้ดโปรแกรมภาษา C++ ที่ทำงานในโหมด Non-blocking ทั้งหมด มีการกรองสัญญาณรบกวนปุ่มกด (Software Debounce) และใช้ FSM ร่วมกับโครงสร้างของภาษาที่ดี

### หัวข้อที่ 5: การปรับปรุงและเพิ่มเติมแบบฝึกหัด (Expanded Exercises)
ปรับปรุงคำถามท้ายบทเรียนให้มีความท้าทายในเชิงคำนวณเชิงตัวเลขและการออกแบบระบบ:
* **ข้อที่ 1 (ทฤษฎีสถาปัตยกรรม):** ให้คำนวณประสิทธิภาพแบนด์วิดท์ของระบบบัสหน่วยความจำของ Von Neumann เปรียบเทียบกับ Harvard ในการรันชุดคำสั่งจำลอง
* **ข้อที่ 2 (การออกแบบฮาร์ดแวร์):** โจทย์ออกแบบวงจรอิเล็กทรอนิกส์เพื่อป้องกันขา Boot Strapping (เช่น GPIO 12 หรือ GPIO 0) โดยระบุค่าของตัวต้านทาน Pull-up/Pull-down และการใช้ Buffer เพื่อแยกสัญญาณไม่ให้ขัดขวางการเข้าสู่โหมดบูตของ ESP32 ขณะทำงานจริง
* **ข้อที่ 3 (การเขียนโค้ดระดับรีจิสเตอร์):** ให้เขียนโค้ดภาษา C++ สำหรับ ESP32 เพื่อทำการสั่งงานพินเอาต์พุต (GPIO 18, 19, 21) ให้ทำงาน HIGH หรือ LOW พร้อมกันในระดับไมโครวินาที โดยใช้การควบคุมผ่าน Register (Bit Manipulation)
* **ข้อที่ 4 (การคำนวณแปลงค่าทางวิศวกรรม):** กำหนดให้เซ็นเซอร์อุณหภูมิมีแรงดันเอาต์พุต $V_{out} = 10 \text{ mV/}^\circ\text{C} + 500\text{ mV}$ ต่อเข้าระบบ ADC ของ ESP32 ขนาด 12-bit (ตั้งค่า Attenuation ที่ 11dB ช่วงแรงดัน 0-3.1V):
  1. จงหาความสัมพันธ์ทางคณิตศาสตร์เพื่อแปลงค่าดิบจาก ADC (0-4095) ให้เป็นค่าอุณหภูมิในหน่วยองศาเซลเซียส ($^\circ\text{C}$)
  2. หากอ่านค่า ADC ได้เป็น 2048 อุณหภูมิที่วัดได้จะมีค่ากี่องศาเซลเซียส?
* **ข้อที่ 5 (การออกแบบระบบอัตโนมัติ):** ให้ออกแบบ FSM สำหรับระบบหม้อนึ่งความดันฆ่าเชื้ออุตสาหกรรม (Autoclave) ประกอบด้วยขั้นตอน: เปิดประตู (`DOOR_OPEN`), ทำความร้อน (`HEATING`), ฆ่าเชื้ออุณหภูมิสูง (`STERILIZING` - รักษาอุณหภูมิ 121°C นาน 20 นาที), ระบายแรงดัน (`EXHAUST`), และทำงานเสร็จสิ้น (`COMPLETE`) จงเขียนตารางการเปลี่ยนสถานะ (State Transition Table) และโครงร่างโค้ด (Skeleton Code) ภาษา C++ โดยใช้ FSM และ `millis()`

## 3. วิธีการตรวจสอบความถูกต้องของงาน (Verification Protocol)

เพื่อให้มั่นใจว่าข้อมูลที่ปรับปรุงมีความถูกต้องตามหลักการวิศวกรรมคอมพิวเตอร์และระบบควบคุมฝังตัว:
1. **การตรวจสอบไวยากรณ์โค้ด C++ (Syntax Compilation Test):**
   * โค้ดในกรณีศึกษาระบบหม้อต้ม และตัวอย่างการใช้ GPIO Register จะต้องทดสอบนำไปคอมไพล์ผ่านเครื่องมือ Arduino IDE (ใช้บอร์ด ESP32 Dev Module) หรือคอมไพล์บน Wokwi Simulator เพื่อให้มั่นใจว่าโค้ดไม่มีข้อผิดพลาดทางไวยากรณ์ (Compilation error)
2. **การตรวจสอบผลการคำนวณทางคณิตศาสตร์ (Calculation Verification):**
   * ตรวจสอบสมการแปลงค่า ADC และผลเฉลยของอุณหภูมิที่ 2048 รอบว่าคำนวณได้ถูกต้องตามสูตรการลดทอนของแรงดันของ ESP32 จริง
3. **การทดสอบการบูตและการทำงานของฮาร์ดแวร์ (Bootstrapping Pin Behavior Verification):**
   * ตรวจสอบความถูกต้องของตารางโหมดการเข้าสู่ Bootloader ของขา GPIO 0, 2, 5, 12, 15 เทียบกับคู่มือ Hardware Design Guidelines อย่างเป็นทางการของ Espressif
