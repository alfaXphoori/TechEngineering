## วัตถุประสงค์และขอบเขต

โครงงาน IoT เป็นการบูรณาการความรู้ตลอดทั้งรายวิชา ให้ผู้เรียนสร้างระบบ IoT ครบวงจร (End-to-End) ตั้งแต่ระดับอุปกรณ์ปลายทางจนถึงการแสดงผลและควบคุมจากระยะไกล โดยจำลองสถานการณ์จริงในงานวิศวกรรมเครื่องกลหรืออุตสาหกรรม

### สิ่งที่โครงงานต้องทำให้ครบ (End-to-End)

```
เซนเซอร์/Input  →  ESP32 (Logic)  →  Wi-Fi + MQTT  →  ThingsBoard Cloud  →  Dashboard/Control
```

- **ชั้นอุปกรณ์ (Device):** ESP32 จำลองบน Wokwi อ่านเซนเซอร์และควบคุมตัวกระทำ
- **ชั้นเครือข่าย (Network):** ส่งข้อมูลผ่าน Wi-Fi โดยใช้โพรโทคอล MQTT
- **ชั้นคลาวด์ (Cloud):** จัดเก็บและประมวลผลข้อมูลบน ThingsBoard Cloud
- **ชั้นแอปพลิเคชัน (Application):** แดชบอร์ดแสดงผลแบบ Real-time และควบคุมอุปกรณ์ผ่าน RPC

## ข้อกำหนดขั้นต่ำ (Minimum Requirements)

โครงงานทุกกลุ่ม **ต้องผ่านข้อกำหนดต่อไปนี้ทั้งหมด** เพื่อให้ได้คะแนน:

| # | ข้อกำหนด | รายละเอียด |
|:---:|---|---|
| 1 | **เซนเซอร์อย่างน้อย 1 ตัว** | อ่านค่าจริง/จำลองบน Wokwi ได้ (เช่น DHT22, LDR, Potentiometer, HC-SR04) |
| 2 | **ตัวกระทำอย่างน้อย 1 ตัว** | ควบคุมโดยตรรกะ (เช่น เปิดพัดลมเมื่ออุณหภูมิสูง) หรือควบคุมผ่าน RPC จาก Dashboard |
| 3 | **ส่งข้อมูล MQTT ไปยัง ThingsBoard** | ใช้ topic `v1/devices/me/telemetry` ส่งข้อมูล JSON อย่างน้อย 2 ค่า |
| 4 | **Dashboard บน ThingsBoard** | มี Widget อย่างน้อย 2 ชนิด (เช่น กราฟ + gauge หรือ กราฟ + status) |
| 5 | **ระบบแจ้งเตือน/Threshold** | ตั้ง Alarm หรือตรรกะแจ้งเตือนเมื่อค่าผิดเกณฑ์อย่างน้อย 1 กรณี |
| 6 | **รายงานโครงงาน** | รายงานสั้น 5–8 หน้า (รูปแบบตามที่กำหนด ดูหัวข้อการส่งงาน) |
| 7 | **การนำเสนอ** | Demo สดการทำงานของระบบ 10 นาที + ตอบคำถาม 5 นาที |

## ตัวอย่างหัวข้อโครงงาน

กลุ่มสามารถเลือกหัวข้อต่อไปนี้ หรือเสนอหัวข้อใหม่ที่สอดคล้องกับงานวิศวกรรมเครื่องกล (ต้องได้รับอนุมัติจากอาจารย์ก่อนสัปดาห์ที่ 12)

### 1. ระบบตรวจสอบการสั่นสะเทือนของเครื่องจักร (Predictive Maintenance via Vibration)

**บริบท:** โรงงานผลิตชิ้นส่วนยานยนต์ต้องการตรวจสอบสุขภาพมอเตอร์ผ่านแรงสั่นสะเทือน  
**เซนเซอร์:** Accelerometer (ADXL345) + เซนเซอร์อุณหภูมิ  
**ตัวกระทำ:** LED สีต่างๆ บ่งบอกระดับความเสี่ยง + Buzzer แจ้งเตือน  
**Dashboard:** กราฟ vibration, gauge อุณหภูมิ, status "ปกติ/เฝ้าระวัง/ผิดปกติ"

---

### 2. ระบบตรวจสอบหม้อต้มและปั๊มน้ำ (Smart Boiler & Pump Monitor)

**บริบท:** โรงงานอุตสาหกรรมต้องการควบคุมหม้อต้มไอน้ำให้ทำงานในช่วงอุณหภูมิและความดันที่ปลอดภัย  
**เซนเซอร์:** DHT22 (อุณหภูมิ/ความชื้น) + LDR จำลองเป็นเซนเซอร์ความดัน  
**ตัวกระทำ:** รีเลย์ควบคุมปั๊มน้ำ + LED แสดงสถานะ  
**Dashboard:** กราฟเส้นอุณหภูมิ/ความดัน, เกจ, ปุ่ม RPC เปิด/ปิดปั๊มจากระยะไกล, Alarm เมื่ออุณหภูมิ > 95°C

---

### 3. ระบบควบคุมสภาพแวดล้อมในโรงเก็บชิ้นส่วน (Environmental Control / Smart HVAC)

**บริบท:** โรงเก็บชิ้นส่วนอิเล็กทรอนิกส์ต้องรักษาอุณหภูมิ 20–25°C ความชื้น < 60%  
**เซนเซอร์:** DHT22 (อุณหภูมิ + ความชื้น)  
**ตัวกระทำ:** เซอร์โวหมุนเปิด/ปิดช่องระบายอากาศ + พัดลม DC (PWM)  
**Dashboard:** กราฟเส้น 2 เส้น (อุณหภูมิ + ความชื้น), gauge, status zone (สี), ปุ่มควบคุม Manual/Auto

---

### 4. ระบบนับชิ้นงานบนสายพานลำเลียง (Conveyor Part Counter)

**บริบท:** สายพานในสายการผลิต ต้องการนับจำนวนชิ้นงานที่ผ่านในแต่ละชั่วโมง  
**เซนเซอร์:** HC-SR04 (วัดระยะ จำลองเป็น sensor ตรวจจับชิ้นงาน) + เซนเซอร์อุณหภูมิมอเตอร์  
**ตัวกระทำ:** LED แจ้งเตือน, Buzzer เมื่อผิดเป้า  
**Dashboard:** Counter widget, Bar chart ชิ้นงานต่อชั่วโมง, gauge อุณหภูมิมอเตอร์, ตั้ง target แบบ RPC

---

### 5. ระบบตรวจจับแก๊สรั่วและระบายอากาศ (Gas Leak Detection & Ventilation)

**บริบท:** ห้อง compressor หรือห้องเก็บก๊าซอุตสาหกรรม ต้องการแจ้งเตือนเมื่อตรวจพบก๊าซ  
**เซนเซอร์:** MQ-2 (เซนเซอร์แก๊ส) + DHT22  
**ตัวกระทำ:** พัดลมระบายอากาศ (relay) + Buzzer + LED แจ้งเตือน  
**Dashboard:** Gauge ระดับแก๊ส, กราฟเส้น, สถานะ "ปลอดภัย/อันตราย", ปุ่มระบายอากาศ Manual

---

### 6. ระบบวัดและบันทึกพลังงานไฟฟ้า (Energy Monitoring)

**บริบท:** โรงงานต้องการติดตามการใช้พลังงานของเครื่องจักรเพื่อลดต้นทุน  
**เซนเซอร์:** Potentiometer จำลองเป็นเซนเซอร์กระแสไฟฟ้า + เซนเซอร์แรงดัน (ADC)  
**ตัวกระทำ:** Relay ตัดวงจรเมื่อกระแสเกิน, LED บ่งบอกระดับการใช้พลังงาน  
**Dashboard:** กราฟเส้นพลังงาน, ค่าพลังงานสะสม (kWh), Alarm เมื่อเกิน threshold

---

### 7. ระบบตรวจสอบระดับน้ำในถัง (Tank Level Monitoring)

**บริบท:** ถังเก็บน้ำหล่อเย็นหรือน้ำมันหล่อลื่นในโรงงาน ต้องควบคุมระดับให้เหมาะสม  
**เซนเซอร์:** HC-SR04 วัดระดับน้ำ + Potentiometer จำลองเซนเซอร์อุณหภูมิน้ำ  
**ตัวกระทำ:** รีเลย์ควบคุมปั๊มเติมน้ำ  
**Dashboard:** กราฟระดับน้ำ, gauge, สถานะปั๊ม, ปุ่มสั่งงานปั๊ม, Alarm ระดับต่ำ/สูงเกิน

---

### 8. ระบบที่จอดรถ/ตรวจสอบช่องว่าง (Parking / Space Detection)

**บริบท:** ลานจอดรถโรงงาน ต้องการแสดงจำนวนช่องว่างแบบ Real-time  
**เซนเซอร์:** HC-SR04 × 2–3 ตัว จำลองตรวจสอบหลายช่อง  
**ตัวกระทำ:** LED Matrix หรือ LED หลายดวงแสดงสถานะช่อง  
**Dashboard:** แผนผัง Top-down แสดงสถานะช่องว่าง, จำนวนช่องว่างทั้งหมด, ประวัติ

## เกณฑ์การให้คะแนน (Rubric)

คะแนนโครงงานเต็ม = **15 คะแนน** ของคะแนนรวม ประกอบด้วย 6 เกณฑ์

| เกณฑ์ | CLO | น้ำหนัก (คะแนน) | ดีเยี่ยม (4) | ดี (3) | พอใช้ (2) | ต้องปรับปรุง (1) |
|---|:---:|:---:|---|---|---|---|
| **1. ความสมบูรณ์ End-to-End** | CLO3 | /4 | ระบบทำงานครบ 4 ชั้น IoT ไม่มีสะดุด ส่งข้อมูลต่อเนื่อง | ระบบทำงานครบแต่มีสะดุดเล็กน้อย หรือหนึ่งชั้นยังบกพร่อง | ระบบทำงานได้บางส่วน เช่น ส่งข้อมูลได้แต่ RPC ไม่ทำงาน | ระบบเชื่อมต่อไม่ได้หรือใช้งานไม่ได้จริง |
| **2. ความถูกต้องของโค้ดและตรรกะ** | CLO3 | /3 | โค้ดสะอาด มี comment, ตรรกะควบคุมถูกต้อง ไม่มี blocking delay, จัดการ reconnect | โค้ดทำงานถูกต้อง มี comment บ้าง มี delay แต่ยอมรับได้ | โค้ดทำงานได้แต่ตรรกะบางจุดผิดหรือไม่ครบ ขาด comment | โค้ดทำงานได้บางส่วน มีข้อผิดพลาดหลายจุด |
| **3. Dashboard และ UX** | CLO3 | /3 | Widget ≥ 2 ชนิด ออกแบบ layout สวยงาม ใช้งานง่าย หน่วยถูกต้อง มีชื่อ label ครบ | Widget ≥ 2 ชนิด ใช้งานได้ดี layout พอใช้ อาจขาด label หรือหน่วย | มี Widget แต่ชนิดเดียว หรือ layout สับสน | Dashboard ไม่สมบูรณ์ ข้อมูลไม่อัปเดตหรือผิดพลาด |
| **4. การแจ้งเตือน/Threshold** | CLO4 | /2 | Alarm ทำงานถูกต้อง ตั้ง threshold สมเหตุสมผล มีการแจ้งเตือนหลายระดับหรือควบคุม actuator อัตโนมัติ | Alarm ทำงานได้ threshold สมเหตุสมผล | Alarm ตั้งได้แต่ไม่ trigger อย่างถูกต้อง หรือ threshold ไม่สมเหตุสมผล | ไม่มีระบบ Alarm หรือ threshold ไม่ทำงาน |
| **5. รายงานและการนำเสนอ** | CLO4 | /2 | รายงานครบถ้วน เขียนได้ดี มีแผนภาพระบบ; นำเสนอชัดเจน ตอบคำถามได้ดี | รายงานครบ เขียนได้พอใช้; นำเสนอเข้าใจได้ ตอบคำถามได้ส่วนใหญ่ | รายงานไม่ครบหรือขาดรายละเอียดสำคัญ; นำเสนอพอใช้ | รายงานไม่ส่งหรือไม่สมบูรณ์มาก; นำเสนอขาดความชัดเจน |
| **6. การทำงานเป็นทีม** | CLO4 | /1 | สมาชิกทุกคนมีส่วนร่วมชัดเจน ตอบคำถามเกี่ยวกับงานตัวเองได้ทุกคน | สมาชิกส่วนใหญ่มีส่วนร่วม มีบางคนที่ตอบได้ไม่ครบ | การแบ่งงานไม่สมดุล สมาชิกบางคนไม่เข้าใจงานส่วนอื่น | สมาชิกส่วนใหญ่ไม่เข้าใจระบบ มีคนเดียวที่ทำงาน |
| **รวม** | CLO3, CLO4 | **/15** | | | | |

> **หมายเหตุ:** คะแนนแต่ละเกณฑ์ใช้ระดับ 1–4 แล้วปรับตามน้ำหนักของแต่ละเกณฑ์เพื่อให้รวมได้ 15 คะแนน คะแนนต่ำกว่า 7.5 (50%) ถือว่าโครงงานไม่ผ่านเกณฑ์ขั้นต่ำ

- **หน้า 3 — สถาปัตยกรรมระบบ:** Block Diagram แสดงการไหลของข้อมูล, รายการอุปกรณ์/เซนเซอร์
- **หน้า 4 — การออกแบบฮาร์ดแวร์:** แผนผังวงจร Wokwi, อธิบายการต่อวงจร
- **หน้า 5–6 — โค้ดและตรรกะ:** อธิบายส่วนสำคัญของโค้ด, โครงสร้าง MQTT topic, JSON payload ตัวอย่าง
- **หน้า 7 — แดชบอร์ดและผลลัพธ์:** Screenshot Dashboard, อธิบาย Widget, ผลการทดสอบ
- **หน้า 8 — สรุปและข้อเสนอแนะ:** สรุปผล, ปัญหาที่พบและวิธีแก้ไข, แนวทางพัฒนาต่อ

## Checklist ก่อนส่งงาน

ใช้ checklist นี้ตรวจสอบก่อนส่งงานและก่อนนำเสนอ:

### ด้านเทคนิค

- [ ] ESP32 บน Wokwi เชื่อมต่อ Wi-Fi ได้สำเร็จ (เห็น "Connected" ใน Serial Monitor)
- [ ] ส่งข้อมูล telemetry ไปยัง ThingsBoard ได้ (เห็น data ใน Latest Telemetry บน ThingsBoard)
- [ ] Dashboard มี Widget อย่างน้อย 2 ชนิดที่แสดงข้อมูล Real-time
- [ ] ตัวกระทำ (actuator) ทำงานตอบสนองต่อค่าเซนเซอร์หรือคำสั่ง RPC ได้
- [ ] Alarm/Threshold ทำงานเมื่อค่าเกินที่ตั้ง (สามารถสาธิตได้)
- [ ] โค้ดมี comment อธิบายส่วนสำคัญ และ handle การ reconnect MQTT

### ด้านเอกสาร

- [ ] รายงาน PDF มีเนื้อหาครบตามรูปแบบที่กำหนด (5–8 หน้า)
- [ ] Block Diagram ระบบชัดเจนและถูกต้อง
- [ ] Screenshot Dashboard ที่มีข้อมูลจริงแนบในรายงาน
- [ ] ระบุ MQTT Topic ที่ใช้ในรายงาน
- [ ] ระบุชื่อสมาชิกและบทบาทของแต่ละคนในรายงาน

### ด้านการนำเสนอ

- [ ] ทุกคนในทีมเข้าใจระบบและพร้อมตอบคำถาม
- [ ] เปิด Wokwi Simulator พร้อม demo สด (ไม่ใช่ video บันทึก)
- [ ] เปิด ThingsBoard Dashboard ที่แสดงข้อมูลจริงพร้อม
- [ ] เตรียมทดสอบ threshold triggering ระหว่าง demo

## คำถามที่มักถามบ่อย (FAQ)

**Q: หากทีมมีสมาชิก 2 คน ได้หรือไม่?**  
A: ได้ แต่ต้องแจ้งอาจารย์ล่วงหน้า และความซับซ้อนของโครงงานจะถูกปรับลดตามสัดส่วน

**Q: สามารถใช้ ESP32 จริง (ไม่ใช้ Wokwi) ได้หรือไม่?**  
A: ได้ เป็นโบนัส แต่การ demo หลักต้องมีบน Wokwi ด้วยเพื่อให้เพื่อนร่วมชั้นและอาจารย์ตรวจสอบวงจรได้

**Q: หัวข้อโครงงานต้องเกี่ยวกับวิศวกรรมเครื่องกลเท่านั้นหรือ?**  
A: ไม่จำเป็น แต่ต้องเป็นโจทย์จากงานจริง (Real-world Problem) ที่มีการ monitor และควบคุมระบบฟิสิกส์บางอย่าง

**Q: ThingsBoard ฟรีทำได้กี่ Device?**  
A: ThingsBoard Community Edition (cloud demo) รองรับได้เพียงพอสำหรับงานนี้ — กลุ่มต้องการเพียง 1 Device บัญชีต่อกลุ่ม

**Q: ถ้า Wokwi ไม่รองรับเซนเซอร์บางตัว (เช่น MQ-2) จะทำอย่างไร?**  
A: ใช้ Potentiometer หรือ Slider จำลองค่าเซนเซอร์นั้นได้ ระบุไว้ในรายงานว่าจำลองด้วยอะไร
