## 6.1 พื้นฐานการสื่อสารไร้สาย (Wireless Communication Fundamentals)

การสื่อสารไร้สาย (Wireless Communication) คือการส่งข้อมูลระหว่างอุปกรณ์โดยอาศัย **คลื่นแม่เหล็กไฟฟ้า (Electromagnetic Waves)** แทนการใช้สายสัญญาณทางกายภาพ ซึ่งเป็นหัวใจสำคัญของระบบ IoT เพราะอุปกรณ์เซ็นเซอร์จำนวนมากติดตั้งอยู่ในจุดที่เดินสายลำบาก

### 6.1.1 คลื่นความถี่และย่าน ISM

คลื่นวิทยุ (Radio Frequency — RF) ที่ใช้ในงาน IoT ส่วนใหญ่อยู่ในย่าน **ISM Band (Industrial, Scientific and Medical)** ซึ่งเป็นย่านความถี่ที่สามารถใช้งานได้โดยไม่ต้องมีใบอนุญาต (Unlicensed) ย่านที่พบบ่อย ได้แก่

| ย่านความถี่ | ตัวอย่างการใช้งาน |
|---|---|
| 433 MHz | รีโมตคอนโทรล, LoRa (บางภูมิภาค) |
| 868 MHz (ยุโรป) / 915 MHz (อเมริกา) / 923 MHz (ไทย, อาเซียน) | LoRa, Zigbee |
| 2.4 GHz | Wi-Fi, Bluetooth, Zigbee |
| 5 GHz | Wi-Fi 5/6 |

### 6.1.2 แบนด์วิดท์ (Bandwidth)

แบนด์วิดท์คือ **ช่วงความถี่ที่ใช้ในการส่งข้อมูล** ยิ่งแบนด์วิดท์กว้างมาก ยิ่งส่งข้อมูลได้เร็ว แต่มักกินพลังงานมากขึ้นตามไปด้วย สำหรับ IoT หลายงานไม่ต้องการแบนด์วิดท์สูง เช่น การส่งค่าอุณหภูมิเพียงไม่กี่ไบต์ทุก ๆ นาที

### 6.1.3 การลดทอนสัญญาณ (Signal Attenuation)

สัญญาณไร้สายจะ **อ่อนลง** ตามระยะทางและสิ่งกีดขวาง ปัจจัยหลักที่มีผล ได้แก่

- **ระยะทาง (Distance)** — พลังงานสัญญาณลดลงตามกำลังสองของระยะทาง (Inverse Square Law)
- **สิ่งกีดขวาง (Obstacles)** — ผนังคอนกรีต โลหะ น้ำ ดูดซับสัญญาณได้มาก
- **สัญญาณรบกวน (Interference)** — อุปกรณ์อื่นที่ใช้ความถี่เดียวกัน เช่น เตาไมโครเวฟรบกวนย่าน 2.4 GHz

> 💡 **เคล็ดลับ:** หน่วยวัดกำลังสัญญาณที่นิยมใช้คือ **dBm** ค่ายิ่งติดลบมาก สัญญาณยิ่งอ่อน เช่น -30 dBm = สัญญาณดีมาก, -90 dBm = สัญญาณอ่อนมาก อาจเชื่อมต่อไม่ได้

## 6.2 Wi-Fi

### 6.2.1 มาตรฐาน IEEE 802.11

Wi-Fi เป็นเทคโนโลยีไร้สายที่แพร่หลายที่สุดในชีวิตประจำวัน ทำงานตามมาตรฐาน **IEEE 802.11** มาตรฐานที่เกี่ยวข้องกับ IoT ได้แก่

| มาตรฐาน | ชื่อทางการค้า | ความถี่ | อัตราข้อมูลสูงสุด |
|---|---|---|---|
| 802.11b | Wi-Fi 1 | 2.4 GHz | 11 Mbps |
| 802.11g | Wi-Fi 3 | 2.4 GHz | 54 Mbps |
| 802.11n | Wi-Fi 4 | 2.4 / 5 GHz | 600 Mbps |
| 802.11ac | Wi-Fi 5 | 5 GHz | 6.9 Gbps |
| 802.11ax | Wi-Fi 6/6E | 2.4 / 5 / 6 GHz | 9.6 Gbps |
| 802.11ah | Wi-Fi HaLow | Sub-1 GHz | 8.7 Mbps |

สำหรับบอร์ด **ESP32** จะรองรับ Wi-Fi 4 (802.11b/g/n) ที่ย่าน **2.4 GHz** เท่านั้น

### 6.2.2 ย่าน 2.4 GHz vs 5 GHz

- **2.4 GHz** — ทะลุผ่านสิ่งกีดขวางได้ดีกว่า ระยะไกลกว่า แต่มีแค่ 3 ช่องสัญญาณที่ไม่ซ้อนทับ (Non-overlapping Channels) และมักแออัด
- **5 GHz** — ช่องสัญญาณมากกว่า รบกวนน้อย แต่ทะลุผนังได้แย่กว่า ระยะสั้นกว่า

### 6.2.3 โหมด Station และ Access Point

ESP32 สามารถทำงานได้ 2 โหมดหลัก:

- **Station Mode (STA)** — เชื่อมต่อเข้ากับเราเตอร์ Wi-Fi เหมือนมือถือหรือโน้ตบุ๊กทั่วไป
- **Access Point Mode (AP)** — ตัว ESP32 เป็นจุดกระจายสัญญาณเอง ให้อุปกรณ์อื่นเชื่อมเข้ามา
- **AP+STA Mode** — ทำทั้งสองอย่างพร้อมกัน

ตัวอย่างโค้ด ESP32 เชื่อมต่อ Wi-Fi ในโหมด Station:

```cpp
#include <WiFi.h>

const char* ssid = "MyNetwork";
const char* password = "MyPassword";

void setup() {
  Serial.begin(115200);
  WiFi.begin(ssid, password);

  Serial.print("Connecting to Wi-Fi");
  while (WiFi.status() != WL_CONNECTED) {
    delay(500);
    Serial.print(".");
  }
  Serial.println("\nConnected!");
  Serial.print("IP Address: ");
  Serial.println(WiFi.localIP());
}

void loop() {
  // ทำงานอื่น ๆ ที่ต้องการ
}
```

### 6.2.4 จุดเด่นและข้อจำกัดสำหรับ IoT

| จุดเด่น | ข้อจำกัด |
|---|---|
| อัตราข้อมูลสูง เหมาะส่งภาพ/เสียง | กินพลังงานค่อนข้างมาก |
| โครงสร้างพื้นฐานแพร่หลาย มีเราเตอร์ทุกที่ | ไม่เหมาะกับอุปกรณ์ที่ใช้แบตเตอรี่ยาวนาน |
| ESP32 รองรับในตัว ไม่ต้องซื้อโมดูลเพิ่ม | ระยะทางจำกัด (ภายในอาคาร ~30-50 ม.) |

## 6.3 Bluetooth และ BLE (Bluetooth Low Energy)

### 6.3.1 Bluetooth Classic vs BLE

- **Bluetooth Classic** — ออกแบบมาเพื่อส่งข้อมูลต่อเนื่อง (Streaming) เช่น เสียงเพลง อัตราข้อมูลสูงถึง 3 Mbps แต่กินพลังงานมาก
- **BLE (Bluetooth Low Energy)** — เพิ่มเข้ามาใน Bluetooth 4.0 ออกแบบมาเพื่อ **ประหยัดพลังงานสูงสุด** ส่งข้อมูลเป็นชุดเล็ก ๆ (เช่น ค่าเซ็นเซอร์) แล้วกลับไปหลับ (Sleep)

### 6.3.2 สถาปัตยกรรม GATT

BLE ใช้โครงสร้างข้อมูลแบบ **GATT (Generic Attribute Profile)** ประกอบด้วย

- **Service** — กลุ่มของข้อมูลที่สัมพันธ์กัน (เช่น Heart Rate Service)
- **Characteristic** — ค่าข้อมูลแต่ละตัวภายใน Service (เช่น Heart Rate Measurement)

### 6.3.3 การประยุกต์ใช้ใน IoT

- สายรัดข้อมือวัดสุขภาพ (Wearable Sensor)
- iBeacon / Eddystone สำหรับระบุตำแหน่งภายในอาคาร (Indoor Positioning)
- รีโมตควบคุมอุปกรณ์ในบ้าน

ESP32 รองรับทั้ง **Bluetooth Classic** และ **BLE** ในชิปตัวเดียว

## 6.4 Zigbee

### 6.4.1 ภาพรวม

Zigbee ทำงานตามมาตรฐาน **IEEE 802.15.4** ที่ย่าน 2.4 GHz มีอัตราข้อมูล 250 kbps ออกแบบมาเพื่อ **เครือข่ายเซ็นเซอร์พลังงานต่ำ** ในระยะใกล้ถึงปานกลาง

### 6.4.2 เครือข่ายแบบ Mesh

จุดเด่นสำคัญของ Zigbee คือ **Mesh Topology** — อุปกรณ์สามารถส่งต่อข้อมูลผ่านโหนดอื่นได้ ทำให้:

- ครอบคลุมพื้นที่กว้างขึ้นโดยไม่ต้องเพิ่มกำลังส่ง
- ระบบมีความทนทาน (Resilient) ถ้าโหนดหนึ่งเสียหาย ข้อมูลหาเส้นทางอื่นได้

### 6.4.3 บทบาทของอุปกรณ์ในเครือข่าย Zigbee

- **Coordinator** — ศูนย์กลางเครือข่าย ต้องมี 1 ตัวเสมอ
- **Router** — รับ-ส่งต่อข้อมูล เปิดทำงานตลอด
- **End Device** — อุปกรณ์ปลายทาง เช่น เซ็นเซอร์ ส่งข้อมูลแล้วเข้าสู่โหมดหลับเพื่อประหยัดพลังงาน

> 💡 **รู้หรือไม่:** ระบบ Smart Home ยอดนิยมอย่าง Philips Hue, IKEA TRÅDFRI และ SmartThings ล้วนใช้โปรโตคอล Zigbee (หรือ Matter ซึ่งอิงพื้นฐาน Zigbee) ในการสื่อสารระหว่างอุปกรณ์

## 6.5 LoRa / LoRaWAN

### 6.5.1 LoRa คืออะไร

**LoRa (Long Range)** เป็นเทคโนโลยีการมอดูเลตแบบ **CSS (Chirp Spread Spectrum)** ที่สามารถส่งสัญญาณได้ระยะไกลมาก (ในพื้นที่โล่ง 5-15 กม.) โดยใช้พลังงานต่ำ จัดอยู่ในกลุ่ม **LPWAN (Low Power Wide Area Network)**

### 6.5.2 LoRaWAN

**LoRaWAN** คือโปรโตคอลชั้นเครือข่ายที่ทำงานบน LoRa กำหนดเรื่อง:

- การเข้ารหัสข้อมูล (Encryption — AES-128)
- การจัดคลาสอุปกรณ์ (Class A / B / C) ตามความต้องการรับข้อมูล
- สถาปัตยกรรม: End Device → Gateway → Network Server → Application Server

### 6.5.3 ข้อแลกเปลี่ยน

- **อัตราข้อมูลต่ำมาก** (0.3-50 kbps) เหมาะส่งค่าเซ็นเซอร์สั้น ๆ เท่านั้น
- มีข้อจำกัดเรื่อง **Duty Cycle** ในบางประเทศ ไม่สามารถส่งข้อมูลถี่ ๆ ได้

### 6.5.4 ตัวอย่างการใช้งาน

- ระบบตรวจวัดสภาพอากาศในไร่นา (Smart Agriculture)
- มิเตอร์น้ำ/ไฟฟ้าอัจฉริยะ (Smart Metering)
- การติดตามสินทรัพย์ (Asset Tracking)

## 6.6 เครือข่ายเซลลูลาร์สำหรับ IoT (Cellular IoT)

### 6.6.1 NB-IoT (Narrowband IoT)

- ใช้โครงข่ายเสาโทรศัพท์มือถือที่มีอยู่ ไม่ต้องติดตั้ง Gateway เอง
- แบนด์วิดท์แคบมาก (~200 kHz) อัตราข้อมูล ~250 kbps
- เจาะทะลุอาคาร/ใต้ดินได้ดี (Deep Indoor Penetration)
- เหมาะกับ Smart Metering, Parking Sensor

### 6.6.2 LTE-M (LTE Cat-M1)

- รองรับ Handover ระหว่างเสา — เหมาะกับอุปกรณ์ที่เคลื่อนที่ (Mobility)
- อัตราข้อมูลสูงกว่า NB-IoT (~1 Mbps)
- รองรับเสียง (VoLTE) ได้

### 6.6.3 5G สำหรับ IoT

5G แบ่งย่อยเป็นหลายสเปก:

- **mMTC (massive Machine-Type Communications)** — เชื่อมอุปกรณ์จำนวนมหาศาลต่อตร.กม.
- **URLLC (Ultra-Reliable Low Latency Communications)** — ความหน่วงต่ำมาก เหมาะกับรถยนต์ไร้คนขับ หุ่นยนต์อุตสาหกรรม

### 6.6.4 ข้อดีและข้อจำกัดของ Cellular IoT

| ข้อดี | ข้อจำกัด |
|---|---|
| ครอบคลุมพื้นที่กว้างระดับประเทศ | ต้องจ่ายค่าบริการรายเดือนให้ผู้ให้บริการ |
| ไม่ต้องติดตั้ง Gateway เอง | ต้องใช้ซิมการ์ด |
| ความปลอดภัยระดับเครือข่ายโทรศัพท์ | กินพลังงานมากกว่า LoRa, Zigbee |

## 6.7 ตารางเปรียบเทียบเทคโนโลยีไร้สาย (Comparison Table)

ตารางด้านล่างเปรียบเทียบเทคโนโลยีไร้สายหลักที่ใช้ในระบบ IoT ทั้ง 6 เทคโนโลยี:

| เกณฑ์ | Wi-Fi | BLE | Zigbee | LoRa/LoRaWAN | NB-IoT | LTE-M |
|---|---|---|---|---|---|---|
| **ระยะทาง** | 30-100 ม. | 10-100 ม. | 10-100 ม. (mesh ขยายได้) | 2-15 กม. | 1-10 กม. | 1-10 กม. |
| **อัตราข้อมูล** | สูงมาก (Mbps-Gbps) | ปานกลาง (1-2 Mbps) | ต่ำ (250 kbps) | ต่ำมาก (0.3-50 kbps) | ต่ำ (~250 kbps) | ปานกลาง (~1 Mbps) |
| **การใช้พลังงาน** | สูง | ต่ำ | ต่ำมาก | ต่ำมาก | ต่ำ | ปานกลาง |
| **อายุแบตเตอรี่** | ชั่วโมง-วัน | เดือน-ปี | ปี | ปี (5-10 ปี) | ปี (10+ ปี) | ปี |
| **โทโพโลยี** | Star | Star, Mesh (BT Mesh) | Mesh | Star-of-Stars | Star (ผ่านเสาสัญญาณ) | Star |
| **ย่านความถี่** | 2.4 / 5 GHz | 2.4 GHz | 2.4 GHz / Sub-GHz | Sub-GHz (ISM) | Licensed LTE Band | Licensed LTE Band |
| **ค่าใช้จ่าย** | ต่ำ (ไม่มีค่ารายเดือน) | ต่ำ | ปานกลาง | ต่ำ-ปานกลาง | มีค่ารายเดือน | มีค่ารายเดือน |
| **ตัวอย่างใช้งาน** | กล้อง IP, จอแสดงผล | Wearable, Beacon | Smart Home, โรงงาน | เกษตร, มิเตอร์ | มิเตอร์, จอดรถ | Tracker, Wearable |

> 💡 **กฎง่าย ๆ ในการจำ:** ถ้าต้องการ **ข้อมูลเยอะ ระยะใกล้** → Wi-Fi, ถ้าต้องการ **ระยะไกล ข้อมูลน้อย** → LoRa, ถ้าต้องการ **ประหยัดพลังงานสุด ๆ ระยะใกล้** → BLE/Zigbee, ถ้าต้องการ **ครอบคลุมทุกที่ไม่ต้องติดตั้งอะไรเอง** → Cellular IoT

## 6.8 เกณฑ์การเลือกเทคโนโลยีให้เหมาะกับโจทย์

### 6.8.1 ปัจจัยหลักในการตัดสินใจ

เมื่อต้องเลือกเทคโนโลยีไร้สายสำหรับโปรเจกต์ IoT ให้พิจารณาปัจจัยต่อไปนี้:

1. **ระยะทาง (Range)** — อุปกรณ์อยู่ห่างจากจุดรับข้อมูลแค่ไหน?
2. **ปริมาณข้อมูล (Data Rate)** — ต้องส่งข้อมูลมากน้อยเพียงใด? ส่งบ่อยแค่ไหน?
3. **แหล่งพลังงาน (Power Source)** — ต่อไฟบ้านได้ หรือต้องใช้แบตเตอรี่?
4. **จำนวนอุปกรณ์ (Scalability)** — ต้องเชื่อมต่อกี่ตัว?
5. **ต้นทุน (Cost)** — ทั้งค่าอุปกรณ์และค่าบริการต่อเนื่อง
6. **สภาพแวดล้อม (Environment)** — ในอาคาร กลางแจ้ง ใต้ดิน หรือในโรงงาน?
7. **ความปลอดภัย (Security)** — ข้อมูลมีความอ่อนไหวมากน้อยแค่ไหน?
8. **ความพร้อมโครงสร้างพื้นฐาน (Infrastructure)** — มีเราเตอร์ Wi-Fi อยู่แล้วหรือไม่? มีเครือข่าย NB-IoT ครอบคลุมหรือไม่?

### 6.8.2 กรณีศึกษา (Case Studies)

#### กรณีที่ 1: ระบบตรวจวัดอุณหภูมิในห้องเซิร์ฟเวอร์

- **โจทย์:** วัดอุณหภูมิ 20 จุดภายในอาคาร ส่งค่าทุก 30 วินาที มีไฟเลี้ยงตลอด
- **การวิเคราะห์:** ระยะใกล้ ข้อมูลน้อย มีไฟเลี้ยง มีเราเตอร์อยู่แล้ว
- ✅ **เลือก: Wi-Fi (ESP32)** — ใช้โครงสร้างเดิม เขียนโค้ดง่าย ส่งค่าขึ้น Dashboard ได้เลย

#### กรณีที่ 2: ระบบตรวจวัดความชื้นดินในแปลงเกษตร

- **โจทย์:** แปลงขนาด 50 ไร่ เซ็นเซอร์ 100 จุด ใช้แบตเตอรี่ ส่งค่าทุก 15 นาที
- **การวิเคราะห์:** ระยะไกล ข้อมูลน้อย ต้องประหยัดพลังงานมาก ไม่มีไฟฟ้า
- ✅ **เลือก: LoRa/LoRaWAN** — ระยะไกล ประหยัดแบตเตอรี่ได้หลายปี

#### กรณีที่ 3: สายรัดข้อมือวัดชีพจรสำหรับนักกีฬา

- **โจทย์:** ส่งค่าชีพจรไปยังมือถือแบบ Real-time ใช้แบตเตอรี่เล็ก ระยะ < 5 ม.
- **การวิเคราะห์:** ระยะใกล้มาก ข้อมูลน้อย ประหยัดพลังงาน จับคู่กับมือถือ
- ✅ **เลือก: BLE** — ประหยัดพลังงาน มือถือรองรับอยู่แล้ว

> 💡 **สำหรับการทดลองบน Wokwi:** ตัวจำลอง Wokwi รองรับ **Wi-Fi** ของ ESP32 ได้ ทำให้สามารถจำลองการเชื่อมต่อ Wi-Fi, ส่งข้อมูลผ่าน HTTP/MQTT ได้เสมือนจริง แต่ยังไม่รองรับ BLE, LoRa, Zigbee ในตัวจำลอง ดังนั้นในแลปเราจะเน้นทดลอง Wi-Fi เป็นหลัก

### 6.8.3 ผังตัดสินใจอย่างง่าย (Decision Flowchart)

```
เริ่มต้น
  │
  ├─ ระยะทาง > 1 กม.?
  │    ├─ ใช่ → ข้อมูล > 100 kbps?
  │    │          ├─ ใช่ → Cellular (NB-IoT / LTE-M)
  │    │          └─ ไม่  → LoRa / LoRaWAN
  │    └─ ไม่  → ต้องการอัตราข้อมูลสูง?
  │               ├─ ใช่ → Wi-Fi
  │               └─ ไม่  → ต้องการ Mesh?
  │                          ├─ ใช่ → Zigbee
  │                          └─ ไม่  → BLE
```

## สรุปบทเรียน (Summary)

1. การสื่อสารไร้สายอาศัยคลื่นแม่เหล็กไฟฟ้า โดย IoT ใช้ย่าน ISM Band เป็นหลัก
2. **Wi-Fi** เหมาะกับงานที่ต้องการข้อมูลมาก ระยะใกล้ มีไฟเลี้ยง — ESP32 รองรับในตัว
3. **BLE** เหมาะกับ Wearable และเซ็นเซอร์ใกล้ตัวที่ต้องประหยัดพลังงาน
4. **Zigbee** เหมาะกับระบบ Smart Home ที่ต้องการ Mesh Network
5. **LoRa/LoRaWAN** เหมาะกับเซ็นเซอร์ระยะไกลในพื้นที่เปิด เช่น เกษตรกรรม
6. **NB-IoT / LTE-M / 5G** เหมาะกับงานที่ต้องครอบคลุมระดับเมืองหรือประเทศ
7. ไม่มีเทคโนโลยีใดดีที่สุดสำหรับทุกงาน — ต้อง **วิเคราะห์โจทย์** แล้วเลือกให้เหมาะสม

## แบบฝึกหัดท้ายบท (Exercises)

**ข้อ 1:** จงอธิบายว่าทำไมย่าน ISM Band จึงเป็นที่นิยมสำหรับอุปกรณ์ IoT และมีข้อเสียอะไรบ้าง

**ข้อ 2:** สัญญาณ Wi-Fi ที่ย่าน 2.4 GHz และ 5 GHz มีข้อดี-ข้อเสียต่างกันอย่างไร? ถ้าต้องติดตั้งเซ็นเซอร์ IoT ในโรงงานที่มีผนังคอนกรีตหนา ควรเลือกย่านใด เพราะเหตุใด?

**ข้อ 3:** บอร์ด ESP32 ทำงานในโหมด Access Point (AP) ต่างจากโหมด Station (STA) อย่างไร? ให้ยกตัวอย่างสถานการณ์ที่เหมาะสมกับแต่ละโหมด อย่างละ 1 สถานการณ์

**ข้อ 4:** จงเปรียบเทียบ Bluetooth Classic กับ BLE ในแง่ของพลังงาน อัตราข้อมูล และรูปแบบการใช้งานที่เหมาะสม

**ข้อ 5:** โรงงานแห่งหนึ่งต้องการติดตั้งเซ็นเซอร์วัดการสั่นสะเทือน (Vibration Sensor) จำนวน 200 จุด กระจายทั่วพื้นที่โรงงาน 10,000 ตร.ม. ส่งค่าทุก 1 วินาที มีไฟเลี้ยงทุกจุด จงเลือกเทคโนโลยีไร้สายที่เหมาะสมที่สุด พร้อมให้เหตุผลประกอบ

**ข้อ 6:** ฟาร์มอัจฉริยะต้องการติดตั้งเซ็นเซอร์วัดความชื้นดินและอุณหภูมิจำนวน 50 จุด กระจายในพื้นที่ 100 ไร่ ใช้โซลาร์เซลล์ขนาดเล็ก ส่งค่าทุก 30 นาที จงเลือกเทคโนโลยีและอธิบายเหตุผล

**ข้อ 7:** จงเขียนโค้ด Arduino สำหรับ ESP32 ที่ทำงานในโหมด Access Point (AP) ตั้งชื่อเครือข่าย "IoT_Lab" รหัส "12345678" และแสดง IP Address ของตัว AP บน Serial Monitor (ทดสอบบน Wokwi)

**ข้อ 8:** จากตารางเปรียบเทียบในหัวข้อ 6.7 จงวิเคราะห์ว่าเทคโนโลยีใดเหมาะสมที่สุดสำหรับระบบติดตามรถขนส่งสินค้า (Fleet Tracking) ที่วิ่งทั่วประเทศ ต้องส่งพิกัด GPS ทุก 5 นาที พร้อมอธิบายว่าทำไมเทคโนโลยีอื่นไม่เหมาะสม
