## 8.1 คลาวด์คอมพิวติงคืออะไร

**คลาวด์คอมพิวติง (Cloud Computing)** คือการให้บริการทรัพยากรคอมพิวเตอร์ เช่น เซิร์ฟเวอร์ พื้นที่จัดเก็บข้อมูล ฐานข้อมูล และซอฟต์แวร์ ผ่านทางอินเทอร์เน็ต โดยผู้ใช้ไม่จำเป็นต้องลงทุนซื้อเครื่องเซิร์ฟเวอร์เอง แต่จ่ายค่าบริการตามที่ใช้จริง (Pay-as-you-go)

เปรียบเทียบง่าย ๆ: เหมือนกับการใช้ไฟฟ้า — เราไม่จำเป็นต้องสร้างโรงไฟฟ้าเอง เพียงเสียบปลั๊กแล้วจ่ายตามมิเตอร์

### ทำไมคลาวด์จึงสำคัญสำหรับ IoT?

ในระบบ IoT อุปกรณ์เซนเซอร์จำนวนมากจะส่งข้อมูลขึ้นมาอย่างต่อเนื่อง บอร์ด ESP32 ตัวเล็ก ๆ ไม่สามารถเก็บข้อมูลเป็นล้าน ๆ รายการได้ จึงต้องอาศัย "คลาวด์" เป็นศูนย์กลาง

| ประโยชน์ | คำอธิบาย |
|---|---|
| **การขยายตัว (Scalability)** | รองรับอุปกรณ์จำนวนมากได้โดยไม่ต้องเพิ่มฮาร์ดแวร์เอง |
| **การจัดเก็บ (Storage)** | เก็บข้อมูลเซนเซอร์ย้อนหลังเป็นเดือน/ปีได้ ไม่ต้องกังวลเรื่องพื้นที่ |
| **การวิเคราะห์ (Analytics)** | ใช้เครื่องมือบนคลาวด์วิเคราะห์แนวโน้ม ทำ Dashboard แสดงผลแบบ Real-time |
| **การเข้าถึง (Accessibility)** | เปิดดูข้อมูลจากที่ไหนก็ได้ ผ่านเว็บหรือแอปมือถือ |
| **ลดต้นทุน (Cost Reduction)** | ไม่ต้องซื้อเซิร์ฟเวอร์ ไม่ต้องจ้างคนดูแลระบบ |

> 💡 **เคล็ดลับ:** สำหรับโปรเจกต์ IoT ระดับนักศึกษา แพลตฟอร์มอย่าง ThingSpeak หรือ Blynk มีแผนฟรีให้ใช้งานได้ทันที ไม่ต้องใช้บัตรเครดิต เหมาะกับการเรียนรู้และทดลอง

## 8.2 รูปแบบบริการคลาวด์ (Cloud Service Models)

บริการคลาวด์แบ่งออกเป็น 3 ระดับหลัก ตามขอบเขตที่ผู้ให้บริการดูแล:

### IaaS — Infrastructure as a Service

ให้บริการ "โครงสร้างพื้นฐาน" เช่น เซิร์ฟเวอร์เสมือน (Virtual Machine), พื้นที่จัดเก็บ (Storage), เครือข่าย (Network) ผู้ใช้ต้องติดตั้งระบบปฏิบัติการและซอฟต์แวร์เอง

**ตัวอย่าง:** AWS EC2, Google Compute Engine, DigitalOcean

### PaaS — Platform as a Service

ให้บริการ "แพลตฟอร์ม" สำเร็จรูปสำหรับพัฒนาและรันแอปพลิเคชัน ผู้ใช้เขียนโค้ดแล้ว Deploy ได้เลย ไม่ต้องจัดการเซิร์ฟเวอร์

**ตัวอย่าง:** Heroku, Google App Engine, Azure App Service

### SaaS — Software as a Service

ให้บริการ "ซอฟต์แวร์" พร้อมใช้งานผ่านเว็บเบราว์เซอร์ ผู้ใช้ไม่ต้องติดตั้งอะไร

**ตัวอย่าง:** Google Docs, Microsoft 365, ThingSpeak

### ตารางเปรียบเทียบรูปแบบบริการ

| รายการ | IaaS | PaaS | SaaS |
|---|---|---|---|
| ผู้ใช้จัดการ | OS, Runtime, App, Data | App, Data | Data เท่านั้น |
| ผู้ให้บริการจัดการ | Network, Storage, Server | + OS, Runtime | ทุกอย่าง |
| ความยืดหยุ่น | สูงมาก | ปานกลาง | ต่ำ |
| ความง่ายในการใช้งาน | ต่ำ (ต้องมีความรู้ระบบ) | ปานกลาง | สูงมาก |
| ตัวอย่างในบริบท IoT | ตั้งเซิร์ฟเวอร์ MQTT เอง | Deploy Node-RED บนคลาวด์ | ใช้ ThingSpeak แสดงกราฟ |

## 8.3 รูปแบบการติดตั้งคลาวด์ (Cloud Deployment Models)

### Public Cloud

คลาวด์ที่เปิดให้บุคคลทั่วไปใช้บริการ ทรัพยากรถูกแชร์ระหว่างผู้ใช้หลายราย (Multi-tenant) มีค่าใช้จ่ายต่ำและขยายตัวง่าย

**ตัวอย่าง:** AWS, Google Cloud, Microsoft Azure

### Private Cloud

คลาวด์ที่องค์กรสร้างขึ้นเองเพื่อใช้ภายในองค์กร มีความปลอดภัยสูง แต่ต้องลงทุนและดูแลเอง

**ตัวอย่าง:** ระบบ Data Center ของโรงงานอุตสาหกรรม

### Hybrid Cloud

ผสมผสานระหว่าง Public และ Private Cloud เช่น เก็บข้อมูลสำคัญไว้ใน Private Cloud แต่ใช้ Public Cloud สำหรับการวิเคราะห์ข้อมูลที่ต้องการพลังประมวลผลสูง

ในงานวิศวกรรมเครื่องกล เช่น ระบบ Smart Factory อาจใช้ Hybrid Cloud — ข้อมูลการผลิตที่เป็นความลับเก็บในเซิร์ฟเวอร์ภายในโรงงาน แต่ส่งข้อมูลเซนเซอร์สิ่งแวดล้อม (อุณหภูมิ ความชื้น) ขึ้น Public Cloud เพื่อวิเคราะห์แนวโน้ม

## 8.4 องค์ประกอบของแพลตฟอร์ม IoT

แพลตฟอร์ม IoT ที่สมบูรณ์ประกอบด้วยส่วนหลัก 6 ส่วน:

### 1. Device Management (การจัดการอุปกรณ์)
ลงทะเบียน ตรวจสอบสถานะ และอัปเดตเฟิร์มแวร์ของอุปกรณ์ IoT จากระยะไกล เช่น ตรวจสอบว่า ESP32 แต่ละตัวออนไลน์อยู่หรือไม่

### 2. Data Ingestion (การรับข้อมูลเข้า)
รับข้อมูลจากอุปกรณ์หลายพันตัวพร้อมกัน ผ่านโปรโตคอลต่าง ๆ เช่น MQTT, HTTP, CoAP โดยต้องรองรับปริมาณข้อมูลจำนวนมาก (High Throughput)

### 3. Data Storage (การจัดเก็บข้อมูล)
เก็บข้อมูลเซนเซอร์ลงฐานข้อมูล โดยมักใช้ฐานข้อมูลแบบ Time-series ที่ออกแบบมาสำหรับข้อมูลที่มี Timestamp กำกับ

### 4. Analytics (การวิเคราะห์ข้อมูล)
ประมวลผลข้อมูลเพื่อหาแนวโน้ม (Trend) ความผิดปกติ (Anomaly) หรือสร้างโมเดลทำนาย (Prediction)

### 5. Visualization (การแสดงผล)
แสดงข้อมูลในรูปแบบกราฟ แผนที่ หรือ Dashboard ให้มนุษย์อ่านเข้าใจง่าย

### 6. Rules & Alerts (กฎและการแจ้งเตือน)
ตั้งเงื่อนไขอัตโนมัติ เช่น "ถ้าอุณหภูมิเกิน 80°C ให้ส่ง Line Notify แจ้งเตือน" หรือ "ถ้าความชื้นในดินต่ำกว่า 30% ให้สั่งเปิดวาล์วน้ำ"

> 💡 **น่ารู้:** ในโรงงานอุตสาหกรรม ระบบ Predictive Maintenance ใช้ข้อมูลจากเซนเซอร์วัดแรงสั่นสะเทือน (Vibration Sensor) ส่งขึ้นแพลตฟอร์ม IoT เพื่อวิเคราะห์ว่าเครื่องจักรตัวไหนมีแนวโน้มจะเสียหาย ช่วยลดค่าซ่อมบำรุงได้มหาศาล

## 8.5 แพลตฟอร์ม IoT ยอดนิยม

### ThingSpeak

แพลตฟอร์มโอเพนซอร์สจาก MathWorks ใช้งานง่ายมาก รองรับ MATLAB สำหรับวิเคราะห์ข้อมูล เหมาะสำหรับผู้เริ่มต้นและงานวิจัย มีแผนฟรีให้ส่งข้อมูลได้ทุก 15 วินาที

### Blynk

แพลตฟอร์มที่เน้นการสร้างแอปมือถือควบคุม IoT ลากวาง Widget ได้ง่าย รองรับ ESP32 โดยตรงผ่าน Library สำเร็จรูป

### AWS IoT Core

บริการจาก Amazon Web Services รองรับอุปกรณ์จำนวนมหาศาล มี Device Shadow, Rules Engine และเชื่อมต่อกับบริการ AWS อื่น ๆ ได้อย่างราบรื่น เหมาะสำหรับงานระดับอุตสาหกรรม

### Azure IoT Hub

บริการจาก Microsoft รองรับ Device Twin, Message Routing และเชื่อมกับ Power BI สำหรับทำ Dashboard เหมาะกับองค์กรที่ใช้ระบบนิเวศ Microsoft อยู่แล้ว

### Google Cloud IoT

บริการจาก Google Cloud มี Pub/Sub สำหรับรับส่งข้อมูล BigQuery สำหรับวิเคราะห์ข้อมูลขนาดใหญ่ และ Looker Studio สำหรับแสดงผล

### ตารางเปรียบเทียบแพลตฟอร์ม IoT

| คุณสมบัติ | ThingSpeak | Blynk | AWS IoT Core | Azure IoT Hub | Google Cloud IoT |
|---|---|---|---|---|---|
| ความยากในการเริ่มต้น | ง่ายมาก | ง่าย | ปานกลาง-ยาก | ปานกลาง-ยาก | ปานกลาง |
| แผนฟรี | ✅ มี | ✅ มี (จำกัด) | ✅ Free Tier 12 เดือน | ✅ Free Tier | ✅ Free Tier |
| โปรโตคอลหลัก | HTTP | HTTP, MQTT | MQTT, HTTP | MQTT, AMQP, HTTP | MQTT, HTTP |
| แอปมือถือ | ไม่มี | ✅ มี (เด่นมาก) | ไม่มีในตัว | ไม่มีในตัว | ไม่มีในตัว |
| การวิเคราะห์ข้อมูล | MATLAB | พื้นฐาน | Lambda, SageMaker | Stream Analytics | BigQuery, AI/ML |
| เหมาะกับ | การเรียน, วิจัย | Prototype, DIY | งานระดับ Enterprise | องค์กร Microsoft | งาน Data-heavy |

## 8.6 การเชื่อมต่ออุปกรณ์กับคลาวด์

### แนวคิดพื้นฐาน

การส่งข้อมูลจาก ESP32 ขึ้นคลาวด์มีขั้นตอนหลัก:

1. **ลงทะเบียน** — สร้างบัญชีบนแพลตฟอร์ม และสร้าง Channel/Device
2. **รับ API Key** — แพลตฟอร์มจะออก API Key หรือ Token ให้ใช้ยืนยันตัวตน
3. **เขียนโค้ด** — โปรแกรม ESP32 ให้อ่านค่าเซนเซอร์แล้วส่งข้อมูลไปยัง URL ของแพลตฟอร์ม
4. **ดูผลลัพธ์** — เปิด Dashboard บนเว็บเพื่อดูกราฟแบบ Real-time

### ตัวอย่างโค้ด: ส่งข้อมูลอุณหภูมิไปยัง ThingSpeak ผ่าน HTTP

```cpp
#include <WiFi.h>
#include <HTTPClient.h>

const char* ssid = "WiFi_Name";
const char* password = "WiFi_Password";
const char* apiKey = "YOUR_THINGSPEAK_API_KEY";
const char* server = "http://api.thingspeak.com/update";

void setup() {
  Serial.begin(115200);
  WiFi.begin(ssid, password);
  while (WiFi.status() != WL_CONNECTED) {
    delay(500);
    Serial.print(".");
  }
  Serial.println("\nเชื่อมต่อ WiFi สำเร็จ!");
}

void loop() {
  if (WiFi.status() == WL_CONNECTED) {
    HTTPClient http;

    // สมมติอ่านค่าอุณหภูมิจากเซนเซอร์ (ตัวอย่างใช้ค่าสุ่ม)
    float temperature = 25.0 + random(0, 100) / 10.0;

    // สร้าง URL พร้อม API Key และข้อมูล
    String url = String(server) + "?api_key=" + apiKey
                 + "&field1=" + String(temperature);

    http.begin(url);
    int httpCode = http.GET();

    if (httpCode > 0) {
      Serial.println("ส่งข้อมูลสำเร็จ: " + String(temperature) + " °C");
    } else {
      Serial.println("ส่งข้อมูลล้มเหลว");
    }
    http.end();
  }

  delay(20000); // ThingSpeak แผนฟรีต้องรออย่างน้อย 15 วินาที
}
```

### ตัวอย่างโค้ด: ส่งข้อมูลผ่าน MQTT

```cpp
#include <WiFi.h>
#include <PubSubClient.h>

const char* ssid = "WiFi_Name";
const char* password = "WiFi_Password";
const char* mqttServer = "broker.hivemq.com";
const int mqttPort = 1883;
const char* topic = "eng/factory/temp";

WiFiClient espClient;
PubSubClient client(espClient);

void setup() {
  Serial.begin(115200);
  WiFi.begin(ssid, password);
  while (WiFi.status() != WL_CONNECTED) {
    delay(500);
  }
  client.setServer(mqttServer, mqttPort);
  while (!client.connected()) {
    if (client.connect("ESP32_Sensor_01")) {
      Serial.println("เชื่อมต่อ MQTT Broker สำเร็จ!");
    } else {
      delay(2000);
    }
  }
}

void loop() {
  client.loop();

  float temperature = 25.0 + random(0, 100) / 10.0;
  String payload = "{\"temp\":" + String(temperature) + ",\"unit\":\"C\"}";

  client.publish(topic, payload.c_str());
  Serial.println("Publish: " + payload);

  delay(5000);
}
```

> 💡 **หมายเหตุ Wokwi:** โค้ดข้างต้นสามารถทดสอบบน Wokwi ได้โดยใช้บอร์ด ESP32 ร่วมกับเซนเซอร์จำลอง เช่น DHT22 เพียงเปิดโปรเจกต์ใหม่บน [wokwi.com](https://wokwi.com) เลือกบอร์ด ESP32 แล้ววางโค้ด จากนั้นตั้งค่า WiFi ในส่วน Wokwi Network ได้เลย

## 8.7 ฐานข้อมูล Time-Series และการจัดเก็บข้อมูลเซนเซอร์

### ทำไมต้อง Time-Series Database?

ข้อมูลเซนเซอร์มีลักษณะเฉพาะ คือ "เวลา" เป็นแกนหลัก — ทุกจุดข้อมูลมี Timestamp กำกับ เช่น:

```
2025-06-22T10:30:00  อุณหภูมิ = 32.5°C  ความชื้น = 65%
2025-06-22T10:30:15  อุณหภูมิ = 32.7°C  ความชื้น = 64%
2025-06-22T10:30:30  อุณหภูมิ = 32.6°C  ความชื้น = 65%
```

ฐานข้อมูลแบบ Time-series ถูกออกแบบมาเพื่อ:

- **เขียนข้อมูลเร็วมาก** — รองรับข้อมูลหลายพันจุดต่อวินาที
- **บีบอัดข้อมูลเก่า** — ข้อมูลเก่าถูกบีบอัดอัตโนมัติเพื่อประหยัดพื้นที่
- **Query ตามช่วงเวลาเร็ว** — เช่น "ดึงค่าอุณหภูมิเฉลี่ยรายชั่วโมงของเดือนที่แล้ว"

### ตัวอย่างฐานข้อมูล Time-Series ที่นิยม

- **InfluxDB** — โอเพนซอร์ส ใช้กันแพร่หลาย มีภาษา Query เฉพาะชื่อ Flux
- **TimescaleDB** — สร้างบน PostgreSQL เหมาะกับผู้ที่คุ้นเคย SQL
- **Prometheus** — นิยมใช้ในงาน Monitoring ระบบ

### เปรียบเทียบกับฐานข้อมูลทั่วไป

ฐานข้อมูลเชิงสัมพันธ์ (เช่น MySQL) ก็เก็บข้อมูลเซนเซอร์ได้ แต่เมื่อข้อมูลมีจำนวนหลายล้านแถว ประสิทธิภาพจะลดลงอย่างมาก Time-series DB ถูกออกแบบมาโดยเฉพาะจึงจัดการข้อมูลลักษณะนี้ได้ดีกว่า

## 8.8 ความปลอดภัยและความเป็นส่วนตัวเบื้องต้น

### ภัยคุกคามในระบบ IoT

ระบบ IoT มีจุดอ่อนหลายจุดเนื่องจากอุปกรณ์ขนาดเล็กมักมีทรัพยากรจำกัด:

- **การดักจับข้อมูล (Eavesdropping)** — ผู้ไม่หวังดีดักฟังข้อมูลที่ส่งระหว่างอุปกรณ์กับคลาวด์
- **การปลอมแปลงอุปกรณ์ (Spoofing)** — ส่งข้อมูลปลอมเข้าระบบ ทำให้เกิดการตัดสินใจผิดพลาด
- **การโจมตี DDoS** — ส่งข้อมูลจำนวนมหาศาลให้เซิร์ฟเวอร์ล่ม
- **การเข้าถึงโดยไม่ได้รับอนุญาต** — แฮกเข้ามาควบคุมอุปกรณ์

### แนวทางป้องกันเบื้องต้น

| มาตรการ | รายละเอียด |
|---|---|
| **เข้ารหัสข้อมูล (Encryption)** | ใช้ HTTPS แทน HTTP, ใช้ MQTT over TLS แทน MQTT ธรรมดา |
| **ยืนยันตัวตน (Authentication)** | ใช้ API Key, Token หรือ Certificate ทุกครั้งที่เชื่อมต่อ |
| **สิทธิ์ขั้นต่ำ (Least Privilege)** | ให้อุปกรณ์มีสิทธิ์เฉพาะที่จำเป็นเท่านั้น เช่น เขียนข้อมูลได้แต่ลบไม่ได้ |
| **อัปเดตเฟิร์มแวร์ (OTA Update)** | อัปเดตซอฟต์แวร์อุปกรณ์จากระยะไกลเพื่อแก้ไขช่องโหว่ |
| **แยกเครือข่าย (Network Segmentation)** | แยกเครือข่าย IoT ออกจากเครือข่ายหลักขององค์กร |

### ความเป็นส่วนตัว (Privacy)

ในโปรเจกต์ IoT ที่เกี่ยวข้องกับผู้คน เช่น Smart Home หรือ Wearable Device ต้องคำนึงถึง:

- **พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล (PDPA)** — กฎหมายไทยที่กำหนดให้ต้องได้รับความยินยอมก่อนเก็บข้อมูลส่วนบุคคล
- **GDPR** — กฎหมายของสหภาพยุโรปที่มีผลกับระบบที่ให้บริการผู้ใช้ในยุโรป
- หลักการทั่วไป: **เก็บข้อมูลเท่าที่จำเป็น ลบเมื่อไม่ต้องใช้ แจ้งผู้ใช้ให้ชัดเจน**

> 💡 **สำหรับนักศึกษา:** ในการทำโปรเจกต์ควรปฏิบัติให้เป็นนิสัย — อย่า Hardcode รหัส WiFi หรือ API Key ลงในโค้ดที่แชร์บน GitHub ให้ใช้ไฟล์ `config.h` แยกต่างหากและเพิ่มชื่อไฟล์นี้ใน `.gitignore`

## สรุปบทที่ 8

- **คลาวด์คอมพิวติง** เป็นโครงสร้างพื้นฐานสำคัญของระบบ IoT ช่วยให้อุปกรณ์ขนาดเล็กอย่าง ESP32 สามารถจัดเก็บ วิเคราะห์ และแสดงผลข้อมูลเซนเซอร์ได้อย่างมีประสิทธิภาพ
- บริการคลาวด์แบ่งเป็น **IaaS** (จัดการทุกอย่างเอง), **PaaS** (จัดการเฉพาะแอป), **SaaS** (ใช้งานเลย)
- แพลตฟอร์ม IoT ประกอบด้วย 6 ส่วนหลัก: Device Management, Data Ingestion, Storage, Analytics, Visualization, Rules & Alerts
- **ThingSpeak** และ **Blynk** เหมาะกับผู้เริ่มต้น ส่วน AWS/Azure/Google Cloud เหมาะกับงานระดับองค์กร
- การส่งข้อมูลขึ้นคลาวด์ทำได้ผ่าน **HTTP** (ง่าย เหมาะส่งทุก 15+ วินาที) หรือ **MQTT** (เบา เร็ว เหมาะ Real-time)
- **ฐานข้อมูล Time-series** เช่น InfluxDB ออกแบบมาเพื่อข้อมูลเซนเซอร์โดยเฉพาะ
- ความปลอดภัยเป็นเรื่องสำคัญ — ใช้การเข้ารหัส ยืนยันตัวตน และระมัดระวังข้อมูลส่วนบุคคล

## แบบฝึกหัดท้ายบท

**ข้อ 1:** จงอธิบายความแตกต่างระหว่าง IaaS, PaaS และ SaaS โดยยกตัวอย่างในบริบทของระบบ IoT สำหรับ Smart Factory อย่างละ 1 ตัวอย่าง

**ข้อ 2:** นักศึกษาต้องการสร้างระบบตรวจวัดอุณหภูมิและความชื้นในห้องเซิร์ฟเวอร์ โดยใช้ ESP32 + DHT22 และแสดงผลกราฟบนเว็บ จงเลือกแพลตฟอร์ม IoT ที่เหมาะสมพร้อมให้เหตุผล

**ข้อ 3:** จงอธิบายขั้นตอนการส่งข้อมูลจาก ESP32 ไปยัง ThingSpeak ผ่าน HTTP ตั้งแต่เริ่มต้นจนเห็นกราฟบน Dashboard (อธิบายเป็นขั้นตอน)

**ข้อ 4:** จงเปรียบเทียบข้อดีข้อเสียของการใช้ HTTP กับ MQTT ในการส่งข้อมูล IoT ขึ้นคลาวด์ ในสถานการณ์ใดควรเลือกใช้โปรโตคอลใด?

**ข้อ 5:** ทำไมการเก็บข้อมูลเซนเซอร์จึงควรใช้ฐานข้อมูล Time-series แทนฐานข้อมูลเชิงสัมพันธ์ทั่วไป? อธิบายพร้อมยกตัวอย่างสถานการณ์

**ข้อ 6:** นักศึกษาพัฒนาระบบ IoT สำหรับบ้านอัจฉริยะ (Smart Home) ที่เก็บข้อมูลการเปิด-ปิดประตูและกล้องวงจรปิด จงระบุข้อควรระวังด้านความเป็นส่วนตัวตาม พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล (PDPA) อย่างน้อย 3 ข้อ

**ข้อ 7 (ปฏิบัติ):** สร้างโปรเจกต์บน Wokwi โดยใช้ ESP32 อ่านค่าจากเซนเซอร์อุณหภูมิ (จำลอง) แล้วส่งข้อมูลไปยัง ThingSpeak ทุก 20 วินาที จับภาพหน้าจอกราฟที่ได้พร้อมส่งโค้ดและ Wokwi Link
