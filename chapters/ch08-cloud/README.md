# Chapter 8: เทคโนโลยีคลาวด์และแพลตฟอร์ม IoT
## Cloud Technologies & IoT Platforms
---

ในรายวิชาเทคโนโลยีดิจิทัลสำหรับวิศวกรรม (Digital Technology for Engineering) สำหรับนักศึกษาวิศวกรรมเครื่องกลชั้นปีที่ 1 หลังจากที่เราได้เรียนรู้เกี่ยวกับการทำงานของไมโครคอนโทรลเลอร์ ESP32 การเชื่อมต่อเซนเซอร์เพื่อวัดค่าทางฟิสิกส์ และการควบคุมอุปกรณ์เบื้องต้นผ่านโปรแกรมจำลอง Wokwi ไปแล้ว ในบทนี้เราจะก้าวเข้าสู่หัวใจสำคัญของ Internet of Things (IoT) นั่นคือ "เทคโนโลยีคลาวด์" (Cloud Technology)

เทคโนโลยีคลาวด์ทำหน้าที่เป็นศูนย์กลางในการรวบรวม ประมวลผล และแสดงผลข้อมูลที่ได้จากอุปกรณ์ IoT จำนวนมาก การนำข้อมูลเหล่านี้ขึ้นสู่ระบบคลาวด์ทำให้นักวิศวกรสามารถตรวจสอบสถานะของเครื่องจักรหรือระบบต่างๆ ได้จากทุกที่ทั่วโลกตลอด 24 ชั่วโมง ซึ่งเป็นทักษะที่สำคัญอย่างยิ่งในยุคอุตสาหกรรม 4.0

### 8.1 คลาวด์คอมพิวติงคืออะไรและประโยชน์สำหรับ IoT

**คลาวด์คอมพิวติง (Cloud Computing)** คือการให้บริการทรัพยากรคอมพิวเตอร์ เช่น เซิร์ฟเวอร์ พื้นที่จัดเก็บข้อมูล ฐานข้อมูล เครือข่าย ซอฟต์แวร์ และระบบวิเคราะห์ข้อมูล ผ่านอินเทอร์เน็ต ("คลาวด์") แทนที่เราจะต้องลงทุนซื้อเครื่องเซิร์ฟเวอร์ราคาแพงและดูแลรักษาระบบเครือข่ายด้วยตนเอง เราสามารถเช่าใช้บริการเหล่านี้จากผู้ให้บริการคลาวด์และจ่ายค่าบริการตามการใช้งานจริง (Pay-as-you-go)

> 💡 **ข้อสังเกตสำหรับวิศวกรเครื่องกล:** ในอดีต หากเราต้องการเก็บข้อมูลอุณหภูมิและการสั่นสะเทือนของมอเตอร์ในโรงงาน เราต้องใช้ Data Logger และคอมพิวเตอร์ตั้งโต๊ะเฉพาะที่อยู่ในพื้นที่ แต่ด้วยเทคโนโลยีคลาวด์และการประยุกต์ใช้ IoT เราสามารถส่งข้อมูลจาก ESP32 ผ่าน Wi-Fi ขึ้นไปยังเซิร์ฟเวอร์บนอินเทอร์เน็ตได้โดยตรง ทำให้เราสามารถมอนิเตอร์และวิเคราะห์ข้อมูลจากที่บ้านหรือที่ใดก็ได้ในโลกที่มีอินเทอร์เน็ต

**ประโยชน์ของ Cloud Computing สำหรับ IoT ในงานวิศวกรรม:**
1. **Scalability (ความสามารถในการขยายตัว):** ระบบคลาวด์สามารถรองรับอุปกรณ์เซนเซอร์จากหลักสิบไปจนถึงนับหมื่นตัวได้พร้อมกันโดยไม่ต้องกังวลเรื่องการตั้งค่าเซิร์ฟเวอร์ใหม่ หากโรงงานขยายเครื่องจักรและต้องการเพิ่มจำนวน ESP32 เข้าสู่ระบบ ก็สามารถทำได้อย่างไร้รอยต่อ
2. **Accessibility (การเข้าถึงได้อย่างอิสระ):** ข้อมูลที่ถูกส่งขึ้นสู่ระบบคลาวด์สามารถเข้าถึงได้ผ่านสมาร์ทโฟน แท็บเล็ต หรือเว็บเบราว์เซอร์ ช่วยให้การเฝ้าระวังระบบ (System Monitoring) ทำได้ตลอดเวลา
3. **Cost-effectiveness (ความคุ้มค่าเชิงต้นทุน):** ผู้ใช้งานไม่ต้องกังวลกับต้นทุนฮาร์ดแวร์เซิร์ฟเวอร์ ไม่ต้องเสียค่าบำรุงรักษา หรือจ้างเจ้าหน้าที่ดูแลระบบเครือข่ายขั้นสูง ทำให้การริเริ่มโปรเจกต์ IoT ในยุคนี้เข้าถึงง่ายขึ้นมาก
4. **Reliability (ความน่าเชื่อถือระดับสูง):** ผู้ให้บริการคลาวด์ระดับโลกมักมีศูนย์ข้อมูลที่มีระบบไฟฟ้าสำรองและการทำสำเนาข้อมูล (Data Replication) ข้อมูลจากเซนเซอร์จึงมีความปลอดภัยจากการสูญหายมากกว่าการเก็บไว้ในแฟลชไดรฟ์หรือคอมพิวเตอร์ส่วนตัว
5. **Data Integration & Analytics (การรวมและวิเคราะห์ข้อมูลอัจฉริยะ):** คลาวด์คอมพิวติงมักมาพร้อมกับเครื่องมือประมวลผลข้อมูลปริมาณมหาศาล (Big Data) และปัญญาประดิษฐ์ (AI) ทำให้นักวิศวกรสามารถนำข้อมูลดิบมาวิเคราะห์หาแนวโน้ม เช่น การทำ Predictive Maintenance หรือการทำนายเวลาที่ชิ้นส่วนเครื่องจักรจะเสื่อมสภาพ

### 8.2 รูปแบบบริการ IaaS/PaaS/SaaS

บริการคลาวด์คอมพิวติงสามารถแบ่งออกเป็น 3 รูปแบบหลักๆ ตามระดับความรับผิดชอบและขอบเขตการควบคุมที่ผู้ใช้จะได้รับ ซึ่งในฐานะวิศวกร เราต้องเลือกใช้งานให้เหมาะสมกับความซับซ้อนของโครงงาน:

1. **IaaS (Infrastructure as a Service - บริการโครงสร้างพื้นฐาน):** ในรูปแบบนี้ ผู้ให้บริการจะจัดเตรียมเซิร์ฟเวอร์เสมือน พื้นที่เก็บข้อมูล และโครงข่ายอินเทอร์เน็ตไว้ให้ ผู้ใช้งานเปรียบเสมือนการเช่าคอมพิวเตอร์เปล่าๆ หนึ่งเครื่อง โดยต้องรับผิดชอบในการติดตั้งระบบปฏิบัติการ (เช่น Linux หรือ Windows) ฐานข้อมูล และซอฟต์แวร์แอปพลิเคชันทั้งหมดด้วยตนเอง เหมาะสำหรับผู้เชี่ยวชาญที่ต้องการควบคุมระบบแบบเบ็ดเสร็จ
2. **PaaS (Platform as a Service - บริการแพลตฟอร์ม):** ขยับความสะดวกสบายขึ้นมาอีกระดับ ผู้ให้บริการจะเตรียมทั้งเซิร์ฟเวอร์ ระบบปฏิบัติการ และชุดเครื่องมือสำหรับการพัฒนาโปรแกรมไว้ให้พร้อม ผู้ใช้ไม่ต้องเสียเวลาตั้งค่าพื้นฐาน มีหน้าที่เพียงนำโค้ดหรือแอปพลิเคชันที่ตนเองพัฒนามาฝากไว้ให้ทำงานบนแพลตฟอร์มนี้
3. **SaaS (Software as a Service - บริการซอฟต์แวร์):** เป็นบริการคลาวด์ที่สำเร็จรูปและพร้อมใช้งานมากที่สุด ผู้ใช้ไม่ต้องติดตั้งโปรแกรมหรือดูแลระบบเซิร์ฟเวอร์ใดๆ เบื้องหลังเลย เพียงแค่สมัครบัญชีผู้ใช้และล็อกอินเข้าใช้งานผ่านเว็บเบราว์เซอร์หรือแอปพลิเคชัน ตัวอย่างที่เห็นได้ชัดเจนคือระบบอีเมลทั่วไป หรือแพลตฟอร์ม IoT แบบสำเร็จรูปที่เน้นความง่ายในการใช้งาน

**ตารางที่ 1: เปรียบเทียบรูปแบบบริการ IaaS, PaaS และ SaaS สำหรับงานเทคโนโลยีดิจิทัลและ IoT**

| รูปแบบบริการคลาวด์ | คำอธิบายโดยสังเขป | ระดับภาระการจัดการของผู้ใช้ | ตัวอย่างผู้ให้บริการ / แพลตฟอร์ม | ความเหมาะสมสำหรับงาน IoT ในรายวิชานี้ |
| :--- | :--- | :--- | :--- | :--- |
| **IaaS** | ให้บริการเช่าฮาร์ดแวร์คอมพิวเตอร์เสมือน | **สูงสุด** (ผู้ใช้ต้องลง OS, ติดตั้งฐานข้อมูล และเขียนแอปพลิเคชันเอง) | Amazon EC2, Google Compute Engine, DigitalOcean | **ต่ำ** - ใช้เวลาในการเซ็ตอัประบบนานเกินไป ไม่เหมาะกับผู้เริ่มต้น |
| **PaaS** | ให้บริการเช่าสภาพแวดล้อมแพลตฟอร์มสำหรับสร้างซอฟต์แวร์ | **ปานกลาง** (ผู้ใช้ต้องจัดการเขียนโค้ดฝั่งแอปพลิเคชันและหน้าเว็บ) | AWS Elastic Beanstalk, Heroku, Google App Engine | **ปานกลาง** - เหมาะสำหรับโปรเจกต์ที่ต้องการเขียนแอปขึ้นมาใหม่โดยเฉพาะ |
| **SaaS** | ให้บริการเช่าซอฟต์แวร์พร้อมใช้งานได้ทันที | **ต่ำสุด** (ผู้ใช้ไม่ต้องสนระบบเบื้องหลัง ล็อกอินแล้วใช้งานได้เลย) | **Blynk**, Gmail, Microsoft 365, Dropbox | **สูงที่สุด** - ใช้งานง่าย เห็นผลลัพธ์ไว ได้หน้าจอสวยงามโดยไม่ต้องเขียนเว็บ |

### 8.3 รูปแบบการติดตั้ง Public/Private/Hybrid

นอกจากจะแบ่งตามรูปแบบบริการแล้ว ระบบคลาวด์ยังถูกจัดประเภทตาม "รูปแบบการติดตั้ง" (Deployment Models) ซึ่งเกี่ยวโยงกับระดับความเป็นส่วนตัว งบประมาณ และความปลอดภัยของข้อมูล:

*   **Public Cloud (คลาวด์สาธารณะ):** เป็นโครงสร้างพื้นฐานที่ผู้ให้บริการคลาวด์รายใหญ่ (เช่น AWS, Google, Microsoft หรือแม้แต่แพลตฟอร์มอย่าง Blynk) สร้างขึ้นเพื่อเปิดให้บุคคลทั่วไปและหลายๆ องค์กรเช่าใช้งานร่วมกันบนโครงข่ายอินเทอร์เน็ตสาธารณะ มีข้อดีคือต้นทุนต่ำ เริ่มต้นใช้งานได้ทันที ไม่ต้องซื้อฮาร์ดแวร์ แต่ข้อเสียคือข้อมูลของเราจะถูกจัดเก็บในสถานที่ของผู้ให้บริการ ซึ่งอาจถูกแชร์พื้นที่ฮาร์ดแวร์กับผู้ใช้อื่น
*   **Private Cloud (คลาวด์ส่วนตัว):** เป็นระบบคลาวด์ที่สร้างขึ้นมาโดยเฉพาะเพื่อใช้งานภายในองค์กรเดียวเท่านั้น อาจตั้งเซิร์ฟเวอร์อยู่ในอาคารของบริษัท (On-premises) หรือเช่าพื้นที่ศูนย์ข้อมูลเพื่อทำคลาวด์ปิด ข้อดีคือมีความปลอดภัยและรักษาความลับของข้อมูลได้ในระดับสูงสุด เหมาะสำหรับโรงงานผลิตที่ข้อมูลเครื่องจักรเป็นความลับทางการค้า แต่มีข้อเสียคือใช้เงินลงทุนสูงมากและต้องมีทีมวิศวกรไอทีดูแลระบบตลอด 24 ชั่วโมง
*   **Hybrid Cloud (คลาวด์แบบผสมผสาน):** เป็นสถาปัตยกรรมที่นำ Public Cloud และ Private Cloud มาทำงานร่วมกันอย่างเป็นระบบ องค์กรอาจเลือกเก็บข้อมูลการผลิตหลักที่เป็นความลับไว้ใน Private Cloud แต่ใช้ความสามารถของ Public Cloud ในการรันแอปพลิเคชันติดต่อลูกค้า หรือใช้ในช่วงที่มีปริมาณการประมวลผลเพิ่มขึ้นแบบฉับพลัน

สำหรับนักศึกษาที่กำลังเรียนวิชานี้และมุ่งเน้นการสร้างโปรเจกต์ต้นแบบ (Prototype) ด้าน IoT เราจะใช้รูปแบบ **Public Cloud** เป็นหลัก เนื่องจากสะดวก เข้าถึงได้จากอินเทอร์เน็ตทั่วไป และมีบริการแบบไม่มีค่าใช้จ่ายสำหรับผู้เริ่มต้น

### 8.4 องค์ประกอบแพลตฟอร์ม IoT

**IoT Platform (แพลตฟอร์ม IoT)** คือระบบซอฟต์แวร์ระดับกลาง (Middleware) บนระบบคลาวด์ ที่ทำหน้าที่เป็นสะพานเชื่อมต่อระหว่างฮาร์ดแวร์ขนาดเล็ก (เช่น บอร์ดไมโครคอนโทรลเลอร์ ESP32) กับแอปพลิเคชันหรือหน้าจอแสดงผลที่ผู้ใช้งานมองเห็น แพลตฟอร์ม IoT ที่ได้มาตรฐานและสมบูรณ์แบบระดับอุตสาหกรรม มักประกอบด้วยองค์ประกอบหลัก 5 ส่วนด้วยกัน:

1. **Device Management (ระบบการจัดการอุปกรณ์):**
   ทำหน้าที่บริหารจัดการอุปกรณ์ฮาร์ดแวร์จำนวนมากให้เป็นระเบียบ ครอบคลุมตั้งแต่การลงทะเบียนอุปกรณ์ใหม่เข้าระบบ การควบคุมสิทธิ์การเข้าถึง การตรวจสอบสถานะแบบเรียลไทม์ว่าอุปกรณ์ใดออนไลน์หรือออฟไลน์อยู่ และความสามารถที่สำคัญมากคือการอัปเดตเฟิร์มแวร์ผ่านอากาศ (Over-The-Air: OTA) โดยไม่ต้องเดินไปเสียบสาย USB ที่ตัวเครื่อง
2. **Data Ingestion (ช่องทางการรับเข้าข้อมูล):**
   ฮาร์ดแวร์จะส่งข้อมูลเข้าแพลตฟอร์มผ่านโปรโตคอลมาตรฐานต่างๆ สำหรับ IoT เช่น MQTT, HTTP/HTTPS, หรือ CoAP องค์ประกอบนี้ต้องถูกออกแบบให้มีประสิทธิภาพสูง สามารถรับข้อมูลจำนวนมหาศาล (High throughput) ที่ส่งมาจากเซนเซอร์หลายพันตัวพร้อมๆ กันโดยไม่คอขวด
3. **Data Storage (คลังพื้นที่จัดเก็บข้อมูล):**
   ฐานข้อมูลสำหรับจัดเก็บค่าที่อ่านได้จากเซนเซอร์ เนื่องจากข้อมูล IoT มักมีความถี่ในการส่งสูงและมีเวลาเข้ามาเกี่ยวข้องเสมอ ฐานข้อมูลที่ใช้งานส่วนใหญ่จึงเป็นประเภทฐานข้อมูลอนุกรมเวลา (Time-series data) เพื่อให้สามารถดึงข้อมูลในอดีตมาแสดงเป็นกราฟหรือนำไปวิเคราะห์ได้อย่างรวดเร็ว
4. **Visualization (การแสดงผลข้อมูลเป็นภาพ):**
   คือส่วนที่ผู้ใช้ทั่วไปจะมองเห็นบ่อยที่สุด เป็นเครื่องมือสร้างหน้าปัดควบคุม (Dashboard) แบบกราฟิก ซึ่งส่วนใหญ่มักออกแบบมาให้ใช้งานง่ายในรูปแบบ "ลากแล้ววาง" (Drag-and-drop) มีวิดเจ็ตมาตรฐานให้เลือกใช้มากมาย เช่น กราฟเส้นแบบต่อเนื่อง, มาตรวัดความดัน (Gauge), ตัวเลขดิจิทัล, ปุ่มกดเปิด-ปิดสวิตช์ หรือแถบเลื่อน (Slider) ทำให้นักวิศวกรสามารถสร้างเครื่องมือควบคุมได้รวดเร็วโดยไม่ต้องรู้ภาษาทำเว็บ
5. **Rule Engine & Alerts (ระบบกฎเกณฑ์อัตโนมัติและการแจ้งเตือน):**
   หัวใจสำคัญที่ทำให้ IoT มีความอัจฉริยะ คือความสามารถในการกำหนดเงื่อนไข (Condition) ให้แพลตฟอร์มตัดสินใจเอง เช่น "หากอุณหภูมิมอเตอร์ที่อ่านได้จากเซนเซอร์ ESP32 มีค่าเกิน 80 องศาเซลเซียส ให้ระบบส่งคำสั่งตัดไฟมอเตอร์ทันที พร้อมกับส่งการแจ้งเตือน (Push notification/Email) ไปยังโทรศัพท์มือถือของหัวหน้าวิศวกร"

### 8.5 แพลตฟอร์ม IoT ยอดนิยม

ในโลกแห่งการพัฒนาอินเทอร์เน็ตของสรรพสิ่ง มีแพลตฟอร์มมากมายหลายระดับให้ผู้พัฒนาเลือกใช้ ตั้งแต่แพลตฟอร์มเพื่อการศึกษา ไปจนถึงแพลตฟอร์มระดับโลกสำหรับองค์กรธุรกิจขนาดใหญ่ 

สำหรับรายวิชานี้ เราจะเลือกใช้แพลตฟอร์มที่ชื่อว่า **Blynk (Blynk Cloud / blynk.io)** เป็นเครื่องมือหลัก

**ทำไมเราจึงใช้ Blynk เป็นแพลตฟอร์มหลักในวิชานี้?**
Blynk ถูกออกแบบมาเป็นแพลตฟอร์มแบบ SaaS ที่เน้น "ความง่าย" และ "รวดเร็ว" ในการพัฒนาโปรเจกต์ IoT มีแอปพลิเคชันรองรับทั้งบนสมาร์ทโฟน (iOS/Android) และคอนโซลบนเว็บเบราว์เซอร์ ข้อดีที่โดดเด่นมากสำหรับนักศึกษาวิศวกรรมเครื่องกลที่อาจจะไม่ได้ถนัดการเขียนโค้ดหน้าเว็บ คือระบบการสร้าง Dashboard ที่สวยงามแบบลากวาง ซึ่งทำให้เราสามารถเชื่อมต่อเซนเซอร์และควบคุมบอร์ด ESP32 ได้ภายในเวลาไม่ถึงชั่วโมง

นอกเหนือจาก Blynk แล้ว ในแวดวงอุตสาหกรรมขนาดใหญ่ (Enterprise Level) เราจะพบแพลตฟอร์มอื่นที่มีความซับซ้อนและรองรับสเกลการทำงานที่ใหญ่ขึ้นมาก ดังเปรียบเทียบในตารางด้านล่าง

### 8.6 การเชื่อมต่ออุปกรณ์กับคลาวด์ พร้อมตัวอย่างโค้ด ESP32

การจะทำให้อุปกรณ์ฮาร์ดแวร์อย่าง ESP32 หรือการจำลองบน Wokwi ส่งข้อมูลไปถึงแพลตฟอร์มคลาวด์ได้นั้น จะต้องอาศัยการเชื่อมต่อเครือข่ายอินเทอร์เน็ตผ่าน Wi-Fi จากนั้นจึงนำไลบรารีของแพลตฟอร์มมาใช้ในโค้ด เพื่อทำการจัดรูปแบบข้อมูล (Data Packet) ให้ถูกต้องตามที่ระบบปลายทางต้องการ

**ขั้นตอนและแนวคิดหลักในการส่งข้อมูลขึ้น Blynk:**
1. **เตรียมโปรเจกต์บนคลาวด์:** เข้าสู่หน้าเว็บ blynk.io สร้างแม่แบบโปรเจกต์ (Template) และเพิ่มอุปกรณ์ใหม่ (Device) ลงไป
2. **รับกุญแจยืนยันตัวตน:** ระบบจะมอบ **Auth Token** ซึ่งเป็นสตริงตัวอักษรยาวๆ ที่ใช้เป็นรหัสผ่านเฉพาะของอุปกรณ์ตัวนั้น ห้ามให้ผู้อื่นรู้เด็ดขาด
3. **รู้จักพินเสมือน (Virtual Pin):** นี่คือหัวใจของ Blynk แทนที่เราจะส่งข้อมูลไปยังพินฮาร์ดแวร์ (เช่น ขา 2, ขา 4) ตรงๆ เราจะส่งข้อมูลเข้า "ช่องทางเสมือน" เช่น `V0`, `V1`, `V2` ที่อยู่บนคลาวด์แทน ทำให้การเขียนโค้ดและการสร้าง Dashboard มีความเป็นอิสระต่อกัน
4. **การเขียนคำสั่งใน C++:** ใช้ฟังก์ชันของไลบรารี Blynk เช่น `Blynk.virtualWrite(VirtualPin, Data)` เพื่อส่งค่าตัวเลขหรือข้อความจาก ESP32 วิ่งผ่านอินเทอร์เน็ตไปจัดเก็บในช่องสัญญาณ Virtual Pin ที่กำหนดไว้

> 💡 **ข้อแนะนำการปฏิบัติในโปรแกรมจำลอง Wokwi:** สำหรับการจำลองวงจรบนเว็บไซต์ Wokwi คุณไม่ต้องกังวลเรื่องการเชื่อมต่อ Wi-Fi ที่บ้าน เพราะ Wokwi มีฟีเจอร์เกตเวย์จำลอง ให้กำหนดชื่อ SSID เป็น `"Wokwi-GUEST"` และเว้นช่องรหัสผ่านว่างไว้ (`""`) บอร์ด ESP32 เสมือนของคุณก็จะสามารถเชื่อมต่ออินเทอร์เน็ตและพูดคุยกับเซิร์ฟเวอร์ของ Blynk ได้ทันที

**ตัวอย่างโค้ด: การจำลองการส่งข้อมูลเซนเซอร์ผ่าน ESP32 ขึ้นสู่ Blynk Cloud**

ตัวอย่างโค้ดด้านล่างนี้แสดงวิธีการเชื่อมต่อ ESP32 เข้ากับ Blynk และตั้งเวลาให้ส่งข้อมูลแบบสุ่ม (เสมือนเป็นการอ่านค่าจากเซนเซอร์อุณหภูมิ) ขึ้นไปยังช่องทาง **Virtual Pin 1 (V1)** ทุกๆ 2 วินาทีอย่างต่อเนื่อง

```cpp
// ส่วนหัวที่สำคัญมาก 3 บรรทัดนี้ จะต้องนำมาจากหน้า Device Info บน Blynk Console ของผู้ใช้เอง
#define BLYNK_TEMPLATE_ID "TMPLxxxxxx"
#define BLYNK_TEMPLATE_NAME "My ESP32 Project"
#define BLYNK_AUTH_TOKEN "Your_Auth_Token_Here" // นำ Auth Token ของคุณมาใส่แทนที่ข้อความนี้

// ตั้งค่าให้ระบบสามารถพิมพ์ข้อความแจ้งเตือนสถานะต่างๆ ออกทาง Serial Monitor ได้
#define BLYNK_PRINT Serial

// เรียกใช้งานไลบรารีที่จำเป็นสำหรับการเชื่อมต่อเครือข่ายและการใช้ Blynk บน ESP32
#include <WiFi.h>
#include <WiFiClient.h>
#include <BlynkSimpleEsp32.h>

// ตั้งค่าเครือข่าย Wi-Fi
// หากต่ออุปกรณ์จริง ให้เปลี่ยนเป็นชื่อและรหัสผ่าน Wi-Fi บ้านของคุณ
// หากทดลองบน Wokwi ให้คงค่า "Wokwi-GUEST" เอาไว้โดยไม่ต้องมีรหัสผ่าน
char ssid[] = "Wokwi-GUEST";
char pass[] = "";

// สร้างออบเจกต์ Timer สำหรับควบคุมจังหวะการทำงาน แทนการใช้ฟังก์ชัน delay()
BlynkTimer timer;

// ฟังก์ชันสำหรับจำลองการอ่านค่าจากเซนเซอร์และส่งขึ้นระบบคลาวด์
void sendSensorData() {
  // จำลองตัวเลขอุณหภูมิแบบสุ่ม (หากทำโปรเจกต์จริง ให้แทนที่ด้วยคำสั่งอ่านค่าจากเซนเซอร์ DHT หรืออื่นๆ)
  float simulatedTemperature = random(200, 350) / 10.0; 
  
  Serial.print("Sending Temperature to Cloud: ");
  Serial.println(simulatedTemperature);
  
  // คำสั่งที่สำคัญที่สุด: ส่งค่าตัวแปรอุณหภูมิ ขึ้นไปยังคลาวด์ที่ช่อง Virtual Pin 1 (V1)
  Blynk.virtualWrite(V1, simulatedTemperature);
}

void setup() {
  // เริ่มต้นเปิดการสื่อสารกับคอมพิวเตอร์ผ่านสาย USB ด้วยความเร็ว 115200 บิตต่อวินาที
  Serial.begin(115200);
  
  // เริ่มต้นกระบวนการเชื่อมต่ออินเทอร์เน็ตและเข้าสู่ระบบของ Blynk ด้วยกุญแจ Auth Token
  Blynk.begin(BLYNK_AUTH_TOKEN, ssid, pass);
  
  // ตั้งค่าให้ Timer ทำการเรียกฟังก์ชัน sendSensorData ขึ้นมาทำงาน ทุกๆ 2000 มิลลิวินาที (2 วินาที)
  timer.setInterval(2000L, sendSensorData);
}

void loop() {
  // สั่งให้ระบบหลังบ้านของ Blynk ทำงานอย่างต่อเนื่อง (รับ-ส่งข้อมูล)
  Blynk.run();
  
  // สั่งให้ระบบ Timer ตรวจสอบเวลาอย่างต่อเนื่อง ว่าครบ 2 วินาทีหรือยัง
  timer.run();
}
```

*ข้อควรระวังขั้นวิกฤตสำหรับการเขียนโปรแกรม IoT: คุณต้องหลีกเลี่ยงการใช้คำสั่ง `delay(1000);` ลงไปในบล็อกของ `void loop()` โดยเด็ดขาด เพราะการใช้ `delay()` จะทำให้ระบบประมวลผลหยุดนิ่ง และอาจส่งผลให้การเชื่อมต่อระหว่าง ESP32 และเซิร์ฟเวอร์ของ Blynk ขาดการเชื่อมต่อ (Connection Timeout) ดังนั้นการใช้ `BlynkTimer` ตามตัวอย่างด้านบนจึงเป็นวิธีการมาตรฐานที่ปลอดภัยที่สุด*

### 8.7 ฐานข้อมูล time-series และการจัดเก็บข้อมูลเซนเซอร์

เมื่อข้อมูลถูกส่งด้วยโค้ด `Blynk.virtualWrite()` ผ่านอินเทอร์เน็ตไปถึงคลาวด์เรียบร้อยแล้ว แพลตฟอร์มจำเป็นต้องบันทึกข้อมูลเหล่านั้นเพื่อนำมาใช้ประโยชน์ ระบบของ IoT มักจะต้องรับมือกับข้อมูลที่วิ่งเข้ามาตลอดเวลา เช่น วินาทีละ 1 ครั้ง หากเราใช้เซนเซอร์ 1,000 ตัว ข้อมูลก็จะไหลเข้าอย่างมหาศาล 

ด้วยลักษณะเฉพาะของข้อมูลเซนเซอร์ที่เกิดขึ้นอย่างต่อเนื่อง แพลตฟอร์มส่วนใหญ่จึงหลีกเลี่ยงฐานข้อมูลแบบตารางทั่วไป (Relational Database อย่าง MySQL) แล้วหันมาใช้ **ฐานข้อมูลอนุกรมเวลา (Time-series Database)** แทน ซึ่งออกแบบมาโดยเฉพาะสำหรับข้อมูลที่มีแกนหลักเป็นเวลา (Timestamp) คู่กับค่าตัวเลขของข้อมูล

**ตารางที่ 3: ความแตกต่างระหว่างฐานข้อมูลแบบดั้งเดิม (SQL) และ Time-series Database สำหรับโปรเจกต์ IoT**

| คุณลักษณะเด่น | Relational Database (SQL แบบดั้งเดิม) | Time-series Database (TSDB สำหรับ IoT) |
| :--- | :--- | :--- |
| **โครงสร้างการจัดเก็บข้อมูล** | เก็บข้อมูลเป็นตารางที่มีความสัมพันธ์ (Relationships) ผูกกันอย่างซับซ้อน | เน้นการเก็บข้อมูลเรียงกันตามลำดับของเวลาอย่างง่าย (มีแค่เวลาและค่าที่อ่านได้) |
| **ประสิทธิภาพความเร็วในการเขียน** | ปานกลาง (อาจเกิดปัญหาขวดคอ หากมีข้อมูลส่งเข้ามาหลายหมื่นรายการต่อวินาที) | **สูงมาก** (โครงสร้างถูกออกแบบมาให้เขียนข้อมูลชุดใหม่ทับลงไปได้เรื่อยๆ โดยไม่สะดุด) |
| **วัตถุประสงค์ในการจัดการข้อมูลเก่า** | สนับสนุนการกลับไปแก้ไขปรับปรุง หรือลบข้อมูลเก่าได้อย่างรวดเร็ว | มักเป็นข้อมูลแบบเขียนแล้วจบ (Append-only) เพราะค่าเซนเซอร์ในอดีตไม่จำเป็นต้องแก้ไข |
| **ประสิทธิภาพการสร้างกราฟย้อนหลัง** | ต้องใช้คำสั่ง Query ดึงข้อมูลที่ซับซ้อนและกินทรัพยากร | ถูกปรับจูนมาให้ดึงข้อมูลเป็นช่วง (เช่น ย้อนหลัง 7 วัน) ได้รวดเร็ว เพื่อแสดงกราฟ |

ข้อดีอย่างหนึ่งของการใช้งานแพลตฟอร์มสำเร็จรูปอย่าง Blynk คือ นักศึกษาไม่จำเป็นต้องติดตั้งหรือบริหารจัดการ Time-series Database ด้วยตัวเองเลย ระบบเบื้องหลังของคลาวด์จัดการให้ทั้งหมด เพียงแค่เราเลือกนำวิดเจ็ตประเภท "กราฟ (Chart)" มาวางบนหน้าจอ แพลตฟอร์มก็จะทำหน้าที่ดึงข้อมูลจากฐานข้อมูลมาวาดเป็นเส้นกราฟอุณหภูมิที่สวยงามตามช่วงเวลาที่เราต้องการได้โดยอัตโนมัติ

### 8.8 ความปลอดภัยและความเป็นส่วนตัวเบื้องต้น

สำหรับฮาร์ดแวร์ IoT ที่มักถูกติดตั้งทิ้งไว้ตามจุดต่างๆ ทั่วโรงงานและทำงานแบบ 24 ชั่วโมง ความตระหนักด้านความปลอดภัยไซเบอร์เป็นสิ่งที่วิศวกรทุกคนละเลยไม่ได้เด็ดขาด เพราะหากระบบถูกโจมตี อาจหมายถึงการเข้าควบคุมเครื่องจักร หรือขโมยข้อมูลความลับขององค์กรไปได้

หลักการรักษาความปลอดภัยเบื้องต้นที่เกี่ยวข้องกับการทำโปรเจกต์ IoT มีดังนี้:

1. **Authentication (การยืนยันและระบุตัวตน):** การเข้าถึงแพลตฟอร์มคลาวด์ต้องมีการระบุตัวตนที่ชัดเจน สำหรับอุปกรณ์ที่ใช้เชื่อมต่อกับแพลตฟอร์ม Blynk จะมีการใช้ **Auth Token** ซึ่งเปรียบเสมือนบัตรประจำตัวและรหัสผ่าน หาก Auth Token นี้หลุดรอดไปอยู่บนอินเทอร์เน็ต ผู้ไม่หวังดีจะสามารถส่งคำสั่งมาปิด-เปิดสวิตช์อุปกรณ์ของคุณได้จากระยะไกล ดังนั้น ต้องเก็บรหัสนี้ไว้เป็นความลับ และเมื่อส่งงานอาจต้องทำเซนเซอร์รหัสบางส่วนไว้
2. **Encryption (การเข้ารหัสข้อมูลระหว่างทาง):** ข้อมูลเซนเซอร์ที่เดินทางออกจากบอร์ด ESP32 ผ่านชั้นบรรยากาศ (Wi-Fi) และสายไฟเบอร์ออฟติกสู่อินเทอร์เน็ต มีโอกาสถูกดักจับข้อมูลได้ (Man-in-the-middle attack) แพลตฟอร์ม IoT มาตรฐานจึงมักบังคับใช้โพรโทคอลที่มีการเข้ารหัส เช่น HTTPS หรือ MQTTS ทำให้ถึงแม้จะมีคนดักข้อมูลไปได้ ก็จะเห็นเป็นเพียงรหัสที่อ่านไม่รู้เรื่อง
3. **Firmware Integrity (ความปลอดภัยของโค้ดโปรแกรม):** เฟิร์มแวร์หรือโค้ดภาษา C++ ที่เราเขียนลงไปในไมโครคอนโทรลเลอร์จะต้องได้รับการทดสอบอย่างรัดกุม ไม่ให้เกิดข้อผิดพลาดในการรับข้อมูล (Buffer overflow) ซึ่งอาจเป็นช่องโหว่ให้แฮกเกอร์ฝังโค้ดอันตรายเข้าควบคุมตัวบอร์ดได้

### สรุปท้ายบท

เทคโนโลยีคลาวด์และแพลตฟอร์ม IoT นับเป็นกุญแจสำคัญที่เข้ามาพลิกโฉมวิธีการทำงานของวิศวกรเครื่องกลในยุคปัจจุบัน โดยเปลี่ยนวิธีการจัดเก็บและวิเคราะห์ข้อมูลจากการอ่านค่ามิเตอร์ในพื้นที่หน้างาน ให้กลายเป็นการส่งต่อข้อมูลแบบอัตโนมัติไปยังระบบศูนย์กลางส่วนกลาง ที่สามารถมอนิเตอร์และแสดงผลในแบบเรียลไทม์ (Real-time) ได้ตลอด 24 ชั่วโมง

การเลือกใช้บริการคลาวด์แบบ SaaS เช่น **Blynk** เป็นเครื่องมือเริ่มต้นที่ทรงประสิทธิภาพที่สุดสำหรับนักศึกษาชั้นปีที่ 1 เนื่องจากตัวแพลตฟอร์มได้จัดการความซับซ้อนของฐานข้อมูลเซนเซอร์ (Time-series Database) ออกไปจนหมด เหลือเพียงแค่ความสะดวกในการสร้างแดชบอร์ด (Dashboard) ที่สวยงาม และการเรียนรู้การเขียนโค้ดภาษา C++ ร่วมกับไลบรารีเพื่อสื่อสารผ่านพินเสมือน (Virtual Pin) สิ่งเหล่านี้คือรากฐานความรู้ที่แข็งแกร่ง ในการนำไปประยุกต์ใช้เพื่อการพัฒนาระบบควบคุมอัตโนมัติระยะไกลในรายวิชาโปรเจกต์ของสาขาวิศวกรรมเครื่องกลต่อไปในอนาคต

**ส่วนที่ 2: แบบฝึกหัดข้อปฏิบัติ (Practical Exercise - ให้ใช้โปรแกรม Wokwi ในการทำภารกิจ)**

7. **ภารกิจการจำลองส่งข้อมูลขึ้นคลาวด์:**
   ให้นักศึกษาเข้าสู่หน้าเว็บไซต์ของระบบจำลองฮาร์ดแวร์ Wokwi และสร้างโปรเจกต์ใหม่โดยใช้บอร์ดจำลอง ESP32 จากนั้นให้ทำการลงทะเบียนเข้าใช้งานแพลตฟอร์มคลาวด์ของ Blynk (blynk.io) พร้อมกับสร้าง Template ระบบจำลองใหม่ เพื่อขอรับค่า BLYNK_TEMPLATE_ID และ BLYNK_AUTH_TOKEN ประจำตัวนักศึกษา
8. **ภารกิจการเขียนโปรแกรมและการสร้างแดชบอร์ด (ส่งภาพหน้าจอเป็นคำตอบ):**
   ให้นำความรู้จากตัวอย่างโค้ดในบทเรียนมาดัดแปลง จงเขียนโปรแกรมภาษา C++ เพื่อให้บอร์ด ESP32 บน Wokwi ทำการสุ่มตัวเลขเพื่อจำลองการอ่านค่า "ความกดอากาศ (Pressure)" ในช่วงระหว่าง 900.0 ถึง 1100.0 hPa แล้วส่งข้อมูลตัวเลขนี้ขึ้นไปยังเซิร์ฟเวอร์ของ Blynk ผ่านช่องทาง Virtual Pin ที่ 3 (`V3`) ในทุกๆ จังหวะเวลา 3 วินาที 
   หลังจากนั้น ให้เข้าไปที่ Blynk Web Console และสร้าง Web Dashboard นำวิดเจ็ตประเภทมาตรวัด (Gauge) และประเภทกราฟต่อเนื่อง (Chart) มาวาง ผูกค่าการอ่านเข้ากับ `V3` เพื่อแสดงการเปลี่ยนแปลงของความกดอากาศแบบเรียลไทม์ ให้นักศึกษาทดสอบรันโปรแกรมจนกว่าค่าบนกราฟจะเปลี่ยนแปลง แล้วทำการแคปเจอร์หน้าจอ (Screen Capture) ของทั้งโค้ดและหน้าเว็บ Dashboard ส่งเป็นผลงาน
